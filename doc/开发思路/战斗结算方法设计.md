## 2. 战斗结算方法设计总原则

### 2.1 战斗结算必须只有统一正式入口
战斗结算链路必须只有一个正式 Workflow 入口，不允许：

- Controller 直接查掉落组后写背包
- 页面层自己把首通奖励和掉落拼起来
- 后台工具绕过正式结算链手动发放结算结果

正式结算结果必须由正式 Workflow 基于正式状态与正式服务链统一生成。

### 2.2 战斗结算必须建立在正式战斗准备结果之上
战斗结算不是孤立动作，必须建立在正式战斗准备链返回的上下文基础上。

至少要能对应：

- 本次关卡难度
- 本次怪物来源
- 本次角色正式状态
- 本次战斗上下文标识或等价上下文

不允许完全脱离战斗准备链，直接凭页面临时参数生成正式结算。

### 2.3 掉落链、奖励链、入包链必须分链实现后再汇总
战斗结算中可以同时出现：

- 怪物随机掉落
- 副本难度首通固定奖励

但它们必须：

- 分别调用正式服务
- 分别生成正式结果
- 最后统一汇总返回

不能混成一套模糊流程。

### 2.4 正式结算返回必须基于正式写入成功结果
战斗结算接口返回的内容，不应只是“理论结果”，而应基于以下正式成功结果：

- 掉落解析结果
- 奖励发放结果
- 入包结果
- 装备实例创建结果

### 2.5 核心写入步骤必须可追溯
涉及以下内容的结算步骤，必须可追溯：

- 本次掉了什么
- 本次是否发了首通奖励
- 本次发奖是否成功
- 本次创建了哪些装备实例
- 本次有哪些内容正式入包

## 3. 战斗结算链路目标与边界

## 3.1 链路目标
在战斗结束后，根据正式战斗结果，生成并落定本次正式产出结果，最终返回可展示、可追溯、可核查的正式结算结果。

## 3.2 链路起点
客户端提交“战斗结算”请求，至少包含：

- `user_id`
- `character_id`
- `stage_difficulty_id`
- `battle_context_id` 或等价上下文标识
- 本次击败怪物结果摘要
- 本次战斗是否通关

## 3.3 链路终点
满足以下条件时，战斗结算链路才算完成：

1. 战斗结果已通过正式校验
2. 掉落结果已正式生成
3. 若满足首通条件，奖励已正式发放或明确失败状态
4. 掉落与奖励已通过入包链路承接
5. 返回正式结算结果

## 3.4 链路不负责内容
战斗结算链路不负责：

- 客户端实时战斗驱动
- 战斗表现与动画
- 掉落规则定义
- 奖励模板定义
- 装备成长系统定义

---

## 4. 方法分层建议

当前战斗结算链路建议拆为以下三层方法。

### 4.1 Query / Config 层
负责读取正式上下文、正式配置、正式记录状态。

建议方法：

- `getStageDifficultyById()`
- `getBattlePrepareContext()`（若第一阶段未持久化，可由结算入参替代等价上下文校验）
- `getMonsterMapByIds()`
- `getFirstClearRewardStatus()`

### 4.2 Domain 层
负责单点校验、掉落结果整理、结算结果组装。

建议方法：

- `validateBattleSettlementRequest()`
- `assertBattleContextValid()`
- `assertBattleResultValid()`
- `extractKilledMonsterIds()`
- `buildDropSettlementContext()`
- `buildRewardSettlementContext()`
- `buildBattleSettlementPayload()`

### 4.3 Workflow 层
负责组织完整战斗结算流程。

建议方法：

- `settleBattle()`

## 6. Workflow 方法设计

## 6.1 `settleBattle()`

### 所属层级
Workflow

### 方法目标
作为战斗结算链路唯一正式入口，组织结算校验、掉落解析、奖励发放、入包承接与正式结果汇总。

### 建议签名
```php
public function settleBattle(int $userId, int $characterId, string $stageDifficultyId, array $battleResult): array
```

### 输入说明
- `userId`：当前用户 ID
- `characterId`：目标角色 ID
- `stageDifficultyId`：目标关卡难度 ID
- `battleResult`：本次战斗结果摘要，至少包含：
  - `battle_context_id` 或等价上下文
  - `is_cleared`
  - `killed_monsters`

### 输出说明
返回正式战斗结算结果，建议至少包含：

- `stage_difficulty`
- `is_cleared`
- `drop_results`
- `reward_results`
- `inventory_results`
- `created_equipment_instances`
- `first_clear_reward_status`
- `settlement_summary`

### 执行步骤
1. 调用 `validateBattleSettlementRequest()` 做前置校验
2. 读取正式关卡难度信息
3. 校验战斗上下文 `assertBattleContextValid()`
4. 校验战斗结果合法性 `assertBattleResultValid()`
5. 提取击败怪物列表 `extractKilledMonsterIds()`
6. 构造掉落结算上下文 `buildDropSettlementContext()`
7. 调用 `DropResolverService` 生成正式掉落结果
8. 若满足首通条件，构造奖励结算上下文 `buildRewardSettlementContext()`
9. 调用 `RewardGrantWorkflow` 处理首通奖励发放
10. 汇总掉落结果、奖励结果、入包结果、实例创建结果
11. 调用 `buildBattleSettlementPayload()` 组装正式结算结果
12. 返回结果

### 事务要求
当前建议事务边界如下：

- 掉落解析本身不一定要求在本 Workflow 中开启总事务
- 但奖励发放链内部必须有独立正式事务
- 入包链内部必须有正式事务保护

第一阶段若结算接口要做到“整次结算全成功或全失败”，可以再进一步收紧为更大事务；若暂不这么做，必须保证各子链正式状态自洽且可追溯。

### 异常处理要求
- 关键校验失败必须直接中断结算
- 掉落解析失败必须阻断正式结算返回
- 首通奖励失败时，必须返回明确失败状态，不允许伪装成功
- 必须记录必要日志，至少包含：`user_id`、`character_id`、`stage_difficulty_id`、`battle_context_id`

### 不负责内容
- 不直接定义掉落规则
- 不直接定义奖励模板
- 不直接实现入包 SQL 细节

## 7. Domain 方法设计

## 7.1 `validateBattleSettlementRequest()`

### 所属层级
Domain

### 方法目标
校验战斗结算请求是否满足正式结算前置条件。

### 建议签名
```php
public function validateBattleSettlementRequest(int $userId, int $characterId, string $stageDifficultyId, array $battleResult): void
```

### 必须校验的内容
1. `stage_difficulty_id` 合法
2. `battle_context_id` 或等价上下文存在
3. `killed_monsters` 格式合法
4. `is_cleared` 字段存在且类型合法
5. 当前用户与角色关系合法

### 依赖方法
- `getStageDifficultyById()`
- `assertBattleContextValid()`
- `assertBattleResultValid()`

### 不负责内容
- 不执行掉落
- 不执行发奖
- 不组装返回结果

---

## 7.2 `assertBattleContextValid()`

### 所属层级
Domain

### 方法目标
校验本次结算提交的战斗上下文是否与正式战斗准备结果一致。

### 建议签名
```php
public function assertBattleContextValid(int $userId, int $characterId, string $stageDifficultyId, array $battleResult, ?array $battlePrepareContext = null): void
```

### 当前阶段要求
即使第一阶段不持久化战斗会话表，也至少应校验：

- `stage_difficulty_id` 是否与本次请求一致
- `character_id` 是否与本次请求一致
- `battle_context_id` 是否存在或格式正确（如已生成）

### 后续扩展方向
后续若持久化战斗准备结果，可进一步校验：

- 怪物列表一致性
- 角色快照一致性
- 上下文有效期

---

## 7.3 `assertBattleResultValid()`

### 所属层级
Domain

### 方法目标
校验本次战斗结果摘要是否满足正式结算要求。

### 建议签名
```php
public function assertBattleResultValid(array $battleResult): void
```

### 当前阶段建议校验
- `killed_monsters` 是否为数组
- 怪物 ID 是否格式合法
- 击败列表是否为空时与 `is_cleared` 的组合是否合理
- 不允许明显超出当前关卡难度怪物范围的怪物 ID

### 不负责内容
- 不执行怪物归属查询
- 不执行掉落解析

---

## 7.4 `extractKilledMonsterIds()`

### 所属层级
Domain

### 方法目标
从战斗结果摘要中提取本次正式用于掉落结算的怪物 ID 列表。

### 建议签名
```php
public function extractKilledMonsterIds(array $battleResult): array
```

### 输出要求
- 返回标准化怪物 ID 列表
- 允许后续按需要保留重复怪物命中次数或数量信息

### 当前阶段建议
若后续掉落按“击败怪物明细次数”处理，可让输出结构保留数量；若当前按逐只怪物结算，建议返回标准化数组。

---

## 7.5 `buildDropSettlementContext()`

### 所属层级
Domain

### 方法目标
将正式怪物击败结果转换为掉落解析服务可直接消费的上下文结构。

### 建议签名
```php
public function buildDropSettlementContext(string $stageDifficultyId, array $killedMonsterIds, array $battleResult = []): array
```

### 建议输出内容
至少包含：

- `stage_difficulty_id`
- `killed_monster_ids`
- `battle_context_id`
- 必要的来源说明字段

### 关键要求
- 只组织上下文
- 不在本方法中写掉落算法

---

## 7.6 `buildRewardSettlementContext()`

### 所属层级
Domain

### 方法目标
将本次战斗通关与首通状态转换为奖励发放服务可直接消费的上下文结构。

### 建议签名
```php
public function buildRewardSettlementContext(int $userId, string $stageDifficultyId, array $battleResult, array $rewardStatus): array
```

### 建议输出内容
至少包含：

- `user_id`
- `source_type = first_clear`
- `source_id = stage_difficulty_id`
- `is_cleared`
- `reward_status`
- 必要的幂等上下文

### 关键要求
- 只负责组织首通奖励上下文
- 不直接写发奖记录

---

## 7.7 `buildBattleSettlementPayload()`

### 所属层级
Domain

### 方法目标
将正式掉落结果、奖励结果、入包结果与实例结果整合成接口或上层可直接消费的正式结算结果。

### 建议签名
```php
public function buildBattleSettlementPayload(array|object $stageDifficulty, array $battleResult, array $dropResults, array $rewardResults = [], array $inventoryResults = [], array $instanceResults = []): array
```

### 建议输出内容
至少包含：

- `stage_difficulty`
- `is_cleared`
- `drop_results`
- `reward_results`
- `inventory_results`
- `created_equipment_instances`
- `first_clear_reward_status`
- `settlement_summary`

### 关键要求
- 必须基于正式成功结果组装
- 不应把“理论奖励”当作“正式发放结果”返回
- 应明确区分掉落部分与奖励部分

## 9. 方法调用关系建议

建议调用顺序如下：

```text
settleBattle()
 ├─ validateBattleSettlementRequest()
 │   ├─ getStageDifficultyById()
 │   ├─ assertBattleContextValid()
 │   └─ assertBattleResultValid()
 ├─ extractKilledMonsterIds()
 ├─ buildDropSettlementContext()
 ├─ DropResolverService::resolve(...)
 ├─ getFirstClearRewardStatus()
 ├─ buildRewardSettlementContext()（满足条件时）
 ├─ RewardGrantWorkflow::grant(...)
 └─ buildBattleSettlementPayload()
```

这个结构的目的，是把：

- 请求校验
- 掉落链调用
- 奖励链调用
- 正式结果汇总

彻底拆开，避免一个方法里直接堆满所有逻辑。

## 10. 事务边界与一致性策略

### 10.1 当前阶段建议
当前阶段可以采用“子链事务自洽 + 总结算结果汇总”的一致性策略。

即：

- 掉落解析服务自身保证结果可靠
- 奖励发放链内部使用正式事务
- 入包链内部使用正式事务
- 战斗结算 Workflow 负责汇总正式结果

### 10.2 当前阶段禁止出现的状态
- 结算返回奖励成功，但发奖记录不存在
- 结算返回装备掉落已入包，但未创建实例
- 结算返回成功，但掉落或奖励结果只是理论结果，没有正式状态支撑

### 10.3 后续扩展方向
后续若要进一步强化一致性，可扩展：

- 持久化战斗会话表
- 持久化结算记录表
- 大事务或补偿事务机制

但第一阶段先以正式子链自洽为主。

## 11. 异常与错误码建议

当前阶段建议至少区分以下错误类型：

### 11.1 关卡难度不存在或无效
适用场景：
- `stage_difficulty_id` 非法

### 11.2 战斗上下文无效
适用场景：
- `battle_context_id` 非法
- 上下文与角色 / 难度不匹配

### 11.3 战斗结果不合法
适用场景：
- `killed_monsters` 结构非法
- 通关标记与怪物结果明显冲突

### 11.4 掉落解析失败
适用场景：
- 掉落来源配置缺失
- 掉落组异常
- 掉落服务执行失败

### 11.5 首通奖励发放失败
适用场景：
- 满足首通条件但正式发奖失败

### 11.6 战斗结算失败
适用场景：
- 正式结算结果组装失败
- 关键子链返回不可接受状态

错误码可以后续统一整理，但当前阶段至少应保证错误原因清晰，不要统一返回“结算失败”。

---

## 12. 测试与自测要点

战斗结算链路至少应覆盖以下测试点：

1. 合法战斗结果可正常返回正式结算结果
2. 非法 `stage_difficulty_id` 不能进入结算
3. 非法或不匹配的 `battle_context_id` 不能进入结算
4. 掉落结果会根据怪物正式产生
5. 首通奖励只会在首次满足条件时正式发放一次
6. 首通奖励失败时会返回明确失败状态
7. 掉落中的装备对象进入背包时会正式实例化
8. 结算结果中的掉落、奖励、实例结果与正式状态一致
9. 同一首通来源不会因重复结算而重复发放奖励
10. 失败场景不会返回半成正式结算结果

## 13. 第一阶段实现建议顺序

建议按以下顺序实现战斗结算链路：

1. 实现 `getStageDifficultyById()`
2. 预留或实现 `getBattlePrepareContext()`
3. 实现 `getMonsterMapByIds()`
4. 实现 `getFirstClearRewardStatus()`
5. 实现 `assertBattleContextValid()`
6. 实现 `assertBattleResultValid()`
7. 实现 `validateBattleSettlementRequest()`
8. 实现 `extractKilledMonsterIds()`
9. 实现 `buildDropSettlementContext()`
10. 实现 `buildRewardSettlementContext()`
11. 对接 `DropResolverService`
12. 对接 `RewardGrantWorkflow`
13. 实现 `buildBattleSettlementPayload()`
14. 最后实现 `settleBattle()` Workflow

这样做的好处是：
- 先把结算校验与上下文组织写稳
- 再接掉落链和奖励链
- 最后统一收口到正式 Workflow

## 14. 当前阶段结论

1. 战斗结算链路必须只有一个正式 Workflow 入口
2. 战斗结算方法应按 Query / Config、Domain、Workflow 三层拆分
3. 掉落链、奖励链、入包链必须分链实现后再统一汇总
4. 战斗结算返回必须基于正式成功结果，而不是理论结果
5. 首通奖励失败时不得伪装成成功结算
6. 后续代码实现、测试、自测与 code review，应以本文档定义的方法结构为准
