## 2. 战斗准备方法设计总原则

### 2.1 战斗准备必须只有统一正式入口
战斗准备链路必须只有一个正式 Workflow 入口，不允许：

- 页面自己拼一份战斗上下文
- Controller 直接拼怪物和角色属性后返回
- 客户端直接绕过后端配置关系自己拼战斗对象

正式战斗准备结果必须由正式 Workflow 基于正式配置与正式状态统一生成。

### 2.2 战斗准备必须基于正式配置与正式状态
战斗准备链路必须统一读取：

- 关卡难度配置
- 关卡难度怪物绑定
- 怪物模板
- 角色主表
- 角色穿戴槽
- 装备实例
- 装备模板
- 职业配置

不允许基于页面临时状态、旧缓存或未落库状态直接构造正式战斗输入。

### 2.3 战斗准备必须输出正式准备结果
战斗准备的结果，不应只是“查几张表后临时返回一点数据”，而应是：

**一份可供战斗链直接消费的正式准备结果。**

它至少要明确：

- 当前进入的是哪个关卡难度
- 本次战斗用哪些怪物
- 当前角色的正式战斗属性是多少
- 本次战斗的关键上下文标识是什么

### 2.4 战斗准备不负责正式结算写入
战斗准备链路只负责“进入战斗前的正式输入生成”，不负责：

- 掉落生成
- 发奖
- 入包
- 写战斗结算结果

### 2.5 角色属性结果必须统一来源
战斗准备中的角色战斗属性，不允许在本链路中再手写第二套算法。

正式计算必须统一走：

- `CharacterStatService`

战斗准备只负责调用和组装，不负责重新定义属性口径。

## 3. 战斗准备链路目标与边界

## 3.1 链路目标
根据当前用户选择的关卡难度，生成一份完整、正式、可直接进入战斗的准备结果。

## 3.2 链路起点
客户端提交“进入战斗”请求，至少包含：

- `user_id`
- `character_id`
- `stage_difficulty_id`

## 3.3 链路终点
满足以下条件时，战斗准备链路才算完成：

1. 关卡难度已被正式校验
2. 怪物清单已按正式绑定关系读取
3. 角色当前正式状态已读取
4. 角色最终战斗属性已正式计算
5. 返回可供战斗链直接消费的正式准备结果

## 3.4 链路不负责内容
战斗准备链路不负责：

- 真实战斗过程驱动
- 掉落结算
- 奖励发放
- 入包
- 战斗结束记录写入

---

## 4. 方法分层建议

当前战斗准备链路建议拆为以下三层方法。

### 4.1 Query / Config 层
负责读取关卡难度、怪物绑定、怪物模板、角色状态、穿戴状态等正式输入。

建议方法：

- `getStageDifficultyById()`
- `getEnabledStageDifficultyById()`
- `getStageMonsterBindings()`
- `getMonsterMapByIds()`
- `getCharacterById()`
- `getCharacterSlotMap()`
- `getEquippedInstancesByCharacter()`
- `getEquipmentTemplateMapByItemIds()`

### 4.2 Domain 层
负责单点校验、上下文构造、怪物列表构造与结果组装。

建议方法：

- `validateBattlePrepareRequest()`
- `assertCharacterOwner()`
- `assertStageDifficultyEnabled()`
- `buildBattleMonsterList()`
- `buildBattleCharacterContext()`
- `buildBattlePreparePayload()`

### 4.3 Workflow 层
负责组织完整战斗准备流程。

建议方法：

- `prepareBattle()`

## 6. Workflow 方法设计

## 6.1 `prepareBattle()`

### 所属层级
Workflow

### 方法目标
作为战斗准备链路唯一正式入口，组织前置校验、配置读取、状态汇总、角色属性计算与正式准备结果组装。

### 建议签名
```php
public function prepareBattle(int $userId, int $characterId, string $stageDifficultyId): array
```

### 输入说明
- `userId`：当前用户 ID
- `characterId`：目标角色 ID
- `stageDifficultyId`：目标关卡难度 ID

### 输出说明
返回正式战斗准备结果，建议至少包含：

- `battle_context_id` 或等价上下文标识
- `stage_difficulty`
- `monster_list`
- `character`
- `character_stats`
- `slot_snapshot`
- `equipped_instances`

### 执行步骤
1. 调用 `validateBattlePrepareRequest()` 做前置校验
2. 读取关卡难度配置
3. 读取关卡难度怪物绑定
4. 读取怪物模板清单
5. 读取角色正式状态
6. 读取角色穿戴槽状态
7. 读取当前已穿戴装备实例
8. 读取对应装备模板
9. 调用 `CharacterStatService` 计算角色最终战斗属性
10. 调用 `buildBattleMonsterList()` 组装怪物列表
11. 调用 `buildBattleCharacterContext()` 组装角色上下文
12. 调用 `buildBattlePreparePayload()` 生成正式准备结果
13. 返回结果

### 事务要求
当前阶段战斗准备以读取为主，原则上不要求数据库事务写入。

若后续需要写战斗会话表或快照表，可再扩展事务边界。

### 异常处理要求
- 任一关键校验失败直接中断
- 不允许返回“半成战斗准备结果”
- 必须记录必要日志，至少包含：`user_id`、`character_id`、`stage_difficulty_id`

### 不负责内容
- 不执行掉落
- 不执行发奖
- 不写背包
- 不提交战斗结果

## 7. Domain 方法设计

## 7.1 `validateBattlePrepareRequest()`

### 所属层级
Domain

### 方法目标
校验战斗准备请求是否满足正式进入战斗的前置条件。

### 建议签名
```php
public function validateBattlePrepareRequest(int $userId, int $characterId, string $stageDifficultyId): void
```

### 必须校验的内容
1. 关卡难度存在且启用
2. 角色存在
3. 角色归属当前用户
4. 关卡难度绑定了有效怪物
5. 怪物模板数据完整可用

### 依赖方法
- `getEnabledStageDifficultyById()`
- `getCharacterById()`
- `assertCharacterOwner()`
- `assertStageDifficultyEnabled()`
- `getStageMonsterBindings()`
- `getMonsterMapByIds()`

### 不负责内容
- 不计算角色战斗属性
- 不组装返回结果

---

## 7.2 `assertCharacterOwner()`

### 所属层级
Domain

### 方法目标
校验目标角色是否归属当前用户。

### 建议签名
```php
public function assertCharacterOwner(int $userId, array|object $character): void
```

### 关键要求
- 若角色不归属当前用户，必须抛异常

---

## 7.3 `assertStageDifficultyEnabled()`

### 所属层级
Domain

### 方法目标
校验关卡难度是否可用于正式战斗准备。

### 建议签名
```php
public function assertStageDifficultyEnabled(array|object $stageDifficulty): void
```

### 当前阶段建议校验
- 关卡难度存在
- 关卡难度启用
- 关卡本体启用（如读取链中已含）
- 章节本体启用（如读取链中已含）

---

## 7.4 `buildBattleMonsterList()`

### 所属层级
Domain

### 方法目标
根据怪物绑定与怪物模板，构造本次战斗使用的正式怪物列表。

### 建议签名
```php
public function buildBattleMonsterList(array $bindings, array $monsterMap): array
```

### 输出内容建议
每个怪物节点至少包含：

- `monster_id`
- `monster_name`
- `monster_role`
- `wave_no`
- 基础战斗属性
- 攻击 / 行为关键字段
- `sort_order`

### 关键要求
- 必须按正式绑定顺序输出
- 不允许在此方法中重新定义怪物模板数值
- 只负责“本次战斗带哪些怪物、按什么顺序出场”

---

## 7.5 `buildBattleCharacterContext()`

### 所属层级
Domain

### 方法目标
将角色主表、穿戴槽、已穿戴实例、装备模板摘要与正式战斗属性整合成一份角色战斗上下文。

### 建议签名
```php
public function buildBattleCharacterContext(array|object $character, array $slotMap, array $equippedInstances, array $equipmentTemplateMap, array $characterStats): array
```

### 输出内容建议
至少包含：

- 角色基础信息
- 当前职业标识
- 当前等级
- 当前加点
- 当前槽位快照
- 当前已穿戴实例摘要
- 当前正式战斗属性

### 关键要求
- 不重新计算一套角色属性
- 正式战斗属性必须直接来自 `CharacterStatService`

---

## 7.6 `buildBattlePreparePayload()`

### 所属层级
Domain

### 方法目标
将战斗上下文、关卡难度与怪物列表整合成接口或上层可直接消费的正式准备结果。

### 建议签名
```php
public function buildBattlePreparePayload(array|object $stageDifficulty, array $monsterList, array $characterContext): array
```

### 建议输出内容
至少包含：

- `battle_context_id` 或等价上下文标识
- `stage_difficulty`
- `monster_list`
- `character`
- `character_stats`
- `slot_snapshot`
- `equipped_instances`

### 关键要求
- 返回结果必须基于正式读取与正式计算结果
- 不应拼接未进入正式链路的临时字段

---

## 8. Query / Config 方法设计

## 8.1 `getEnabledStageDifficultyById()`

### 所属层级
Config / Query

### 方法目标
查询已启用的关卡难度正式配置。

### 建议签名
```php
public function getEnabledStageDifficultyById(string $stageDifficultyId): ?array
```

### 查询要求
- 至少返回 `stage_difficulty_id`、`stage_id`、`difficulty_key`、`difficulty_name`
- 只返回启用状态对象

---

## 8.2 `getStageMonsterBindings()`

### 所属层级
Query

### 方法目标
查询某个关卡难度下的正式怪物绑定。

### 建议签名
```php
public function getStageMonsterBindings(string $stageDifficultyId): array
```

### 查询要求
- 至少返回 `monster_id`、`monster_role`、`wave_no`、`sort_order`
- 只返回启用绑定
- 按 `wave_no`、`sort_order` 排序

---

## 8.3 `getMonsterMapByIds()`

### 所属层级
Config / Query

### 方法目标
批量查询怪物模板，并返回以 `monster_id` 为 key 的映射结构。

### 建议签名
```php
public function getMonsterMapByIds(array $monsterIds): array
```

### 查询要求
- 只返回启用怪物
- 返回构造战斗怪物列表所需的正式模板字段

---

## 8.4 `getCharacterById()`

### 所属层级
Query

### 方法目标
查询角色正式状态。

### 建议签名
```php
public function getCharacterById(int $characterId): ?array
```

### 查询要求
- 至少返回 `character_id`、`user_id`、`class_id`、`level`

## 8.5 `getCharacterSlotMap()`

Query

public function getCharacterSlotMap(int $characterId): array

### 查询要求
- 返回以 `slot_key` 为 key 的映射结构

---

## 8.6 `getEquippedInstancesByCharacter()`

Query

### 方法目标

## 9. 方法调用关系建议

建议调用顺序如下：

```text
prepareBattle()
 ├─ validateBattlePrepareRequest()
 │   ├─ getEnabledStageDifficultyById()
 │   ├─ getCharacterById()
 │   ├─ assertCharacterOwner()
 │   ├─ assertStageDifficultyEnabled()
 │   ├─ getStageMonsterBindings()
 │   └─ getMonsterMapByIds()
 ├─ getStageMonsterBindings()
 ├─ getMonsterMapByIds()
 ├─ getCharacterById()
 ├─ getCharacterSlotMap()
 ├─ getEquippedInstancesByCharacter()
 ├─ getEquipmentTemplateMapByItemIds()
 ├─ CharacterStatService::calculateCharacterStats(...)
 ├─ buildBattleMonsterList()
 ├─ buildBattleCharacterContext()
 └─ buildBattlePreparePayload()
```

这个结构的目的，是把：

- 前置校验
- 配置读取
- 状态汇总
- 角色属性计算
- 正式结果组装

彻底拆开，避免一个方法里同时塞进所有读取、校验、计算和拼装逻辑。

## 10. 快照与准备结果边界设计

### 10.1 当前阶段最低要求
当前阶段即使不单独落一张“战斗会话表”，也必须保证战斗准备接口输出的是：

**一份正式准备结果**

而不是临时拼几个字段给前端凑合用。

### 10.2 正式准备结果至少应固定的内容
- 当前关卡难度标识
- 当前怪物列表
- 当前角色战斗属性
- 当前角色穿戴快照摘要
- 本次战斗上下文标识

### 10.3 后续扩展方向
后续若进入更复杂战斗链路，可扩展为：

- 持久化战斗会话表
- 持久化战斗准备快照表
- 结算时校验上下文一致性

但第一阶段先不强制建表。

## 11. 异常与错误码建议

当前阶段建议至少区分以下错误类型：

### 11.1 关卡难度不存在或未启用
适用场景：
- `stage_difficulty_id` 无效
- 关卡难度已禁用

### 11.2 角色不存在
适用场景：
- `character_id` 无效

### 11.3 角色不归属当前用户
适用场景：
- 访问他人角色

### 11.4 关卡难度怪物绑定为空
适用场景：
- 没有配置怪物
- 绑定全部被禁用

### 11.5 怪物模板缺失或未启用
适用场景：
- 绑定存在但模板无效

### 11.6 战斗准备失败
适用场景：
- 关键配置读取失败
- 角色属性计算失败
- 正式准备结果组装失败

错误码可以后续统一整理，但当前阶段至少应保证错误原因可辨别，不要统一返回“进入战斗失败”。

---

## 12. 测试与自测要点

战斗准备链路至少应覆盖以下测试点：

1. 合法关卡难度可正常返回正式准备结果
2. 非法关卡难度不能进入战斗准备
3. 他人角色不能进入战斗准备
4. 未绑定怪物的关卡难度不能进入战斗准备
5. 禁用怪物模板不会被正常返回到怪物列表
6. 角色换装后，战斗准备中的角色战斗属性会正式变化
7. 角色加点变化后，战斗准备中的角色战斗属性会正式变化
8. 战斗准备返回的怪物顺序与正式绑定顺序一致
9. 战斗准备结果中的角色上下文与正式穿戴状态一致
10. 失败场景不会返回半成准备结果

## 13. 第一阶段实现建议顺序

建议按以下顺序实现战斗准备链路：

1. 实现 `getEnabledStageDifficultyById()`
2. 实现 `getStageMonsterBindings()`
3. 实现 `getMonsterMapByIds()`
4. 实现 `getCharacterById()`
5. 实现 `getCharacterSlotMap()`
6. 实现 `getEquippedInstancesByCharacter()`
7. 实现 `getEquipmentTemplateMapByItemIds()`
8. 实现 `assertCharacterOwner()`
9. 实现 `assertStageDifficultyEnabled()`
10. 实现 `validateBattlePrepareRequest()`
11. 对接 `CharacterStatService`
12. 实现 `buildBattleMonsterList()`
13. 实现 `buildBattleCharacterContext()`
14. 实现 `buildBattlePreparePayload()`
15. 最后实现 `prepareBattle()` Workflow

这样做的好处是：
- 先把读取和校验写稳
- 再接属性计算
- 最后统一收口到 Workflow

---

## 14. 当前阶段结论

1. 战斗准备链路必须只有一个正式 Workflow 入口
2. 战斗准备方法应按 Query / Config、Domain、Workflow 三层拆分
3. 战斗准备必须基于正式配置与正式状态生成正式准备结果
4. 角色战斗属性必须统一来自 `CharacterStatService`，不允许再手写第二套口径
5. 战斗准备失败时不得返回半成正式结果
6. 后续代码实现、测试、自测与 code review，应以本文档定义的方法结构为准
