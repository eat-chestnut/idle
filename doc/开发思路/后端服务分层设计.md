

# 《山海巡厄录》后端服务分层设计

## 1. 文档目的

本文档用于在《核心开发总纲》《模块拆分与职责清单》基础上，进一步明确《山海巡厄录》项目后端服务层的分层方式、各层职责边界、命名建议、依赖方向与当前阶段的落地重点。

本文档重点不是讨论数据库字段，也不是重复系统玩法规则，而是解决以下开发问题：

- 后端服务到底应该分几层
- 每一层服务负责什么，不负责什么
- 哪些逻辑应该放在 Query 层，哪些应该放在 Workflow 层
- 如何避免 Controller 过重、Service 巨石化、逻辑四处散落
- 第一阶段应该先落哪些服务，后落哪些服务

本文档定位为：

- 后端服务拆分基线文档
- Service 命名与职责边界文档
- 业务流程服务组织文档
- 后续《核心业务链路设计》《接口开发清单》《后台开发清单》的服务层前置文档

---

## 2. 服务分层总原则

### 2.1 先分职责，再分文件
服务分层必须优先依据：

- 是配置读取还是状态读取
- 是单点写入还是流程编排
- 是业务规则判断还是后台校验
- 是对外接口适配还是内部业务执行

不允许因为“方便写”就把多层职责塞进同一个 Service。

### 2.2 读写分离
服务层必须尽量区分：

- 读取服务
- 写入服务
- 流程服务

读取服务负责稳定返回结构；写入服务负责状态变更；流程服务负责组织多步业务链路。

### 2.3 模板读取与状态读取分离
必须区分：

- 模板配置读取
- 玩家状态读取

例如：

- 职业、物品、装备模板、怪物、掉落组、奖励组属于配置读取
- 角色、背包、装备实例、穿戴槽、发奖记录属于状态读取

### 2.4 单点动作与流程编排分离
以下两类服务必须分开：

- 单点能力服务：负责某一个明确动作
- 流程编排服务：负责组织多个单点动作形成完整业务链

例如：

- `InventoryWriteService` 负责入包
- `RewardGrantService` 负责发奖流程
- `BattleSettlementService` 负责结算流程

### 2.5 后台逻辑与正式业务逻辑分离
后台侧需要的：

- 配置合法性校验
- 引用完整性检查
- 补发重试
- 数据修复

不得直接混进玩家正式流程服务中。

---

## 3. 后端服务总分层

当前项目建议后端服务按以下五层拆分。

### 3.1 Config 层
负责读取静态配置。

### 3.2 Query 层
负责读取玩家当前状态与业务查询结果。

### 3.3 Domain 层
负责单点业务动作与领域规则执行。

### 3.4 Workflow 层
负责跨多个 Domain / Query / Config 服务的流程编排。

### 3.5 Admin 层
负责后台管理逻辑、配置校验、补发与修复类能力。

---

## 4. 各层职责定义

## 4.1 Config 层

### 4.1.1 负责内容
Config 层只负责读取静态配置对象，例如：

- 职业配置
- 物品配置
- 装备模板配置
- 主线章节配置
- 关卡配置
- 关卡难度配置
- 怪物配置
- 掉落组配置
- 奖励组配置

### 4.1.2 不负责内容
Config 层不负责：

- 玩家状态写入
- 事务处理
- 入包
- 发奖
- 掉落抽取执行
- 战斗结算

### 4.1.3 返回要求
Config 层返回值必须尽量稳定、纯净、可复用。

不应在 Config 层中混入：

- 玩家当前持有状态
- 当前角色上下文
- Controller 直出格式拼接

### 4.1.4 命名建议
建议命名统一为：

- `ClassConfigService`
- `ItemConfigService`
- `EquipmentTemplateConfigService`
- `ChapterConfigService`
- `StageConfigService`
- `StageDifficultyConfigService`
- `MonsterConfigService`
- `DropConfigService`
- `RewardConfigService`

---

## 4.2 Query 层

### 4.2.1 负责内容
Query 层负责读取玩家当前状态与业务查询结果，例如：

- 角色当前信息
- 当前穿戴槽状态
- 可堆叠背包状态
- 装备实例状态
- 首通奖励发放状态
- 发奖记录查询

### 4.2.2 不负责内容
Query 层不负责：

- 状态写入
- 奖励发放
- 入包
- 事务处理
- 复杂流程编排

### 4.2.3 适用场景
以下场景应优先走 Query 层：

- 角色面板查询
- 背包列表查询
- 装备实例详情查询
- 穿戴状态查询
- 首通奖励状态查询
- 后台发奖记录查询

### 4.2.4 命名建议
建议命名统一为：

- `CharacterQueryService`
- `CharacterEquipmentSlotQueryService`
- `InventoryStackQueryService`
- `EquipmentInstanceQueryService`
- `RewardGrantQueryService`
- `StageProgressQueryService`

---

## 4.3 Domain 层

### 4.3.1 负责内容
Domain 层负责执行单点业务动作与领域规则。

它解决的问题是：

- 某个明确动作应该怎么做
- 某个单点规则应该怎么校验
- 某个单点状态应该如何更新

### 4.3.2 适合放在 Domain 层的能力
例如：

- 创建角色
- 初始化穿戴槽
- 穿装备
- 卸装备
- 创建装备实例
- 可堆叠物入包
- 校验首通奖励是否已发
- 写入发奖记录
- 写入发奖明细
- 计算角色属性结果
- 执行掉落组抽取

### 4.3.3 不负责内容
Domain 层不负责：

- 跨多步长流程编排
- Controller 响应结构拼装
- 后台管理逻辑
- 页面表格查询适配

### 4.3.4 命名建议
建议命名统一为：

- `CharacterCreateService`
- `CharacterEquipmentSlotInitService`
- `EquipmentWearService`
- `EquipmentUnequipService`
- `InventoryWriteService`
- `EquipmentInstanceCreateService`
- `CharacterStatService`
- `DropResolverService`
- `RewardGrantRecordService`
- `RewardGrantItemService`

---

## 4.4 Workflow 层

### 4.4.1 负责内容
Workflow 层负责跨多个服务组织完整业务流程。

它解决的问题是：

- 某条完整业务链从头到尾怎么跑
- 哪些步骤要先后执行
- 哪些步骤要放在同一事务中
- 失败时如何回滚

### 4.4.2 适合放在 Workflow 层的能力
例如：

- 角色创建完整流程
- 战斗准备完整流程
- 战斗结算完整流程
- 首通奖励发放完整流程
- 换装完整流程
- 发奖失败补发流程

### 4.4.3 不负责内容
Workflow 层不应自己实现所有底层细节。

它应负责：

- 组织
- 调度
- 事务边界
- 结果汇总

它不应负责：

- 重新实现单点规则
- 重新写 SQL
- 重新定义配置结构

### 4.4.4 命名建议
建议命名统一为：

- `CharacterCreateWorkflow`
- `BattlePrepareWorkflow`
- `BattleSettlementWorkflow`
- `RewardGrantWorkflow`
- `EquipmentChangeWorkflow`
- `RewardRetryWorkflow`

---

## 4.5 Admin 层

### 4.5.1 负责内容
Admin 层负责后台相关的管理逻辑。

包括：

- 配置保存前合法性校验
- 配置引用完整性检查
- 玩家状态后台查看适配
- 发奖失败重试
- 数据修复与临时工具

### 4.5.2 不负责内容
Admin 层不直接替代正式玩家业务链。

例如：

- 不直接替代正式发奖流程
- 不直接替代正式入包流程
- 不直接替代正式战斗结算流程

后台触发正式动作时，应调用正式 Domain / Workflow，而不是自己复制一套逻辑。

### 4.5.3 命名建议
建议命名统一为：

- `AdminConfigValidationService`
- `AdminReferenceCheckService`
- `AdminRewardRetryService`
- `AdminDataRepairService`
- `AdminPlayerStateQueryService`

## 5. 依赖方向原则

### 5.1 推荐依赖方向
推荐依赖方向如下：

- Controller / Command → Workflow / Query / Admin
- Workflow → Domain + Query + Config
- Domain → Query + Config + Repository / Model
- Query → Repository / Model
- Config → Repository / Model
- Admin → Query + Domain + Workflow

### 5.2 禁止依赖方向
以下方向应尽量避免：

- Config 依赖 Workflow
- Query 依赖 Workflow
- Domain 依赖 Controller
- Model 依赖 Service
- Admin 复制一套正式业务逻辑实现

### 5.3 Controller 职责边界
Controller 只负责：

- 接收参数
- 调用服务
- 返回结果

Controller 不负责：

- 复杂业务判断
- 事务控制
- 发奖防重
- 掉落抽取
- 属性重算算法

## 9. 高风险逻辑应该放在哪一层

### 9.1 首通奖励防重
应放在：

- `RewardGrantWorkflow`
- `RewardGrantRecordService`
- `RewardGrantQueryService`

不应放在：

- Controller
- 后台页面回调

### 9.2 掉落抽取
应放在：

- `DropResolverService`

不应放在：

- `MonsterConfigService`
- `BattleSettlementWorkflow` 的内部硬编码

### 9.3 装备穿戴校验
应放在：

- `EquipmentWearService`
- `EquipmentUnequipService`
- `EquipmentChangeWorkflow`

不应放在：

- Controller
- 前端页面

### 9.4 入包逻辑
应放在：

- `InventoryWriteService`
- `EquipmentInstanceCreateService`

不应放在：

- 奖励服务里直接写表
- 掉落服务里直接写表

### 9.5 属性重算
应放在：

- `CharacterStatService`

不应放在：

- `CharacterQueryService`
- `Character` Model 内部硬编码过多算法

## 10. 当前阶段必须避免的服务层误区

### 10.1 误区一：一个 Service 什么都做
例如把：

- 读取配置
- 校验条件
- 写状态
- 拼返回

全部塞进一个大 Service。

### 10.2 误区二：Controller 直接操作多个 Model 完成完整业务
这会导致：

- 事务难控
- 逻辑难复用
- 测试困难

### 10.3 误区三：Workflow 重写所有底层逻辑
Workflow 应该组织流程，不应重写所有 Domain 规则。

### 10.4 误区四：Admin 层复制正式业务逻辑
后台要复用正式业务链，而不是复制一套“后台专用发奖逻辑”。

### 10.5 误区五：Query 层偷偷写状态
Query 层必须保持“只读”。

## 11. 当前阶段结论

1. 当前项目后端建议固定分为 Config、Query、Domain、Workflow、Admin 五层
2. 各层职责已经明确，必须避免跨层滥用
3. Config 层负责静态配置读取，Query 层负责状态查询，Domain 层负责单点业务动作，Workflow 层负责流程编排，Admin 层负责后台逻辑
4. 高风险链路必须通过 Workflow + Domain 组织，不得散落在 Controller 中
5. 第一阶段应优先落 Config、Query、核心 Domain，再逐步补 Workflow 与 Admin
6. 六条主链已经能够映射到明确的服务层结构
7. 后续所有方法级开发设计，都应建立在本服务分层之上

本文档可直接作为项目正式后端服务分层文档使用，并作为后续《核心业务链路设计》《后台开发清单》《高风险点与约束清单》《接口开发清单》的统一服务层基线。
