# 《山海巡厄录》入包方法设计

## 1. 文档目的

本文档用于在《核心开发总纲》《模块拆分与职责清单》《后端服务分层设计》《核心业务链路设计》《第一阶段六条主链清单》《高风险点与约束清单》《接口开发清单》《战斗结算方法设计》《掉落解析方法设计》《奖励发放方法设计》基础上，进一步下沉到方法级，明确《山海巡厄录》第一阶段“入包链路”的方法拆分、方法职责、输入输出、对象分流规则、事务边界、异常处理与实现顺序。

本文档重点不是重复背包系统规则，而是回答以下问题：

- 入包链路到底应该拆成哪些方法
- 掉落结果与奖励结果进入入包链后如何统一承接
- 可堆叠物与装备对象为什么必须分流处理
- 装备实例应在什么环节创建
- 哪些步骤必须放在同一事务中
- 哪些异常应该直接中断正式入包

本文档定位为：

- 入包链路的方法级设计文档
- 掉落链与奖励链的正式承接文档
- 背包写入、自测与 code review 的结构基线
- 第一阶段核心写入链路的关键闭环文档

## 2. 入包方法设计总原则

### 2.1 入包必须只有统一正式入口
入包链路必须只有统一正式入口，不允许：

- 掉落链自己直接写 `inventory_stack_items`
- 奖励链自己直接写 `inventory_equipment_instances`
- 后台工具绕过正式服务直接改背包

所有正式产出对象进入玩家持有状态，都必须统一走入包服务。

### 2.2 入包必须只承接“正式产出结果”
入包链路不负责决定玩家应该获得什么，而只负责承接已经被正式确认的产出结果。

也就是说，入包的输入应该来自：

- 正式掉落解析结果
- 正式奖励发放结果
- 后续正式补发结果

而不是来自页面随意拼接的数据。

### 2.3 可堆叠物与装备对象必须分流处理
当前阶段入包必须明确分成两条承接路径：

1. 可堆叠物承接路径
2. 装备对象承接路径

因为二者在正式状态层的落点不同：

- 可堆叠物 → `inventory_stack_items`
- 装备对象 → `inventory_equipment_instances`

### 2.4 装备对象必须实例化后才算正式入包
掉落链和奖励链输出的装备对象，本质上仍然是模板命中结果。

只有在入包链路中完成：

- 实例创建
- 默认字段初始化

之后，才算正式进入玩家持有状态。

### 2.5 入包结果必须可追溯
正式入包至少应能回答：

- 本次入包的是哪批正式对象
- 哪些对象进入了可堆叠表
- 哪些对象被创建成装备实例
- 为什么失败
- 是否存在部分成功

因此入包结果必须是标准化可追溯结果，而不是只返回一句“成功”。

## 3. 入包链路目标与边界

## 3.1 链路目标
将正式产出结果安全写入玩家正式持有状态，并按对象类型分别落到可堆叠背包表或装备实例表。

## 3.2 链路起点
上游链路传入正式产出对象列表，至少每条包含：

- `item_id`
- `quantity`
- 来源上下文（可选但建议保留）

## 3.3 链路终点
满足以下条件时，入包链路才算完成：

1. 所有可堆叠物已正确写入或累加到 `inventory_stack_items`
2. 所有装备对象已正确创建为 `inventory_equipment_instances`
3. 返回正式入包结果与实例创建结果

## 3.4 链路不负责内容
入包链路不负责：

- 掉落抽取
- 首通判定
- 奖励防重判断
- 角色属性计算
- 装备穿戴

---

## 4. 方法分层建议

当前入包链路建议拆为以下三层方法。

### 4.1 Query / Config 层
负责读取物品主表、装备模板与玩家当前可堆叠持有状态。

建议方法：

- `getItemMapByIds()`
- `getEquipmentTemplateMapByItemIds()`
- `getExistingStackItems()`

### 4.2 Domain 层
负责输入校验、对象分流、可堆叠物写入、装备实例创建、结果组装。

建议方法：

- `validateInventoryWriteContext()`
- `splitInventoryObjects()`
- `buildStackWriteRows()`
- `applyStackWrites()`
- `buildEquipmentInstancePayloads()`
- `createEquipmentInstances()`
- `buildInventoryWriteResult()`

### 4.3 Workflow / Facade 层
负责组织完整入包流程。

建议方法：

- `write()`
- `writeRewards()`
- `writeDrops()`

其中：
- `write()` 作为统一正式入口
- `writeRewards()` / `writeDrops()` 作为上游语义化包装入口，可内部复用 `write()`

## 6. Workflow / Facade 方法设计

## 6.1 `write()`

### 所属层级
Workflow / Facade

### 方法目标
作为入包链路唯一正式入口，组织输入校验、对象分流、可堆叠物写入、装备实例创建与正式结果组装。

### 建议签名
```php
public function write(int $userId, array $objects, array $context = []): array
```

### 输入说明
- `userId`：当前用户 ID
- `objects`：正式产出对象列表，建议每条至少包含：
  - `item_id`
  - `quantity`
  - 可选来源元信息
- `context`：本次入包上下文，例如：
  - `source`（drop / reward / retry）
  - `source_type`
  - `source_id`
  - `reward_grant_id`（如适用）
  - `battle_context_id`（如适用）

### 输出说明
返回正式入包结果，建议至少包含：

- `stack_results`
- `equipment_instance_results`
- `created_equipment_instances`
- `written_item_ids`
- `summary`

### 执行步骤
1. 调用 `validateInventoryWriteContext()` 校验上下文与对象列表
2. 批量读取物品主表 `getItemMapByIds()`
3. 按对象类型调用 `splitInventoryObjects()` 做正式分流
4. 若存在可堆叠物，读取当前已有持有状态 `getExistingStackItems()`
5. 调用 `buildStackWriteRows()` 生成可堆叠物写入计划
6. 若存在装备对象，读取装备模板 `getEquipmentTemplateMapByItemIds()`
7. 调用 `buildEquipmentInstancePayloads()` 生成实例写入计划
8. 开启事务
9. 调用 `applyStackWrites()` 写入可堆叠物结果
10. 调用 `createEquipmentInstances()` 写入装备实例结果
11. 调用 `buildInventoryWriteResult()` 组装正式入包结果
12. 提交事务
13. 返回结果

### 事务要求
以下步骤必须放在同一正式事务中：

- 可堆叠物写入
- 装备实例创建
- 正式结果组装所依赖的核心写入完成

### 异常处理要求
- 任一关键校验失败必须直接中断
- 任一正式写入失败必须整体回滚
- 不允许可堆叠物入包成功但装备实例创建失败后仍返回成功
- 必须记录必要日志，至少包含：`user_id`、`source`、`source_type`、`source_id`

### 不负责内容
- 不判断首通条件
- 不判断奖励是否已发
- 不判断掉落是否应该命中

## 7. Domain 方法设计

## 7.1 `validateInventoryWriteContext()`

### 所属层级
Domain

### 方法目标
校验入包上下文与正式产出对象列表是否合法。

### 建议签名
```php
public function validateInventoryWriteContext(int $userId, array $objects, array $context = []): void
```

### 必须校验的内容
1. `user_id` 合法
2. `objects` 为数组且非空
3. 每条对象至少包含 `item_id`、`quantity`
4. `quantity` 必须为正整数
5. 来源上下文（如有）格式合法

### 不负责内容
- 不读取物品配置
- 不执行分流
- 不写入数据库

---

## 7.2 `splitInventoryObjects()`

### 所属层级
Domain

### 方法目标
根据物品主表类型，将正式产出对象分流为：

- 可堆叠物列表
- 装备对象列表

### 建议签名
```php
public function splitInventoryObjects(array $objects, array $itemMap): array
```

### 输出内容建议
返回结构建议包含：

- `stack_items`
- `equipment_items`

### 当前阶段规则
- `item_type = equipment` 的对象必须进入装备承接路径
- `material` / `reward_item` / 特殊代币等进入可堆叠承接路径
- 普通货币不进入本链路

### 不负责内容
- 不执行具体写入
- 不创建实例

---

## 7.3 `buildStackWriteRows()`

### 所属层级
Domain

### 方法目标
根据当前持有状态与本次产出对象，构造可堆叠物的正式写入计划。

### 建议签名
```php
public function buildStackWriteRows(int $userId, array $stackItems, array $existingStacks): array
```

### 输出内容建议
每条计划至少包含：

- `user_id`
- `item_id`
- `before_quantity`
- `add_quantity`
- `after_quantity`
- `action`（create / increment）

### 关键要求
- 必须基于 `UNIQUE(user_id, item_id)` 的总量模式设计
- 不能生成多条相同 `user_id + item_id` 的重复创建计划

---

## 7.4 `applyStackWrites()`

### 所属层级
Domain

### 方法目标
执行可堆叠物正式写入。

### 建议签名
```php
public function applyStackWrites(array $rows): array
```

### 输出说明
返回正式写入结果，建议每条至少包含：

- `item_id`
- `before_quantity`
- `after_quantity`
- `action`

### 关键要求
- 必须统一处理 create / increment 两类动作
- 必须在正式事务内调用

---

## 7.5 `buildEquipmentInstancePayloads()`

### 所属层级
Domain

### 方法目标
根据装备对象列表与装备模板，构造正式装备实例创建数据。

### 建议签名
```php
public function buildEquipmentInstancePayloads(int $userId, array $equipmentItems, array $equipmentTemplateMap, array $context = []): array
```

### 当前阶段关键规则
每个装备对象必须生成独立实例，不允许：

- 把装备写入可堆叠表
- 把多个装备合并成一条实例

### 每条实例建议字段
至少包含：

- `user_id`
- `item_id`
- `bind_type`
- `enhance_level`
- `durability`
- `max_durability`
- `random_seed`
- `is_locked`
- `extra_attributes`

### 当前阶段默认值建议
- `bind_type = unbound`（如当前口径如此）
- `enhance_level = 0`
- `durability = 模板默认耐久或 0`
- `max_durability = 模板默认耐久或 0`
- `random_seed = 生成正式随机种子`
- `is_locked = 0`
- `extra_attributes = null`

---

## 7.6 `createEquipmentInstances()`

### 所属层级
Domain

### 方法目标
正式创建装备实例。

### 建议签名
```php
public function createEquipmentInstances(array $payloads): array
```

### 输出说明
返回正式实例创建结果，建议每条至少包含：

- `equipment_instance_id`
- `item_id`
- `user_id`
- `bind_type`
- `enhance_level`

### 关键要求
- 必须支持批量创建
- 必须在正式事务中执行
- 若已存在独立 `EquipmentInstanceCreateService`，应统一复用，不重复造轮子

---

## 7.7 `buildInventoryWriteResult()`

### 所属层级
Domain

### 方法目标
将可堆叠物写入结果与装备实例创建结果整合成正式入包结果。

### 建议签名
```php
public function buildInventoryWriteResult(array $stackResults = [], array $instanceResults = [], array $context = []): array
```

### 建议输出内容
至少包含：

- `stack_results`
- `equipment_instance_results`
- `created_equipment_instances`
- `written_item_ids`
- `summary`

### 关键要求
- 返回值必须基于正式成功写入结果
- 不应把理论对象当作正式入包结果返回

---

## 8. Query / Config 方法设计

## 8.1 `getItemMapByIds()`

### 所属层级
Config / Query

### 方法目标

Config / Query

## 9. 方法调用关系建议

建议调用顺序如下：

```text
write()
 ├─ validateInventoryWriteContext()
 ├─ getItemMapByIds()
 ├─ splitInventoryObjects()
 ├─ getExistingStackItems()（有可堆叠物时）
 ├─ buildStackWriteRows()
 ├─ getEquipmentTemplateMapByItemIds()（有装备对象时）
 ├─ buildEquipmentInstancePayloads()
 ├─ applyStackWrites()
 ├─ createEquipmentInstances()
 └─ buildInventoryWriteResult()
```

这个结构的目的，是把：

- 输入校验
- 物品分流
- 可堆叠物承接
- 装备实例承接
- 正式结果组装

彻底拆开，避免一个方法里同时查配置、分流、写背包、建实例、拼返回。

---

## 10. 分流、实例化与事务边界设计

### 10.1 分流边界
当前阶段必须先分流再写入：

- 可堆叠物 → `inventory_stack_items`
- 装备对象 → `inventory_equipment_instances`

### 10.2 实例化边界
装备对象只有在：

- 正式创建 `inventory_equipment_instances`
- 初始化默认实例字段

之后，才算正式进入玩家持有状态。

### 10.3 正式事务边界
以下步骤必须放在同一事务中：

- 可堆叠物正式写入
- 装备实例正式创建
- 正式结果依赖的核心写入完成

### 10.4 当前阶段禁止出现的状态
- 可堆叠物已入包，但装备实例创建失败仍返回成功
- 装备对象被错误写入可堆叠表
- 同一 `user_id + item_id` 被创建出多条可堆叠记录
- 理论实例结果被当作正式实例返回

---

## 11. 异常与错误码建议

当前阶段建议至少区分以下错误类型：

### 11.1 入包上下文不合法
适用场景：
- `objects` 为空
- `quantity` 非法

### 11.2 物品不存在或无效
适用场景：
- `item_id` 不存在
- 物品未启用

### 11.3 装备模板不存在或无效
适用场景：
- 装备对象存在，但模板缺失

### 11.4 可堆叠物写入失败
适用场景：
- 创建失败
- 累加失败

### 11.5 装备实例创建失败
适用场景：
- 实例写入失败
- 默认字段初始化失败

### 11.6 入包失败
适用场景：
- 正式事务失败
- 正式结果组装失败

错误码可以后续统一整理，但当前阶段至少应保证错误原因清晰，不要统一返回“入包失败”。

---

## 12. 测试与自测要点

入包链路至少应覆盖以下测试点：

1. 合法可堆叠物可正确创建或累加
2. 同一 `user_id + item_id` 不会生成重复可堆叠记录
3. 装备对象不会被写进可堆叠表
4. 装备对象会正确创建独立实例
5. 实例默认字段会按正式口径初始化
6. 混合对象列表可同时完成可堆叠物写入与实例创建
7. 任一核心写入失败时整体会回滚
8. 返回结果中的可堆叠写入结果与实例结果与正式状态一致
9. 掉落链与奖励链都可复用本入包链
10. 后续补发也可复用本入包链

---

## 13. 第一阶段实现建议顺序

建议按以下顺序实现入包链路：

1. 实现 `validateInventoryWriteContext()`
2. 实现 `getItemMapByIds()`
3. 实现 `splitInventoryObjects()`
4. 实现 `getExistingStackItems()`
5. 实现 `buildStackWriteRows()`
6. 实现 `applyStackWrites()`
7. 实现 `getEquipmentTemplateMapByItemIds()`
8. 实现 `buildEquipmentInstancePayloads()`
9. 对接或实现 `createEquipmentInstances()`
10. 实现 `buildInventoryWriteResult()`
11. 最后实现 `write()`
12. 最后补 `writeRewards()` / `writeDrops()`

这样做的好处是：
- 先把对象分流与两条承接路径写稳
- 再统一收口到正式入包入口
- 更方便被掉落链、奖励链、补发链复用

---

## 14. 当前阶段结论

1. 入包链路必须只有一个正式入口，不得散落实现
2. 入包方法应按 Query / Config、Domain、Workflow / Facade 三层拆分
3. 可堆叠物承接与装备实例承接必须先分流再写入
4. 装备对象必须实例化后才算正式入包
5. 入包失败时不得返回半成正式结果
6. 后续代码实现、测试、自测与 code review，应以本文档定义的方法结构为准
