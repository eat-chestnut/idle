## 2. 换装方法设计总原则

### 2.1 换装必须只有统一正式入口
换装链路必须只有正式 Workflow 入口，不允许：

- 前端直接拼槽位状态后提交最终结果
- Controller 直接改 `character_equipment_slots`
- 后台工具绕开正式服务直接写槽位

### 2.2 穿戴与卸下都属于正式状态写入
换装不是页面行为，而是正式状态变更。必须围绕：

- `character_equipment_slots`
- `inventory_equipment_instances`
- `equipments`

进行正式写入与正式校验。

### 2.3 模板层与实例层必须分离
换装时必须明确：

- 穿戴的是装备实例 `equipment_instance_id`
- 校验规则来自装备模板 `item_id -> equipments`

不允许把模板对象直接写入槽位。

### 2.4 主副武器联动必须统一处理
以下逻辑必须集中在正式换装服务中处理：

- 双手武器穿戴时清空副武器槽
- 副武器穿戴前校验主武器是否允许副武器
- 更换主武器后重新校验副武器是否仍然合法

### 2.5 正式状态优先于返回结果
换装实现时，必须优先保证：

- 槽位状态正确
- 实例占用关系正确
- 主副武器联动正确

然后再组装返回结果。

### 2.6 异常必须阻断，不允许写半成状态
换装中任一步关键校验失败，都不应继续执行写入。

不允许出现：

- 主武器已更新，但副武器未按规则卸下
- 一个装备实例被多个槽位同时占用
- 替换写了一半就返回成功

## 3. 换装链路目标与边界

## 3.1 链路目标
让角色能够安全、合法地穿戴或卸下装备实例，并使穿戴槽状态与实例占用关系保持一致。

## 3.2 链路起点
客户端发起以下任一请求：

- 穿戴某个装备实例到某个槽位
- 卸下某个槽位当前装备

## 3.3 链路终点
满足以下条件时，换装链路才算完成：

1. 目标槽位状态已正式更新
2. 主副武器联动规则已正确处理
3. 角色穿戴状态可被正式查询
4. 后续角色属性重算输入已稳定

## 3.4 链路不负责内容
换装链路不负责：

- 创建装备实例
- 装备强化、洗练、宝石、套装等成长逻辑
- 掉落结算
- 奖励发放
- 战斗结果计算

---

## 4. 方法分层建议

当前换装链路建议拆为以下三层方法。

### 4.1 Query 层
负责读取角色、槽位、装备实例、装备模板等前置条件。

建议方法：

- `getCharacterById()`
- `getCharacterSlotMap()`
- `getSlotRowByKey()`
- `getEquipmentInstanceById()`
- `getEquipmentTemplateByItemId()`
- `isEquipmentInstanceEquipped()`

### 4.2 Domain 层
负责单点校验、联动判断、槽位写入与结果组装。

建议方法：

- `validateEquipRequest()`
- `validateUnequipRequest()`
- `assertInstanceOwner()`
- `assertSlotCompatible()`
- `assertLevelRequirement()`
- `assertSubWeaponCompatible()`
- `buildEquipPlan()`
- `applyEquipPlan()`
- `clearSlotInstance()`
- `buildEquipmentChangeResult()`

### 4.3 Workflow 层
负责组织完整穿戴 / 卸下流程。

建议方法：

- `equip()`
- `unequip()`

## 6. Workflow 方法设计

## 6.1 `equip()`

### 所属层级
Workflow

### 方法目标
作为正式穿戴入口，组织前置校验、联动方案生成、槽位写入与结果返回。

### 建议签名
```php
public function equip(int $userId, int $characterId, int $equipmentInstanceId, string $targetSlotKey): array
```

### 输入说明
- `userId`：当前用户 ID
- `characterId`：目标角色 ID
- `equipmentInstanceId`：目标装备实例 ID
- `targetSlotKey`：目标槽位键

### 输出说明
返回正式换装结果，建议至少包含：

- `character_id`
- `changed_slots`
- `equipped_instance_id`
- `unequipped_instance_ids`
- `slot_snapshot`

### 执行步骤
1. 调用 `validateEquipRequest()` 做前置校验
2. 读取角色、槽位、装备实例、装备模板
3. 生成正式换装计划 `buildEquipPlan()`
4. 开启事务
5. 调用 `applyEquipPlan()` 执行正式槽位更新
6. 调用 `buildEquipmentChangeResult()` 组装返回结果
7. 提交事务
8. 返回结果

### 事务要求
整个方法必须以单事务包裹：

- 目标槽位更新
- 被替换槽位处理
- 主副武器联动清理

### 异常处理要求
- 任意关键校验失败直接抛异常
- 任意写入失败必须回滚
- 必须记录必要日志，至少包含：`user_id`、`character_id`、`equipment_instance_id`、`target_slot_key`

### 不负责内容
- 不创建装备实例
- 不直接重算角色属性后写库
- 不处理掉落或奖励

---

## 6.2 `unequip()`

### 所属层级
Workflow

### 方法目标
作为正式卸下入口，组织槽位校验、槽位清空与结果返回。

### 建议签名
```php
public function unequip(int $userId, int $characterId, string $targetSlotKey): array
```

### 输出说明
返回正式卸装结果，建议至少包含：

- `character_id`
- `changed_slots`
- `unequipped_instance_ids`
- `slot_snapshot`

### 执行步骤
1. 调用 `validateUnequipRequest()` 做前置校验
2. 开启事务
3. 调用 `clearSlotInstance()` 清空目标槽位
4. 调用 `buildEquipmentChangeResult()` 组装返回结果
5. 提交事务
6. 返回结果

### 事务要求
卸装虽然相对简单，但槽位清空必须在正式事务中完成。

### 不负责内容
- 不删除装备实例
- 不写背包
- 不处理掉落或发奖

## 7. Domain 方法设计

## 7.1 `validateEquipRequest()`

### 所属层级
Domain

### 方法目标
校验穿戴请求是否满足第一阶段正式换装条件。

### 建议签名
```php
public function validateEquipRequest(int $userId, int $characterId, int $equipmentInstanceId, string $targetSlotKey): void
```

### 必须校验的内容
1. 角色存在
2. 目标槽位合法
3. 装备实例存在
4. 装备实例归属当前用户
5. 装备实例未被其他角色或其他槽位占用
6. 装备模板存在且启用
7. 模板装备位与目标槽位兼容
8. 角色等级满足装备模板要求
9. 如为副武器，主武器必须允许该副武器类别

### 依赖方法
- `getCharacterById()`
- `getEquipmentInstanceById()`
- `getEquipmentTemplateByItemId()`
- `isEquipmentInstanceEquipped()`
- `assertInstanceOwner()`
- `assertSlotCompatible()`
- `assertLevelRequirement()`
- `assertSubWeaponCompatible()`

### 不负责内容
- 不写槽位
- 不生成联动方案
- 不拼返回结果

---

## 7.2 `validateUnequipRequest()`

### 所属层级
Domain

### 方法目标
校验卸装请求是否合法。

### 建议签名
```php
public function validateUnequipRequest(int $userId, int $characterId, string $targetSlotKey): void
```

### 必须校验的内容
1. 角色存在且归属当前用户
2. 目标槽位合法
3. 槽位记录存在
4. 当前槽位确实有已穿戴实例

### 不负责内容
- 不直接清空槽位

---

## 7.3 `assertInstanceOwner()`

### 所属层级
Domain

### 方法目标
校验装备实例是否归属于当前用户。

### 建议签名
```php
public function assertInstanceOwner(int $userId, array|object $instance): void
```

### 关键要求
- 若实例不归属当前用户，必须抛异常

---

## 7.4 `assertSlotCompatible()`

### 所属层级
Domain

### 方法目标
校验装备模板与目标槽位是否兼容。

### 建议签名
```php
public function assertSlotCompatible(array|object $equipmentTemplate, string $targetSlotKey): void
```

### 关键规则
- `main_weapon` 模板只能进入 `main_weapon`
- `sub_weapon` 模板只能进入 `sub_weapon`
- `ring` 模板只能进入 `ring_1` / `ring_2`
- `bracelet` 模板只能进入 `bracelet_1` / `bracelet_2`
- 其余装备位按固定一对一规则进入对应槽位

---

## 7.5 `assertLevelRequirement()`

### 所属层级
Domain

### 方法目标
校验角色等级是否满足装备等级需求。

### 建议签名
```php
public function assertLevelRequirement(array|object $character, array|object $equipmentTemplate): void
```

---

## 7.6 `assertSubWeaponCompatible()`

### 所属层级
Domain

### 方法目标
在穿戴副武器时，校验当前主武器是否允许该副武器。

### 建议签名
```php
public function assertSubWeaponCompatible(array|object $mainWeaponTemplate, array|object $subWeaponTemplate): void
```

### 当前阶段重点规则
- 若主武器为双手武器，则不允许副武器
- 若副武器类别不在主武器允许列表中，则不允许穿戴

---

## 7.7 `buildEquipPlan()`

### 所属层级
Domain

### 方法目标
根据当前槽位状态、目标实例与主副武器规则，生成正式换装执行计划。

### 建议签名
```php
public function buildEquipPlan(array $context): array
```

### 计划中建议包含的内容
- 目标槽位要写入哪个实例
- 哪些槽位需要先清空
- 哪些实例会被替换下线
- 是否需要联动清空副武器槽

### 典型场景
#### 场景 1：普通替换
- 新实例写入目标槽
- 原实例移除

#### 场景 2：双手武器替换主武器
- 主武器槽写入新实例
- 副武器槽被联动清空

#### 场景 3：穿戴副武器
- 主武器必须是单手且允许当前副武器
- 副武器槽写入新实例

### 不负责内容
- 不直接落库

---

## 7.8 `applyEquipPlan()`

### 所属层级
Domain

### 方法目标
将正式换装计划落到槽位表中。

### 建议签名
```php
public function applyEquipPlan(array $plan): array
```

### 关键要求
- 必须只负责正式写入
- 必须在事务内执行
- 必须支持多槽位联动更新

### 输出建议
返回本次实际变更的槽位结果，供后续组装返回值使用。

---

## 7.9 `clearSlotInstance()`

### 所属层级
Domain

### 方法目标
清空某个槽位当前的 `equipped_instance_id`。

### 建议签名
```php
public function clearSlotInstance(array|object $slotRow): void
```

### 关键要求
- 只负责正式清空槽位
- 不删除装备实例记录

---

## 7.10 `buildEquipmentChangeResult()`

### 所属层级
Domain

### 方法目标
将正式变更结果组装为接口可直接消费的返回结构。

### 建议签名
```php
public function buildEquipmentChangeResult(int $characterId, array $changedSlots, array $meta = []): array
```

### 建议输出内容
至少包含：

- 角色 ID
- 发生变化的槽位清单
- 新穿上的实例 ID
- 被卸下的实例 ID 列表
- 当前穿戴槽快照摘要

### 关键要求
- 返回值应基于正式写入成功后的状态
- 不应凭空拼不存在的状态字段

## 9. 方法调用关系建议

建议调用顺序如下：

```text
equip()
 ├─ validateEquipRequest()
 │   ├─ getCharacterById()
 │   ├─ getEquipmentInstanceById()
 │   ├─ getEquipmentTemplateByItemId()
 │   ├─ isEquipmentInstanceEquipped()
 │   ├─ assertInstanceOwner()
 │   ├─ assertSlotCompatible()
 │   ├─ assertLevelRequirement()
 │   └─ assertSubWeaponCompatible()
 ├─ getCharacterSlotMap()
 ├─ buildEquipPlan()
 ├─ applyEquipPlan()
 └─ buildEquipmentChangeResult()

unequip()
 ├─ validateUnequipRequest()
 │   ├─ getCharacterById()
 │   └─ getSlotRowByKey()
 ├─ clearSlotInstance()
 └─ buildEquipmentChangeResult()
```

这个结构的目的，是把：

- 前置校验
- 联动规划
- 正式写入
- 结果组装

彻底拆开，避免一个方法既查、又算、又写、又拼返回。

## 10. 事务边界设计

换装链路必须以 `equip()` 和 `unequip()` 作为事务边界。

### `equip()` 事务内必须完成
- 目标槽位更新
- 被替换实例卸下
- 主副武器联动清空

### `unequip()` 事务内必须完成
- 槽位清空

### 严禁出现的状态
- 主武器已写入但副武器未联动清理
- 副武器已写入但主武器不兼容
- 同一实例被多个槽位同时引用

## 11. 异常与错误码建议

当前阶段建议至少区分以下错误类型：

### 11.1 角色不存在
适用场景：
- `character_id` 无效

### 11.2 槽位不合法
适用场景：
- `target_slot_key` 非法

### 11.3 装备实例不存在
适用场景：
- `equipment_instance_id` 无效

### 11.4 实例不归属当前用户
适用场景：
- 穿戴他人装备

### 11.5 装备模板不存在或未启用
适用场景：
- 模板缺失
- 模板已禁用

### 11.6 槽位与装备位不兼容
适用场景：
- 把戒指穿到护甲槽
- 把副武器穿到主武器槽

### 11.7 等级不足
适用场景：
- 角色等级未达到 `level_required`

### 11.8 主副武器不兼容
适用场景：
- 双手武器试图搭配副武器
- 主武器不支持当前副武器类别

### 11.9 装备已被占用
适用场景：
- 某实例已被其他槽位穿戴

### 11.10 换装失败
适用场景：
- 槽位写入失败
- 联动清理失败

---

## 12. 测试与自测要点

换装链路至少应覆盖以下测试点：

1. 合法装备可正常穿戴到正确槽位
2. 不兼容槽位不能穿戴
3. 等级不足不能穿戴
4. 非本人装备不能穿戴
5. 已被占用的实例不能重复穿戴
6. 双手武器穿戴后副武器槽会被联动清空
7. 不兼容副武器不能穿戴
8. 卸装后槽位会被正式清空
9. 换装失败时数据库中不应留下半成槽位状态
10. 换装后角色穿戴列表查询结果应与正式状态一致

## 13. 第一阶段实现建议顺序

建议按以下顺序实现换装链路：

1. 实现 `getCharacterById()`
2. 实现 `getCharacterSlotMap()` / `getSlotRowByKey()`
3. 实现 `getEquipmentInstanceById()`
4. 实现 `getEquipmentTemplateByItemId()`
5. 实现 `isEquipmentInstanceEquipped()`
6. 实现 `assertInstanceOwner()`
7. 实现 `assertSlotCompatible()`
8. 实现 `assertLevelRequirement()`
9. 实现 `assertSubWeaponCompatible()`
10. 实现 `validateEquipRequest()` / `validateUnequipRequest()`
11. 实现 `buildEquipPlan()`
12. 实现 `applyEquipPlan()`
13. 实现 `clearSlotInstance()`
14. 实现 `buildEquipmentChangeResult()`

## 14. 当前阶段结论

1. 换装链路必须只有正式 Workflow 入口，不能分散实现
2. 换装方法应按 Query、Domain、Workflow 三层拆分
3. 主副武器联动逻辑必须集中在正式换装服务中统一处理
4. 换装事务必须覆盖目标槽位更新与联动清理
5. 换装失败时不得留下半成正式状态
