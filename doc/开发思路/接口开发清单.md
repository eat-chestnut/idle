# 《山海巡厄录》角色创建方法设计

## 1. 文档目的

本文档用于在《核心开发总纲》《模块拆分与职责清单》《后端服务分层设计》《核心业务链路设计》《第一阶段六条主链清单》《高风险点与约束清单》《接口开发清单》基础上，进一步下沉到方法级，明确《山海巡厄录》第一阶段“角色创建链路”的方法拆分、方法职责、输入输出、事务边界、异常处理与实现顺序。

本文档不再停留在系统层或链路层，而是回答以下问题：

- 角色创建链路到底应该拆成哪些方法
- 每个方法负责什么，不负责什么
- 哪些方法属于 Query，哪些属于 Domain，哪些属于 Workflow
- 方法之间如何调用
- 角色创建时哪些步骤必须放在同一事务中
- 失败时应该如何处理与回滚

本文档定位为：

- 角色创建链路的方法级设计文档
- 角色创建代码实现前置文档
- 角色创建测试与 code review 的结构基线
- 后续其他方法级设计文档的参考模板

## 2. 角色创建方法设计总原则

### 2.1 先拆流程，再拆方法
角色创建方法设计必须先明确完整流程，再拆解到单点方法，不允许一开始直接堆砌多个零散方法。

### 2.2 创建动作必须走统一入口
角色创建必须只有一个正式入口，不允许：

- 前台创建一套逻辑
- 后台创建一套逻辑
- 测试脚本再写一套逻辑

统一入口应由正式 Workflow 调用正式 Domain 方法完成。

### 2.3 创建链路必须一次完成正式初始化
角色创建不是只插入一条 `characters` 记录，而是一个完整初始化动作，至少包括：

- 角色主表创建
- 默认成长字段初始化
- 12 个固定穿戴槽初始化

### 2.4 正式状态优先于返回结果
实现时必须优先保证数据库正式状态正确，再考虑返回结构。

### 2.5 异常必须回滚，不允许留半成状态
角色创建过程中如果任一步失败，不允许出现：

- 角色存在但槽位缺失
- 槽位只初始化了一部分
- 返回成功但数据库状态不完整

---

## 3. 角色创建链路目标与边界

## 3.1 链路目标
为当前用户创建一个正式可用角色，并初始化该角色的基础状态与固定穿戴槽。

## 3.2 链路起点
客户端提交“创建角色”请求，至少包含：

- `class_id`
- `character_name`

## 3.3 链路终点
满足以下条件时，链路才算完成：

1. `characters` 已存在正式角色记录
2. 角色默认成长字段已初始化
3. `character_equipment_slots` 已初始化 12 条固定槽位记录
4. 返回结果可供前端立即读取角色状态

## 3.4 链路不负责内容
角色创建链路不负责：

- 发放初始奖励
- 发放首通奖励
- 掉落结算
- 战斗准备
- 背包初始化复杂逻辑
- 长期成长系统初始化

如果后续要加“初始物品发放”，也应作为扩展步骤，不能污染第一阶段基础创建链。

---

## 4. 方法分层建议

当前角色创建链路建议拆为以下三层方法。

### 4.1 Query 层
负责读取前置条件。

建议方法：

- `getEnabledClassById()`
- `countUserCharacters()`
- `existsUserCharacterName()`（如第一阶段要求重名校验）

### 4.2 Domain 层
负责单点创建动作。

建议方法：

- `validateCharacterCreateRequest()`
- `buildCharacterCreatePayload()`
- `createCharacterRecord()`
- `buildDefaultEquipmentSlots()`
- `insertCharacterEquipmentSlots()`
- `buildCharacterCreateResult()`

### 4.3 Workflow 层
负责组织完整角色创建流程。

建议方法：

- `createCharacter()`

---

## 5. 方法清单总览

当前建议角色创建链路至少包含以下方法：

1. `createCharacter()`
2. `validateCharacterCreateRequest()`
3. `getEnabledClassById()`
4. `countUserCharacters()`
5. `existsUserCharacterName()`（可选）
6. `buildCharacterCreatePayload()`
7. `createCharacterRecord()`
8. `buildDefaultEquipmentSlots()`
9. `insertCharacterEquipmentSlots()`
10. `buildCharacterCreateResult()`

如果第一阶段保持最小实现，`existsUserCharacterName()` 可以先按业务需要决定是否启用。

## 6. Workflow 方法设计

## 6.1 `createCharacter()`

### 所属层级
Workflow

### 方法目标
作为角色创建链路唯一正式入口，组织前置校验、角色创建、槽位初始化与结果组装。

### 建议签名
```php
public function createCharacter(int $userId, array $input): array
```

### 输入说明
- `userId`：当前用户 ID
- `input`：至少包含 `class_id`、`character_name`

### 输出说明
返回正式角色创建结果数组，建议至少包含：

- `character_id`
- `user_id`
- `class_id`
- `character_name`
- `level`
- `exp`
- `unspent_stat_points`
- `equipment_slots`

### 执行步骤
1. 调用 `validateCharacterCreateRequest()` 做前置校验
2. 开启事务
3. 调用 `buildCharacterCreatePayload()` 生成角色主表写入数据
4. 调用 `createCharacterRecord()` 创建角色主表
5. 调用 `buildDefaultEquipmentSlots()` 生成 12 个固定槽位数据
6. 调用 `insertCharacterEquipmentSlots()` 批量写入槽位表
7. 调用 `buildCharacterCreateResult()` 组装正式返回结果
8. 提交事务
9. 返回结果

### 事务要求
整个方法必须使用单事务包裹：

- 角色主表写入
- 槽位初始化写入

### 异常处理要求
- 任意一步失败都必须回滚
- 不允许吞异常
- 应记录必要日志，至少包含 `user_id`、`class_id`、`character_name`

### 不负责内容
- 不直接查战斗属性
- 不直接创建背包内容
- 不直接发放任何奖励

---

## 7. Domain 方法设计

## 7.1 `validateCharacterCreateRequest()`

### 所属层级
Domain

### 方法目标
校验角色创建请求是否满足第一阶段创建条件。

### 建议签名
```php
public function validateCharacterCreateRequest(int $userId, array $input): void
```

### 必须校验的内容
1. `class_id` 是否存在且启用
2. `character_name` 是否为空
3. `character_name` 长度是否合法
4. 当前用户是否允许继续创建角色
5. 如业务要求，校验角色名是否重复

### 依赖方法
- `getEnabledClassById()`
- `countUserCharacters()`
- `existsUserCharacterName()`（可选）

### 抛错建议
建议明确抛出业务异常，而不是返回布尔值后让上层猜测原因。

### 不负责内容
- 不写数据库
- 不初始化槽位
- 不拼返回结果

---

## 7.2 `buildCharacterCreatePayload()`

### 所属层级
Domain

### 方法目标
将输入转换为正式可落库的角色主表数据。

### 建议签名
```php
public function buildCharacterCreatePayload(int $userId, array $input): array
```

### 输出字段建议
至少包含：

- `user_id`
- `class_id`
- `character_name`
- `level`
- `exp`
- `unspent_stat_points`
- `added_strength`
- `added_mana`
- `added_constitution`
- `added_dexterity`
- `long_term_growth_stage`
- `extra_context`
- `is_active`

### 当前阶段默认值建议
- `level = 1`
- `exp = 0`
- `unspent_stat_points = 0`
- 四维加点全部为 `0`
- `long_term_growth_stage = null`
- `extra_context = null`
- `is_active = 1`（若当前阶段默认新角色即为当前启用角色）

### 不负责内容
- 不执行插入
- 不校验职业合法性

---

## 7.3 `createCharacterRecord()`

### 所属层级
Domain

### 方法目标
将角色主表数据正式写入 `characters`。

### 建议签名
```php
public function createCharacterRecord(array $payload): Character
```

### 输入说明
`$payload` 为 `buildCharacterCreatePayload()` 的输出结果。

### 输出说明
返回创建后的 `Character` 模型或正式数组结果。

### 关键要求
- 必须只负责角色主表创建
- 不在该方法里初始化槽位
- 不在该方法里做复杂查询

### 不负责内容
- 不拼最终返回结构
- 不处理 12 槽位

---

## 7.4 `buildDefaultEquipmentSlots()`

### 所属层级
Domain

### 方法目标
根据角色 ID 生成固定 12 个穿戴槽初始数据。

### 建议签名
```php
public function buildDefaultEquipmentSlots(int $characterId): array
```

### 应生成的槽位
固定 12 个：

1. `main_weapon`
2. `sub_weapon`
3. `armor`
4. `leggings`
5. `gloves`
6. `boots`
7. `cloak`
8. `necklace`
9. `ring_1`
10. `ring_2`
11. `bracelet_1`
12. `bracelet_2`

### 每条记录建议字段
- `character_id`
- `slot_key`
- `equipped_instance_id = null`
- `sort_order`
- `created_at`
- `updated_at`

### 不负责内容
- 不执行插入
- 不查询已有槽位

---

## 7.5 `insertCharacterEquipmentSlots()`

### 所属层级
Domain

### 方法目标
将固定槽位数据正式写入 `character_equipment_slots`。

### 建议签名
```php
public function insertCharacterEquipmentSlots(array $rows): void
```

### 关键要求
- 必须支持批量插入
- 必须依赖数据库唯一约束 `UNIQUE(character_id, slot_key)`
- 不允许一条一条散写而缺少统一失败处理

### 不负责内容
- 不生成槽位数据
- 不查询角色信息

---

## 7.6 `buildCharacterCreateResult()`

### 所属层级
Domain

### 方法目标
将正式创建结果转换为接口或上层可直接消费的结构。

### 建议签名
```php
public function buildCharacterCreateResult(Character $character, array $slotRows): array
```

### 建议输出内容
至少包含：

- 角色基础信息
- 当前等级与经验
- 当前未分配属性点
- 默认四维加点
- 12 个固定槽位摘要

### 关键要求
- 返回结果应基于正式创建成功的数据
- 不应凭空拼接不存在于正式状态中的字段

---

## 8. Query 方法设计

## 8.1 `getEnabledClassById()`

### 所属层级
Query

### 方法目标
查询已启用的职业配置。

### 建议签名
```php
public function getEnabledClassById(string $classId): ?array
```

### 查询要求
- 只返回启用状态的职业
- 返回职业基础信息即可

### 典型用途
- 角色创建前校验职业是否合法

---

## 8.2 `countUserCharacters()`

### 所属层级
Query

### 方法目标
查询当前用户已有角色数量。

### 建议签名
```php
public function countUserCharacters(int $userId): int
```

### 典型用途
- 校验当前用户是否允许继续创建角色

---

## 8.3 `existsUserCharacterName()`

### 所属层级
Query

### 方法目标
按业务需要校验角色名是否已被当前用户使用，或是否全局重复。

### 建议签名
```php
public function existsUserCharacterName(int $userId, string $characterName): bool
```

### 当前阶段说明
若第一阶段暂不要求角色名唯一，可先不启用该校验。

---

## 9. 方法调用关系建议

建议调用顺序如下：

```text
createCharacter()
 ├─ validateCharacterCreateRequest()
 │   ├─ getEnabledClassById()
 │   ├─ countUserCharacters()
 │   └─ existsUserCharacterName()（可选）
 ├─ buildCharacterCreatePayload()
 ├─ createCharacterRecord()
 ├─ buildDefaultEquipmentSlots()
 ├─ insertCharacterEquipmentSlots()
 └─ buildCharacterCreateResult()
```

这个结构的目的，是把：

- 前置校验
- 主表创建
- 槽位初始化
- 结果组装

彻底拆开，避免一个方法既校验、又建表、又拼返回。

---

## 10. 事务边界设计

角色创建链路必须以 `createCharacter()` 为事务边界。

### 事务内必须完成
- 角色主表创建
- 12 个槽位初始化

### 事务外可以做
- 记录额外日志
- 上报埋点（如后续需要）

### 严禁出现的状态
- 角色存在但槽位不存在
- 只初始化部分槽位
- 接口返回成功但数据库未完整落库

---

## 11. 异常与错误码建议

当前阶段建议至少区分以下错误类型：

### 11.1 职业不存在或未启用
适用场景：
- `class_id` 无效
- 职业已禁用

### 11.2 角色名不合法
适用场景：
- 为空
- 超长
- 含非法字符（如第一阶段有该规则）

### 11.3 角色数量超过限制
适用场景：
- 当前用户已达到可创建角色上限

### 11.4 角色名重复（如启用）
适用场景：
- 当前用户角色名重复
- 或全局角色名重复

### 11.5 创建失败
适用场景：
- 数据库写入失败
- 槽位批量插入失败

错误码命名可以后续统一整理，但当前阶段至少应保证错误原因可分辨，不要统一返回“创建失败”。

---

## 12. 测试与自测要点

角色创建链路至少应覆盖以下测试点：

1. 合法职业可正常创建角色
2. 非法职业不能创建角色
3. 禁用职业不能创建角色
4. 创建成功后一定有 12 个槽位
5. 槽位 `slot_key` 是否完整且顺序正确
6. 创建失败时数据库中不应留下半成角色
7. 重复请求不会导致同一角色重复初始化槽位
8. 默认成长字段是否按预期初始化

---

## 13. 第一阶段实现建议顺序

建议按以下顺序实现角色创建链路：

1. 实现 `getEnabledClassById()`
2. 实现 `countUserCharacters()`
3. 实现 `validateCharacterCreateRequest()`
4. 实现 `buildCharacterCreatePayload()`
5. 实现 `createCharacterRecord()`
6. 实现 `buildDefaultEquipmentSlots()`
7. 实现 `insertCharacterEquipmentSlots()`
8. 实现 `buildCharacterCreateResult()`
9. 最后实现 `createCharacter()` Workflow

这样做的好处是：
- 先把底层方法写稳
- 再由 Workflow 串起来
- 更方便单测与逐步联调

---

## 14. 当前阶段结论

1. 角色创建链路必须只有一个正式 Workflow 入口
2. 角色创建不是单表插入，而是“角色主表 + 12 个固定穿戴槽”的完整初始化动作
3. 角色创建方法应按 Query、Domain、Workflow 三层拆分
4. 主表创建与槽位初始化必须放在同一事务中
5. 角色创建链路失败时不得留下半成正式状态
6. 后续代码实现、测试、自测与 code review，应以本文档定义的方法结构为准

# 《山海巡厄录》接口开发清单

## 1. 文档目的

本文档用于在《核心开发总纲》《模块拆分与职责清单》《后端服务分层设计》《核心业务链路设计》《第一阶段六条主链清单》《高风险点与约束清单》基础上，进一步明确《山海巡厄录》第一阶段接口应如何拆分、优先级如何安排、每类接口服务哪条主链、接口层应承担哪些职责、不应承担哪些职责，以及联调时应如何按接口分批推进。

本文档不直接给出字段级接口协议细节，而是用于回答以下问题：

- 第一阶段到底应该先做哪些接口
- 这些接口应按什么顺序推进
- 每类接口属于读接口还是写接口
- 每类接口对应哪条主链与哪条业务链路
- Controller / Handler 这一层到底应该做什么，不该做什么

本文档定位为：

- 第一阶段接口开发优先级清单
- 接口分组清单
- 接口职责边界清单
- 联调顺序与接口排期的上层文档

---

## 2. 接口开发总原则

### 2.1 接口必须服务主链开发
第一阶段接口开发不应按页面零散堆接口，而应优先服务六条主链：

1. 配置可读主链
2. 状态可写主链
3. 掉落可结算主链
4. 奖励可防重主链
5. 装备可实例化主链
6. 角色可重算主链

### 2.2 先做正式业务接口，再做辅助接口
接口开发顺序必须优先保证：

- 角色正式读取接口
- 背包正式读取接口
- 穿戴正式读取与写入接口
- 主线与关卡正式读取接口
- 战斗准备接口
- 战斗结算接口
- 奖励状态查询接口

不应先做：

- 调试便捷接口
- 临时拼数据接口
- 为页面临时凑结构的过渡接口

### 2.3 读接口与写接口必须分清
接口设计时必须明确区分：

- 读接口：用于读取正式状态或正式配置
- 写接口：用于触发正式业务动作

例如：

- 角色信息接口属于读接口
- 装备穿戴接口属于写接口
- 战斗准备接口更接近读 + 业务准备接口
- 战斗结算接口属于正式写接口

### 2.4 接口层不承担核心业务逻辑
Controller / Handler 层只负责：

- 收参
- 调服务
- 返回结果

不得在接口层直接实现：

- 发奖防重
- 掉落抽取
- 入包逻辑
- 属性重算算法
- 主副武器联动逻辑

### 2.5 接口必须围绕正式状态与正式链路返回
接口返回必须尽量围绕：

- 正式配置
- 正式角色状态
- 正式背包状态
- 正式穿戴状态
- 正式掉落与发奖结果

不应长期依赖：

- 临时 mock 数据
- 页面硬编码结构
- 未落库的临时拼接状态

---

## 3. 第一阶段接口总览

第一阶段建议拆为以下五类接口群。

### 3.1 角色与状态接口
负责读取角色当前正式状态与触发基础状态变更。

包括：

1. 角色信息接口
2. 穿戴列表接口
3. 装备穿戴接口
4. 装备卸下接口

### 3.2 背包与装备实例接口
负责读取玩家当前持有状态。

包括：

5. 背包列表接口
6. 装备实例详情接口
7. 装备模板详情接口

## 11. Controller / Handler 层职责边界

## 11.1 应负责内容
接口层应只负责：

- 参数接收
- 参数基础校验
- 调用对应服务
- 返回统一响应结构

## 11.2 不应负责内容
接口层不应负责：

- 掉落抽取
- 奖励发放防重
- 入包
- 属性重算算法
- 主副武器联动逻辑
- 多步事务编排

## 11.3 返回结构原则
接口层返回应尽量：

- 基于正式状态
- 结构稳定
- 可供前端长期复用

不应长期依赖：

- 临时拼字段
- 页面专用临时结构
- 未落正式链路的 mock 结果

---

## 12. 联调推进建议顺序

建议按以下顺序推进前后端联调：

### 第一步
- 角色信息接口
- 背包列表接口
- 穿戴列表接口

### 第二步
- 装备穿戴接口
- 装备卸下接口

### 第三步
- 主线章节列表接口
- 关卡难度列表接口
- 首通奖励状态接口

### 第四步
- 战斗准备接口

### 第五步
- 战斗结算接口

### 第六步
- 装备实例详情接口
- 发奖记录查询接口
- 其他调试接口

这样推进的目的，是先把状态读取与状态变更链路跑通，再进入正式战斗闭环联调。

---

## 13. 当前阶段不应过早开发的接口

以下接口当前第一阶段不应优先开发：

- 强化接口
- 洗练接口
- 宝石接口
- 套装接口
- 经卷接口
- world level 接口
- 商店购买接口
- 活动运营接口
- 复杂统计分析接口

原因：

- 这些能力不在第一阶段正式交付范围内
- 过早开发会稀释主链资源
- 很容易因为结构未稳定而重复开发

---

## 14. 当前阶段结论

1. 第一阶段接口开发必须优先服务六条主链，而不是按页面零散堆接口
2. 接口建议拆为角色与状态接口、背包与装备实例接口、主线与关卡接口、战斗接口、调试与记录接口五类接口群
3. 第一批接口必须优先覆盖角色读取、背包读取、穿戴读写、主线读取、战斗准备、战斗结算与首通奖励状态查询
4. 第二批接口主要服务联调与问题排查
5. Controller / Handler 层必须保持轻量，只负责收参、调服务、回响应
6. 掉落抽取、奖励发放、入包、属性重算等核心逻辑不得写在接口层
7. 接口联调应按“先状态、再主线、再战斗”的顺序推进

本文档可直接作为项目正式接口开发清单文档使用，并作为后续接口字段设计、联调排期与问题排查的统一接口基线。
