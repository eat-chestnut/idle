## 2. 奖励发放方法设计总原则

### 2.1 奖励发放必须只有统一正式入口
奖励发放链路必须只有一个正式 Workflow 入口，不允许：

- Controller 直接写 `user_reward_grants`
- 页面层自己判断首通后直接写背包
- 后台补发单独复制一套“后台专用发奖逻辑”

正式奖励结果必须由统一 Workflow 基于正式来源绑定、正式防重记录与正式入包链生成。

### 2.2 奖励发放必须基于正式来源对象
当前阶段奖励来源明确为：

- `first_clear`

对应来源对象为：

- 具体“关卡 + 难度”目标

因此奖励发放必须统一通过：

- `reward_group_bindings`
- `reward_groups`
- `reward_group_items`
- `user_reward_grants`
- `user_reward_grant_items`

来完成正式发奖。

### 2.3 奖励发放必须同时具备业务防重与请求幂等
奖励发放链路必须同时防两类问题：

1. 同一业务来源被重复发放
2. 同一请求因重试被重复执行

因此必须同时依赖：

- 业务唯一约束：`UNIQUE(user_id, source_type, source_id, reward_group_id)`
- 请求幂等约束：`UNIQUE(user_id, idempotency_key)`

### 2.4 奖励发放只负责“正式发什么”，不直接定义模板内容
奖励发放链路只负责：

- 判断是否能发
- 读取应该发什么
- 记录正式发奖过程
- 调用入包承接
- 返回正式发奖结果

它不负责：

- 定义奖励模板
- 定义物品本体
- 定义装备成长逻辑

### 2.5 奖励发放结果必须可追溯
正式奖励发放必须至少能回答：

- 为什么这次可以发
- 这次发的是哪套奖励组
- 发了哪些对象
- 入包是否成功
- 是否创建了装备实例
- 为什么失败

因此必须有：

- 发奖主记录
- 发奖明细
- 发奖快照
- 正式入包结果

## 3. 奖励发放链路目标与边界

## 3.1 链路目标
当满足副本不同难度首通条件时，按照正式奖励组配置安全发放固定奖励，并确保幂等、防重、可追溯、可补发。

## 3.2 链路起点
上游链路传入奖励发放上下文，至少包含：

- `user_id`
- `source_type`
- `source_id`
- `idempotency_context` 或等价请求上下文
- `is_cleared`

## 3.3 链路终点
满足以下条件时，奖励发放链路才算完成：

1. 当前来源对象已被正式校验
2. 当前来源已绑定奖励组
3. 奖励已通过业务防重与幂等校验
4. 发奖主记录与明细已落库
5. 奖励已通过正式入包链承接
6. 返回正式奖励结果

## 3.4 链路不负责内容
奖励发放链路不负责：

- 定义奖励模板
- 定义物品字段
- 定义装备模板属性
- 掉落解析
- 战斗结果计算

---

## 4. 方法分层建议

当前奖励发放链路建议拆为以下三层方法。

### 4.1 Query / Config 层
负责读取来源绑定、奖励组、奖励组明细、已发记录与物品基础信息。

建议方法：

- `getRewardBindingBySource()`
- `getRewardGroupById()`
- `getRewardGroupItemsByGroupId()`
- `getExistingRewardGrant()`
- `getRewardGrantByIdempotencyKey()`
- `getItemMapByIds()`

### 4.2 Domain 层
负责首通校验、幂等键生成、发奖记录构造、发奖明细构造、结果组装。

建议方法：

- `validateRewardGrantContext()`
- `assertRewardSourceValid()`
- `assertRewardGrantable()`
- `buildRewardIdempotencyKey()`
- `buildRewardGrantRecordPayload()`
- `createRewardGrantRecord()`
- `buildRewardGrantItemRows()`
- `insertRewardGrantItems()`
- `markRewardGrantSuccess()`
- `markRewardGrantFailed()`
- `buildRewardGrantResult()`

### 4.3 Workflow 层
负责组织完整奖励发放流程。

建议方法：

- `grant()`
- `retryGrant()`（后续后台补发可复用）

## 6. Workflow 方法设计

## 6.1 `grant()`

### 所属层级
Workflow

### 方法目标
作为奖励发放链路唯一正式入口，组织来源校验、防重校验、记录写入、明细写入、入包承接与正式结果返回。

### 建议签名
```php
public function grant(array $context): array
```

### 输入说明
`$context` 建议至少包含：

- `user_id`
- `source_type`
- `source_id`
- `is_cleared`
- `idempotency_context` 或等价上下文

### 输出说明
返回正式奖励结果，建议至少包含：

- `reward_grant_id`
- `source_type`
- `source_id`
- `reward_group_id`
- `grant_status`
- `reward_items`
- `inventory_results`
- `created_equipment_instances`
- `idempotency_key`

### 执行步骤
1. 调用 `validateRewardGrantContext()` 校验上下文
2. 调用 `assertRewardSourceValid()` 校验来源对象与绑定关系
3. 调用 `assertRewardGrantable()` 校验当前是否允许发放
4. 调用 `buildRewardIdempotencyKey()` 生成正式幂等键
5. 查询是否已存在同幂等键记录 `getRewardGrantByIdempotencyKey()`
6. 查询是否已存在同业务来源成功记录 `getExistingRewardGrant()`
7. 读取奖励组与奖励组明细
8. 构造发奖主记录 `buildRewardGrantRecordPayload()`
9. 写入发奖主记录 `createRewardGrantRecord()`，初始状态为 `pending`
10. 构造并写入发奖明细 `buildRewardGrantItemRows()` + `insertRewardGrantItems()`
11. 开启正式发奖事务
12. 调用 `InventoryWriteService` 承接正式入包
13. 在同一事务内更新发奖状态为 `success`
14. 若事务内失败，则在独立状态更新中把主记录沉淀为 `failed`
15. 调用 `buildRewardGrantResult()` 组装正式结果
16. 返回结果

### 事务要求
以下步骤必须放在同一正式成功事务中：

- 入包承接
- `success` 状态更新

以下步骤允许在成功事务之外独立执行，用于保留正式失败承接对象：

- `pending` 主记录落库
- 发奖明细落库
- 失败时将主记录更新为 `failed`

### 异常处理要求
- 任一关键校验失败必须直接中断
- 入包失败必须将发奖结果标记为 `failed`
- `failed` 记录必须保留足够的来源快照，支持后台补发与排查
- 不允许返回“表面成功但记录失败”的结果
- 必须记录必要日志，至少包含：`user_id`、`source_type`、`source_id`、`idempotency_key`

### 不负责内容
- 不定义奖励模板
- 不直接写掉落链逻辑
- 不在本方法中重写入包细节

---

## 7. Domain 方法设计

## 7.1 `validateRewardGrantContext()`

### 所属层级
Domain

### 方法目标
校验奖励发放上下文是否合法。

### 建议签名
```php
public function validateRewardGrantContext(array $context): void
```

### 必须校验的内容
1. `user_id` 存在
2. `source_type` 存在且当前阶段只能为 `first_clear`
3. `source_id` 存在
4. `is_cleared` 存在且合法
5. `idempotency_context` 或等价上下文存在

### 不负责内容
- 不读奖励组
- 不写记录

---

## 7.2 `assertRewardSourceValid()`

### 所属层级
Domain

### 方法目标
校验当前奖励来源对象有效且已绑定正式奖励组。

### 建议签名
```php
public function assertRewardSourceValid(array $context, array $binding, array $rewardGroup, array $rewardItems): void
```

### 必须校验的内容
1. 来源对象存在
2. 来源绑定存在且启用
3. 奖励组存在且启用
4. 奖励组明细不为空

### 当前阶段重点
- `source_type` 只能为 `first_clear`
- `source_id` 必须对应关卡难度目标

---

## 7.3 `assertRewardGrantable()`

### 所属层级
Domain

### 方法目标
校验当前请求是否允许正式发放奖励。

### 建议签名
```php
public function assertRewardGrantable(array $context, array $existingGrantStatus = []): void
```

### 必须校验的内容
1. 当前战斗结果满足首通发奖前置条件
2. 当前来源尚未成功发放过
3. 当前不是明显非法重复提交

### 不负责内容
- 不生成幂等键
- 不写发奖记录

---

## 7.4 `buildRewardIdempotencyKey()`

### 所属层级
Domain

### 方法目标
生成奖励发放正式幂等键。

### 建议签名
```php
public function buildRewardIdempotencyKey(array $context): string
```

### 当前阶段建议构成
建议至少基于：

- `user_id`
- `source_type`
- `source_id`
- 结算请求上下文

进行稳定生成。

### 关键要求
- 同一请求重试应产生同一幂等键

## 9. 方法调用关系建议

建议调用顺序如下：

```text
grant()
 ├─ validateRewardGrantContext()
 ├─ getRewardBindingBySource()
 ├─ getRewardGroupById()
 ├─ getRewardGroupItemsByGroupId()
 ├─ assertRewardSourceValid()
 ├─ assertRewardGrantable()
 ├─ buildRewardIdempotencyKey()
 ├─ getRewardGrantByIdempotencyKey()
 ├─ getExistingRewardGrant()
 ├─ buildRewardGrantRecordPayload()
 ├─ createRewardGrantRecord()
 ├─ buildRewardGrantItemRows()
 ├─ insertRewardGrantItems()
 ├─ InventoryWriteService::writeRewards(...)
 ├─ markRewardGrantSuccess() / markRewardGrantFailed()
 └─ buildRewardGrantResult()
```

这个结构的目的，是把：

- 上下文校验
- 正式绑定读取
- 防重 / 幂等校验
- 记录写入
- 入包承接
- 正式结果组装

彻底拆开，避免一个方法里直接混入所有逻辑。

## 10. 幂等、防重与事务边界设计

### 10.1 幂等边界
当前阶段幂等控制必须基于：

- `UNIQUE(user_id, idempotency_key)`

保证同一请求重试不会重复执行正式发奖。

### 10.2 业务防重边界
当前阶段业务防重必须基于：

- `UNIQUE(user_id, source_type, source_id, reward_group_id)`

保证同一业务来源不会重复成功发奖。

### 10.3 正式事务边界
以下步骤必须放在同一事务中：

- 发奖主记录写入
- 发奖明细写入
- 入包承接
- 状态更新为 `success` / `failed`

### 10.4 当前阶段禁止出现的状态
- 发奖结果返回成功，但主记录不存在
- 发奖记录为 success，但入包没有成功
- 发奖明细已写，但主记录失败
- 奖励已入包，但发奖状态仍是 pending

## 11. 异常与错误码建议

当前阶段建议至少区分以下错误类型：

### 11.1 奖励来源不合法
适用场景：
- `source_type` 非法
- `source_id` 缺失

### 11.2 奖励来源绑定缺失
适用场景：
- 当前来源没有正式奖励绑定

### 11.3 奖励组不存在或未启用
适用场景：
- 绑定存在但奖励组无效

### 11.4 奖励组明细为空
适用场景：
- 没有正式可发的奖励明细

### 11.5 奖励已发放
适用场景：
- 同一来源已成功发过奖励

### 11.6 幂等冲突
适用场景：
- 同幂等键重复请求

### 11.7 入包失败
适用场景：
- 正式入包链执行失败
- 装备实例创建失败

### 11.8 奖励发放失败
适用场景：
- 记录写入失败
- 状态更新失败
- 正式结果组装失败

错误码可以后续统一整理，但当前阶段至少应保证错误原因清晰，不要统一返回“奖励发放失败”。

---

## 12. 测试与自测要点

奖励发放链路至少应覆盖以下测试点：

1. 合法首通来源可正常找到唯一奖励组
2. 来源未绑定奖励组时会正确报错
3. 同一来源首次可成功发放奖励
4. 同一来源重复请求不会重复发奖
5. 同一幂等键重复请求不会重复执行
6. 发奖明细会按正式奖励组正确写入
7. 奖励中的装备对象会通过正式入包链创建实例
8. 发奖成功后状态、明细、入包结果一致
9. 发奖失败时状态会标记为 failed
10. 后台补发可复用正式奖励发放链

---

## 13. 第一阶段实现建议顺序

建议按以下顺序实现奖励发放链路：

1. 实现 `validateRewardGrantContext()`
2. 实现 `getRewardBindingBySource()`
3. 实现 `getRewardGroupById()`
4. 实现 `getRewardGroupItemsByGroupId()`
5. 实现 `assertRewardSourceValid()`
6. 实现 `assertRewardGrantable()`
7. 实现 `buildRewardIdempotencyKey()`
8. 实现 `getRewardGrantByIdempotencyKey()`
9. 实现 `getExistingRewardGrant()`
10. 实现 `buildRewardGrantRecordPayload()`
11. 实现 `createRewardGrantRecord()`
12. 实现 `buildRewardGrantItemRows()`
13. 实现 `insertRewardGrantItems()`
14. 对接 `InventoryWriteService`
15. 实现 `markRewardGrantSuccess()` / `markRewardGrantFailed()`
16. 实现 `buildRewardGrantResult()`
17. 最后实现 `grant()` Workflow
18. 最后补 `retryGrant()`

这样做的好处是：
- 先把配置读取与防重逻辑写稳
- 再写正式记录链
- 最后对接入包并统一收口到 Workflow

---

## 14. 当前阶段结论

1. 奖励发放链路必须只有一个正式 Workflow 入口，不得散落实现
2. 奖励发放方法应按 Query / Config、Domain、Workflow 三层拆分
3. 奖励发放必须同时具备业务防重与请求幂等双重保护
4. 发奖主记录、发奖明细、入包结果必须分层处理后再统一返回
5. 奖励发放失败时不得伪装成成功结果
6. 后续代码实现、测试、自测与 code review，应以本文档定义的方法结构为准

本文档可直接作为项目正式《奖励发放方法设计》文档使用，并作为后续 `入包方法设计.md` 的方法级设计上游文档。
