## 2. 掉落解析方法设计总原则

### 2.1 掉落解析必须只有统一正式入口
掉落解析链路必须只有一个正式入口，不允许：

- 在 Controller 中直接查掉落组然后随机
- 在战斗结算 Workflow 中硬编码另一套抽取逻辑
- 在后台工具或测试脚本中各写一份掉落算法

正式掉落结果必须由统一服务基于正式配置生成。

### 2.2 掉落解析必须基于正式来源绑定
当前阶段掉落来源明确为：

- 普通怪
- 精英怪
- Boss

因此掉落解析必须统一通过：

- `drop_group_bindings`
- `drop_groups`
- `drop_group_items`

来获取正式掉落配置。

### 2.3 掉落解析只负责“命中什么”，不负责“玩家怎么持有”
掉落解析链路的职责边界必须固定为：

- 输入：来源对象与上下文
- 输出：标准掉落结果

它不负责：

- 入包
- 发奖
- 装备实例创建
- 背包数量累加

### 2.4 掉落抽取语义必须唯一
当前阶段正式抽取规则只有：

- `weighted_single`
- `weighted_repeat`

这两种规则的语义必须只在一个正式抽取方法中实现，不允许在不同位置各写一套“近似逻辑”。

### 2.5 掉落结果必须标准化
掉落解析输出不应直接耦合前端页面，而应输出标准化结果，至少包含：

- `item_id`
- `quantity`
- `source_type`
- `source_id`
- `drop_group_id`

后续由结算链路决定如何展示、如何入包。

## 3. 掉落解析链路目标与边界

## 3.1 链路目标
根据正式怪物击败结果与正式掉落配置，生成标准化随机掉落结果。

## 3.2 链路起点
上游链路传入掉落解析上下文，至少包含：

- `stage_difficulty_id`
- `killed_monster_ids`
- `battle_context_id` 或等价上下文

## 3.3 链路终点
满足以下条件时，掉落解析链路才算完成：

1. 每个来源对象都已通过正式绑定找到掉落组
2. 掉落组抽取规则已被正式执行
3. 重复命中结果已被标准化合并
4. 返回标准掉落结果列表

## 3.4 链路不负责内容
掉落解析链路不负责：

- 背包写入
- 奖励发放
- 装备实例创建
- 角色属性计算
- 战斗结果总汇总

---

## 4. 方法分层建议

当前掉落解析链路建议拆为以下三层方法。

### 4.1 Query / Config 层
负责读取来源绑定、掉落组、掉落组明细与物品基础信息。

建议方法：

- `getDropBindingsBySources()`
- `getDropGroupMapByIds()`
- `getDropGroupItemsMapByGroupIds()`
- `getItemMapByIds()`

### 4.2 Domain 层
负责来源分组、抽取执行、结果合并与结果标准化。

建议方法：

- `validateDropResolveContext()`
- `groupSourcesForDropResolve()`
- `resolveDropForSource()`
- `drawWeightedSingle()`
- `drawWeightedRepeat()`
- `pickOneByWeight()`
- `mergeDropResults()`
- `buildStandardDropResults()`

### 4.3 Workflow / Facade 层
负责组织完整掉落解析流程。

建议方法：

- `resolve()`

## 6. Workflow / Facade 方法设计

## 6.1 `resolve()`

### 所属层级
Workflow / Facade

### 方法目标
作为掉落解析链路唯一正式入口，组织上下文校验、来源绑定读取、掉落组读取、抽取执行、结果合并与标准化输出。

### 建议签名
```php
public function resolve(array $context): array
```

### 输入说明
`$context` 建议至少包含：

- `stage_difficulty_id`
- `killed_monster_ids`
- `battle_context_id`
- 可选的 `source_type_map` 或等价来源说明

### 输出说明
返回标准掉落结果列表，建议每条至少包含：

- `item_id`
- `quantity`
- `source_type`
- `source_id`
- `drop_group_id`

### 执行步骤
1. 调用 `validateDropResolveContext()` 校验上下文
2. 调用 `groupSourcesForDropResolve()` 标准化来源列表
3. 调用 `getDropBindingsBySources()` 读取正式绑定关系
4. 收集涉及的 `drop_group_id`
5. 调用 `getDropGroupMapByIds()` 读取掉落组主表
6. 调用 `getDropGroupItemsMapByGroupIds()` 读取掉落组明细
7. 遍历来源，逐个调用 `resolveDropForSource()`
8. 调用 `mergeDropResults()` 合并重复命中结果
9. 调用 `buildStandardDropResults()` 输出标准结果
10. 返回正式掉落结果

### 事务要求
当前阶段掉落解析以“读配置 + 算结果”为主，原则上不需要数据库写事务。

### 异常处理要求
- 任一关键配置缺失时直接抛异常
- 抽取规则非法时直接中断
- 不允许返回“半成掉落结果”
- 应记录必要日志，至少包含：`stage_difficulty_id`、`battle_context_id`、`killed_monster_ids`

### 不负责内容
- 不写背包
- 不创建装备实例
- 不发放奖励

## 7. Domain 方法设计

## 7.1 `validateDropResolveContext()`

### 所属层级
Domain

### 方法目标
校验掉落解析上下文是否合法。

### 建议签名
```php
public function validateDropResolveContext(array $context): void
```

### 必须校验的内容
1. `stage_difficulty_id` 存在
2. `killed_monster_ids` 为数组且非空（若业务允许空，则应明确）
3. `battle_context_id` 或等价上下文字段存在
4. 怪物 ID 格式合法

### 不负责内容
- 不读取掉落组
- 不执行抽取

---

## 7.2 `groupSourcesForDropResolve()`

### 所属层级
Domain

### 方法目标
把上游传入的击败怪物结果标准化为可用于读取来源绑定的来源集合。

### 建议签名
```php
public function groupSourcesForDropResolve(array $context): array
```

### 输出内容建议
每个来源节点至少包含：

- `source_type`
- `source_id`
- `kill_count` 或等价数量信息

### 当前阶段建议
若掉落按“每只怪分别触发”解析，可保留怪物明细数量；若上游已逐只展开，也可输出标准来源列表。

---

## 7.3 `resolveDropForSource()`

### 所属层级
Domain

### 方法目标
针对单个来源对象，执行正式掉落组抽取逻辑。

### 建议签名
```php
public function resolveDropForSource(array $source, array $binding, array $dropGroup, array $groupItems): array
```

### 输入说明
- `$source`：来源对象
- `$binding`：来源绑定关系
- `$dropGroup`：掉落组主表
- `$groupItems`：该掉落组明细

### 输出说明
返回该来源命中的原始掉落结果列表，每条至少包含：

- `item_id`
- `quantity`
- `source_type`
- `source_id`
- `drop_group_id`

### 执行步骤
1. 校验掉落组是否启用
2. 校验明细是否存在且可用
3. 根据 `roll_type` 选择抽取方法
4. 返回该来源原始命中结果

### 不负责内容
- 不合并全局重复结果
- 不写背包

---

## 7.4 `drawWeightedSingle()`

### 所属层级
Domain

### 方法目标
执行一次 `weighted_single` 抽取。

### 建议签名
```php
public function drawWeightedSingle(array $groupItems, array $sourceMeta): array
```

### 当前阶段正式语义
- 本次掉落抽取 1 次
- 按权重从明细中命中 1 条结果
- 本轮只产生 1 个命中对象

### 输出说明
返回单条原始命中结果，建议包含：

- `item_id`
- `quantity`
- `source_type`
- `source_id`
- `drop_group_id`

---

## 7.5 `drawWeightedRepeat()`

### 所属层级
Domain

### 方法目标
执行 `weighted_repeat` 抽取。

### 建议签名
```php
public function drawWeightedRepeat(array $groupItems, int $rollTimes, array $sourceMeta): array
```

### 当前阶段正式语义
- 按 `roll_times` 独立抽取多次
- 每次都按全量明细表重新参与权重抽取
- 允许同一对象被重复命中
- 重复命中后由后续合并步骤统一合并数量

### 输出说明
返回该来源的原始命中结果列表。

---

## 7.6 `pickOneByWeight()`

### 所属层级
Domain

### 方法目标
在一组掉落明细中按权重命中 1 条记录。

### 建议签名
```php
public function pickOneByWeight(array $groupItems): array
```

### 关键要求
- 必须只维护一套正式权重抽取实现
- 权重总和、随机区间、边界命中规则都应固定
- 权重为 0 的条目不应进入有效命中池

### 异常处理要求
- 若有效明细为空，必须抛异常
- 若权重总和非法，必须抛异常

---

## 7.7 `mergeDropResults()`

### 所属层级
Domain

### 方法目标
将多来源、多次抽取得到的原始掉落结果合并为标准全局结果。

### 建议签名
```php
public function mergeDropResults(array $rawResults): array
```

### 合并建议
第一阶段建议按以下维度合并：

- `item_id`

若后续需要保留更细来源审计，也可同时保留来源明细，但对最终入包结果建议按 `item_id` 汇总数量。

### 输出说明
返回已合并的标准结果中间结构。

---

## 7.8 `buildStandardDropResults()`

### 所属层级
Domain

### 方法目标
将合并后的中间结果转换为正式标准掉落结果结构。

### 建议签名
```php
public function buildStandardDropResults(array $mergedResults): array
```

### 建议输出字段
每条至少包含：

- `item_id`
- `quantity`
- `source_type`（如单一来源可保留；若多来源合并可提供摘要）
- `source_id`
- `drop_group_id`

### 关键要求
- 输出结构稳定
- 可直接被战斗结算链与入包链消费
- 不夹带页面专用临时字段

---

## 8. Query / Config 方法设计

## 8.1 `getDropBindingsBySources()`

### 所属层级
Query / Config

### 方法目标
根据来源集合批量读取正式掉落来源绑定。

### 建议签名
```php
public function getDropBindingsBySources(array $sources): array
```

### 查询要求
- 只返回启用绑定
- 当前阶段只允许怪物来源
- 返回结果建议以 `source_type + source_id` 为 key

### 当前阶段关键规则
- 一来源一组
- `UNIQUE(source_type, source_id)`

---

## 8.2 `getDropGroupMapByIds()`

### 所属层级
Config / Query

### 方法目标
批量读取掉落组主表配置。

### 建议签名
```php
public function getDropGroupMapByIds(array $dropGroupIds): array
```

### 查询要求
- 只返回启用掉落组
- 返回以 `drop_group_id` 为 key 的映射结构

---

## 8.3 `getDropGroupItemsMapByGroupIds()`

### 所属层级
Config / Query

### 方法目标
批量读取掉落组明细，并按 `drop_group_id` 分组返回。

### 建议签名
```php
public function getDropGroupItemsMapByGroupIds(array $dropGroupIds): array
```

### 查询要求
- 只返回启用明细
- 每组内按 `sort_order` 排序
- 至少返回：`item_id`、`weight`、`min_quantity`、`max_quantity`

---

## 8.4 `getItemMapByIds()`

### 所属层级
Config / Query

### 方法目标
按需读取掉落结果涉及的物品基础信息。

### 建议签名
```php
public function getItemMapByIds(array $itemIds): array
```

### 当前阶段说明
如果标准掉落结果只要求 `item_id + quantity`，本方法可作为按需扩展；如果结算结果需要带名称 / 稀有度摘要，可在正式流程后段使用。

## 9. 方法调用关系建议

建议调用顺序如下：

```text
resolve()
 ├─ validateDropResolveContext()
 ├─ groupSourcesForDropResolve()
 ├─ getDropBindingsBySources()
 ├─ getDropGroupMapByIds()
 ├─ getDropGroupItemsMapByGroupIds()
 ├─ resolveDropForSource()
 │   ├─ drawWeightedSingle()
 │   │   └─ pickOneByWeight()
 │   └─ drawWeightedRepeat()
 │       └─ pickOneByWeight()
 ├─ mergeDropResults()
 └─ buildStandardDropResults()
```

这个结构的目的，是把：

- 上下文校验
- 正式绑定读取
- 掉落组读取
- 抽取执行
- 结果合并
- 标准输出

彻底拆开，避免一个方法里同时塞进查询、抽取、合并和返回组装。

## 10. 抽取语义与边界设计

### 10.1 `weighted_single` 边界
正式语义固定为：

- 只抽 1 次
- 只命中 1 条
- 只生成 1 个对象结果

### 10.2 `weighted_repeat` 边界
正式语义固定为：

- 抽多次
- 每次重新参与全量权重池
- 允许重复命中同一对象
- 重复结果后续统一合并

### 10.3 数量区间边界
当前阶段每次命中后应根据：

- `min_quantity`
- `max_quantity`

生成该次命中的正式数量。

### 10.4 掉落结果边界
掉落链输出的是：

- 模板对象命中结果

不是：

- 背包结果
- 装备实例结果
- 最终展示结果

---

## 11. 异常与错误码建议

当前阶段建议至少区分以下错误类型：

### 11.1 掉落解析上下文不合法
适用场景：
- `killed_monster_ids` 缺失
- `battle_context_id` 缺失

### 11.2 来源绑定缺失
适用场景：
- 某怪物没有正式掉落绑定

### 11.3 掉落组不存在或未启用
适用场景：
- 绑定存在但组无效

### 11.4 掉落组明细为空或非法
适用场景：
- 没有可用明细
- 权重总和非法

### 11.5 抽取规则非法
适用场景：
- `roll_type` 不在正式支持列表中

### 11.6 掉落解析失败
适用场景：
- 抽取过程异常
- 标准结果组装失败

错误码可以后续统一整理，但当前阶段至少应保证错误原因清晰，不要统一返回“掉落解析失败”。

---

## 12. 测试与自测要点

掉落解析链路至少应覆盖以下测试点：

1. 合法来源可正确找到唯一掉落组
2. 无绑定来源会正确报错
3. `weighted_single` 只会命中 1 条结果
4. `weighted_repeat` 会按 `roll_times` 重复抽取
5. 重复命中结果会被正确合并
6. 权重为 0 的条目不会被命中
7. 数量区间会按正式规则生成结果
8. 掉落结果只输出模板对象，不直接写入背包
9. 掉落结果结构稳定，可被结算链直接消费
10. 非法 `roll_type` 会正确中断解析

---

## 13. 第一阶段实现建议顺序

建议按以下顺序实现掉落解析链路：

1. 实现 `validateDropResolveContext()`
2. 实现 `groupSourcesForDropResolve()`
3. 实现 `getDropBindingsBySources()`
4. 实现 `getDropGroupMapByIds()`
5. 实现 `getDropGroupItemsMapByGroupIds()`
6. 实现 `pickOneByWeight()`
7. 实现 `drawWeightedSingle()`
8. 实现 `drawWeightedRepeat()`
9. 实现 `resolveDropForSource()`
10. 实现 `mergeDropResults()`
11. 实现 `buildStandardDropResults()`
12. 最后实现 `resolve()` 正式入口

这样做的好处是：
- 先把配置读取和上下文标准化写稳
- 再把抽取核心写稳
- 最后统一收口到正式掉落服务

---

## 14. 当前阶段结论

1. 掉落解析链路必须只有一个正式入口，不得散落实现
2. 掉落解析方法应按 Query / Config、Domain、Workflow / Facade 三层拆分
3. `weighted_single` 与 `weighted_repeat` 的正式语义必须只维护一套实现
4. 掉落链只输出模板对象结果，不负责背包写入与实例创建
5. 掉落解析失败时不得返回半成结果
6. 后续代码实现、测试、自测与 code review，应以本文档定义的方法结构为准

本文档可直接作为项目正式《掉落解析方法设计》文档使用，并作为后续 `奖励发放方法设计.md`、`入包方法设计.md` 的方法级设计并行参考文档。
