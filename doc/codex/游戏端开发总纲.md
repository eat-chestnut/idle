# 《山海巡厄录》游戏端开发总纲

## 1. 文档目的

本文档用于把当前已经稳定的 phase-one backend 能力，转换成可直接指导游戏端开发的正式总纲。

本文档重点不是系统规则本身，而是明确：
- 游戏端当前应该做什么
- 页面与接口接入顺序是什么
- 哪些能力属于 phase-one 必做
- 哪些能力先不做
- 如何避免客户端重复实现后端业务逻辑

---

## 2. 当前阶段目标

当前游戏端目标不是做完整大系统，而是完成一条“可登录、可创建角色、可看背包穿戴、可进入关卡、可准备战斗、可展示结算结果”的最小可玩链。

也就是：

1. 进入客户端
2. 获取/设置 token
3. 创建角色
4. 查看角色
5. 查看背包与穿戴
6. 查看章节与难度
7. 发起 battle prepare
8. 展示 battle settle 结果
9. 展示首通奖励状态变化

---

## 3. 开发边界

### Phase-one 必做
- 认证接入
- 角色创建页
- 角色详情页
- 背包页
- 穿戴页
- 主线章节页
- 难度选择页
- 战斗准备页
- 战斗结算结果页
- 首通奖励状态展示

### Phase-one 暂不做
- 强化
- 洗练
- 宝石
- 套装
- 经卷
- world level
- 商店
- 活动
- 社交
- 复杂成长树
- 大量动画演出系统

---

## 4. 游戏端开发原则

### 4.1 后端为业务真相
客户端不重复发明：
- 掉落规则
- 奖励规则
- battle_context 合法性
- 所有权规则
- 幂等规则

### 4.2 页面按主链逐步打通
先做单条可玩的竖切，不先铺很多半成页面。

### 4.3 所有页面必须有状态
至少包括：
- loading
- success
- empty
- error
- unauthorized

### 4.4 先接真接口，再做美术包装
当前阶段应优先确认：
- 字段可用
- 流程可通
- 页面状态可收口

---

## 5. 推荐开发顺序

1. token 与环境配置
2. 角色创建页
3. 角色详情页
4. 背包页
5. 穿戴页
6. 章节页
7. 难度页
8. battle prepare 页
9. battle settle 结果页
10. 首通奖励状态显示补齐

---

## 6. 与 backend 的关系

游戏端应直接以以下内容为准：
- `backend/docs/api/phase-one-frontend.openapi.json`
- 根目录 `doc/codex/接口示例文档.md`
- 根目录 `doc/codex/认证与接口公共规则.md`

若三者不一致，以当前真实 backend 行为为准，并推动文档回写。

---

## 7. 当前阶段结论

现在 backend 已经达到可联调阶段，游戏端开发应正式启动。
当前最合理策略是：
- backend 小步收口
- client 正式接 API
- 双线并行推进
