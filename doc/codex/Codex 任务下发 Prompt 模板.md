

# 《山海巡厄录》Codex 任务下发 Prompt 模板

## 1. 文档目的

本文档用于在《Codex 主开发协作清单》基础上，进一步沉淀《山海巡厄录》项目给 Codex 下发开发任务时可长期复用的 Prompt 模板。

本文档目标不是继续补系统设计，而是解决以下实际协作问题：

- 每次给 Codex 下任务时，应该怎么组织 Prompt
- 如何让 Codex 先读文档、先给计划、再改代码
- 如何避免 Prompt 写得过散、过大、过虚，导致 Codex 跑偏
- 如何让不同任务保持一致的表达结构、输出结构与验收结构

本文档定位为：

- Codex 单任务下发模板文档
- Codex 多类任务 Prompt 复用模板文档
- 后续开发、联调、修复、重构任务的统一 Prompt 基线

---

## 2. Prompt 设计总原则

### 2.1 任务必须结构化，不要口语化乱写
给 Codex 的 Prompt 不应只写：

- 帮我做角色系统
- 帮我改下战斗逻辑
- 按文档开发

这种写法太模糊，Codex 会自动补假设，容易跑偏。

正式 Prompt 必须尽量结构化，至少要明确：

- 本次做什么
- 本次不做什么
- 依据哪些文档
- 输出顺序是什么
- 验收标准是什么

### 2.2 单次任务必须边界明确
每次任务建议只解决一个明确边界的问题，例如：

- 只做角色创建链
- 只做换装链
- 只做战斗准备接口
- 只做后台某一组资源页

不要一次 Prompt 同时要求：

- 多条业务链
- 后台 + 接口 + 页面 + 测试全做
- 主链实现 + 大规模重构 + 未来系统扩展

### 2.3 Prompt 必须同时写“做什么”和“不做什么”
很多任务跑偏，不是因为目标不清，而是因为没有明确写出禁止扩展项。

所以正式 Prompt 里必须同时写：

- 本次只做
- 本次不做

### 2.4 Prompt 必须指定文档优先级
Codex 开发不能靠自由发挥，必须在 Prompt 中明确：

- 先读哪些文档
- 以哪份文档为最高优先级
- 如果多个文档有冲突，以哪个为准

### 2.5 Prompt 必须要求先出计划再改代码
正式 Prompt 必须要求 Codex 先输出：

1. 任务理解
2. 实施计划
3. 文件改动清单
4. 风险点

然后再开始改代码。

### 2.6 Prompt 必须带验收标准
不要只说“做完”。

必须明确：

- 哪些结果算完成
- 哪些行为算违规
- 哪些问题不能出现

## 3. 通用 Prompt 模板结构

建议所有正式任务 Prompt 都按以下结构组织。

### 3.1 模板结构清单
1. 任务名称
2. 本次任务目标
3. 本次只做什么
4. 本次不做什么
5. 必读文档
6. 文档优先级
7. 实现要求
8. 输出顺序
9. 验收标准
10. 禁止事项

## 4. 通用 Prompt 主模板

下面这份是最通用的主模板，可用于大部分正式开发任务。

```md
你现在是《山海巡厄录》项目的主开发代理，请严格依据项目文档执行，不要自行扩展未定稿功能。

【任务名称】
<填写本次任务名称>

【本次任务目标】
<填写这次要完成的明确目标>

【本次只做】
1. <事项1>
2. <事项2>
3. <事项3>

【本次不做】
1. <禁止扩展项1>
2. <禁止扩展项2>
3. <禁止扩展项3>

【必读文档】
1. 项目总览.md
2. Codex 主开发协作清单.md
3. <本次任务相关开发思路文档>
4. <本次任务相关方法级设计文档>
5. <本次任务相关接口字段级设计文档，如涉及接口>

【文档优先级】
1. 若有冲突，先以《项目总览》为总入口
2. 再以本次任务对应的方法级设计文档为直接实现依据
3. 再以接口字段级设计或后台开发清单为交付补充依据
4. 不允许自行创造超出文档的新规则

【实现要求】
1. 严格遵守五层服务分层：Config / Query / Domain / Workflow / Admin
2. Controller 保持轻量，只负责收参、调服务、回响应
3. 不允许把正式业务逻辑直接写在 Controller
4. 不允许绕过正式 Service 直接改正式状态表
5. 模板层与实例层必须分离
6. 掉落链与奖励链必须分离
7. 若涉及事务，必须明确事务边界
8. 若涉及高风险链路，必须符合高风险点与约束清单

【输出顺序】
请严格按以下顺序输出：
1. 任务理解
2. 实施计划
3. 将修改的文件清单
4. 关键风险点
5. 开始编码
6. 输出自测步骤
7. 输出未完成项 / 待确认项

【验收标准】
1. <标准1>
2. <标准2>
3. <标准3>

【禁止事项】
1. 不得主动扩展未来系统
2. 不得主动重构无关模块
3. 不得擅自修改未授权文件
4. 不得违反当前项目文档口径
```

## 5. 业务链开发 Prompt 模板

适用场景：

- 角色创建链
- 换装链
- 战斗准备链
- 战斗结算链
- 掉落解析链
- 奖励发放链
- 入包链

```md
你现在是《山海巡厄录》项目的主开发代理，本次任务是实现一条正式业务链。请严格依据项目文档执行，不要扩展未定稿系统。

【任务名称】
<例如：角色创建链路开发>

【本次任务目标】
实现 <填写业务链名称> 的正式代码落地，使其符合当前方法级设计文档。

【本次只做】
1. 实现 <对应 Workflow>
2. 实现 <对应 Domain 方法>
3. 实现 <对应 Query / Config 方法>
4. 接通本链路需要的最小接口或最小调用入口

【本次不做】
1. 不实现无关业务链
2. 不实现未来成长系统
3. 不补做无关后台页
4. 不做无关重构

【必读文档】
1. 项目总览.md
2. Codex 主开发协作清单.md
3. 开发思路/核心业务链路设计.md
4. 开发思路/第一阶段六条主链清单.md
5. 开发思路/高风险点与约束清单.md
6. <对应方法级设计文档>

【实现要求】
1. 严格按方法级设计拆分 Query / Domain / Workflow
2. 不允许把多步流程直接写进 Controller
3. 事务边界必须按文档要求实现
4. 若涉及正式状态写入，失败时不得留下半成状态
5. 若涉及高风险约束，必须同时满足数据库约束与服务层约束口径

【输出顺序】
1. 任务理解
2. 实施计划
3. 文件改动清单
4. 风险点
5. 开始编码
6. 自测步骤
7. 未完成项 / 待确认项

【验收标准】
1. <本链路可以正式跑通>
2. <本链路符合方法级设计文档>
3. <失败时不会留下半成正式状态>
```

## 10. Prompt 中推荐固定出现的要求

建议你以后给 Codex 下任务时，尽量长期固定出现以下要求。

### 10.1 先出计划再编码
建议固定写：

- 先输出任务理解、实施计划、文件改动清单、风险点
- 在我确认后再开始编码

如果你想提高效率，也可以写：

- 先输出计划与文件清单，然后直接开始编码

### 10.2 明确“不允许扩展未来系统”
建议固定写：

- 不允许主动扩展强化、洗练、宝石、套装、经卷、world level 等未来系统

### 10.3 明确“不允许越权改架构”
建议固定写：

- 不允许擅自改变五层服务分层口径
- 不允许把正式业务逻辑写回 Controller

### 10.4 明确“必须复用正式链路”
建议固定写：

- 若任务涉及正式业务动作，必须复用正式 Workflow / Domain，不得复制旁路逻辑

### 10.5 明确“必须给出自测步骤”
建议固定写：

- 完成后必须给出最小自测步骤和未完成项

## 11. Prompt 使用建议

### 11.1 不要每次从零写 Prompt
后续应以本文档模板为底稿，只替换：

- 任务名称
- 本次只做
- 本次不做
- 必读文档
- 验收标准

### 11.2 不要把所有要求塞进一次任务
如果任务很大，建议拆成：

- 先做业务链
- 再做接口
- 再做后台
- 再做修复

### 11.3 大任务要先拆 Prompt，再拆实现
不要直接发“把角色系统全部做完”。

应先拆成：

- 角色创建链 Prompt
- 角色接口 Prompt
- 角色后台查询页 Prompt
- 角色链测试 / 修复 Prompt

### 11.4 Prompt 不要和口头补充互相冲突
如果口头补充与文档冲突，应在 Prompt 里明确：

- 本次以哪条口头补充为准
- 改动范围是什么
- 是否需要同步更新文档

## 12. 当前阶段结论

1. Codex 的任务 Prompt 必须长期模板化，而不是每次自由发挥
2. 每次正式任务 Prompt 都应至少明确：任务目标、任务边界、必读文档、实现要求、输出顺序、验收标准、禁止事项
3. 业务链任务、接口任务、后台任务、修复任务、重构任务，应使用不同子模板
4. Prompt 中必须长期固定强调：先读文档、先给计划、再编码、不得扩展未来系统、不得越权写业务逻辑
5. 后续所有 Codex 任务下发，都建议以本文档为模板来源进行复用

本文档可直接作为项目正式《Codex 任务下发 Prompt 模板》文档使用，并与《Codex 主开发协作清单》一起作为 Codex 主开发协作体系的核心基础文档。
