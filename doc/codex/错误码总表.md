

# 《山海巡厄录》错误码总表

## 1. 文档目的

本文档用于在《项目总览》《接口字段级设计》《接口示例文档》《高风险点与约束清单》以及各方法级设计文档基础上，进一步统一《山海巡厄录》第一阶段正式错误码的编号规则、模块划分、命名规则、错误语义、推荐返回文案与使用边界。

本文档的目标不是替代异常类设计，而是解决以下问题：

- 第一阶段正式错误码到底有哪些
- 不同模块的错误码号段如何划分
- 相同错误语义在不同接口中如何保持一致
- Codex 开发时应如何统一使用错误码，而不是随手返回字符串
- 后续测试、联调、接口文档如何基于统一错误码体系推进

本文档定位为：

- 第一阶段正式错误码基线文档
- 接口错误响应统一口径文档
- 异常与错误码映射参考文档
- Codex 实现接口、Workflow、Domain 异常返回的重要依据

---

## 2. 错误码设计总原则

### 2.1 错误码必须稳定且可复用
错误码一旦进入正式接口响应，就不应频繁变动。

当前阶段要求：

- 同一类错误尽量复用同一个错误码
- 不要因为接口不同就随意创建重复语义的新错误码
- 错误码的含义必须清晰、可长期维护

### 2.2 错误码与错误文案分离
错误码是程序识别依据，`message` 是展示摘要。

因此：

- `code` 必须稳定
- `name` 必须稳定
- `message` 可以在不改变语义的前提下适度优化

### 2.3 错误码优先表达业务语义
第一阶段主要是业务开发阶段，因此错误码优先覆盖：

- 角色状态错误
- 装备穿戴错误
- 战斗准备 / 结算错误
- 掉落解析错误
- 奖励发放错误
- 入包错误
- 参数错误

而不是优先细分所有系统级异常。

### 2.4 错误码要能映射到模块与链路
一个错误码至少应能帮助定位：

- 属于哪个模块
- 属于哪类失败
- 大概在哪条链路发生

### 2.5 第一阶段统一使用整数错误码
当前阶段正式接口错误码统一使用整数。

成功约定：

- `code = 0`

失败约定：

- `code != 0`

## 3. 错误码号段划分

第一阶段建议按模块划分错误码号段。

### 3.1 通用与参数层
- `10000 - 10099`

### 3.2 角色模块
- `10100 - 10199`

### 3.3 装备与换装模块
- `10200 - 10299`

### 3.4 主线 / 关卡模块
- `10300 - 10399`

### 3.5 战斗准备模块
- `10400 - 10499`

### 3.6 战斗结算模块
- `10500 - 10599`

### 3.7 掉落解析模块
- `10600 - 10699`

### 3.8 奖励发放模块
- `10700 - 10799`

### 3.9 入包模块
- `10800 - 10899`

### 3.10 后台 / 审计 / 修复模块
- `10900 - 10999`

当前阶段先以这些号段为准，不要继续随意拆更细的号段。

---

## 4. 错误码命名规则

错误码常量名统一建议使用：

```text
<模块>_<错误语义>
```

统一使用：

- 全大写
- 下划线分隔

例如：

- `CHARACTER_NOT_FOUND`
- `CHARACTER_FORBIDDEN`
- `EQUIPMENT_INSTANCE_NOT_FOUND`
- `EQUIPMENT_SLOT_NOT_COMPATIBLE`
- `BATTLE_CONTEXT_INVALID`
- `DROP_RESOLVE_FAILED`
- `REWARD_ALREADY_GRANTED`
- `INVENTORY_WRITE_FAILED`

不建议：

- `CharacterError1`
- `ErrEquip01`
- `MainWeaponWrong`

---

## 5. 通用返回结构约定

第一阶段接口失败响应统一示例：

```json
{
  "code": 10101,
  "message": "角色不存在",
  "data": null
}
```

当前阶段约定：

- `code`：正式错误码
- `message`：推荐中文摘要
- `data`：失败时统一为 `null`

如后续需要更详细调试信息，可在非正式环境追加：

- `debug_message`
- `trace_id`

但第一阶段正式接口不强制要求。

## 8. Codex 使用错误码时的要求

### 8.1 不得临时自造错误码
Codex 在实现接口或业务链时，不应随手新增：

- 随机错误码
- 语义重复错误码
- 临时调试错误码

如现有错误码不够，应先补文档，再落代码。

### 8.2 不得只返回字符串不返回正式错误码
例如不应只返回：

```json
{
  "message": "角色不存在"
}
```

而应返回统一正式结构。

### 8.3 相同语义必须复用相同错误码
例如：

- 角色不存在，统一使用 `10101`
- 装备实例不存在，统一使用 `10201`
- 入包失败，统一使用 `10806`

不能不同接口写出不同错误码。

### 8.4 高风险链路必须优先用明确错误码
以下链路必须优先使用明确错误码，而不是笼统返回：

- 角色创建
- 换装
- 战斗准备
- 战斗结算
- 掉落解析
- 奖励发放
- 入包

这些是第一阶段最核心、最容易联调反复的地方。

## 10. 后续维护规则

### 10.1 新增错误码时的要求
若后续需要新增错误码，必须同时补充：

- `code`
- `name`
- 推荐 `message`
- 所属模块
- 使用场景

### 10.2 不要随意复用错误码表达不同语义
例如：

- 不要把“角色不存在”和“角色无权限访问”共用一个错误码
- 不要把“奖励已发放”和“幂等冲突”共用一个错误码

### 10.3 先补文档，再改实现
后续扩展时，建议顺序为：

1. 先补错误码总表
2. 再补异常类
3. 再补接口返回
4. 再更新接口示例文档

## 11. 当前阶段结论

1. 第一阶段正式错误码体系已经具备可开发、可联调、可验收的基础
2. 错误码已经按模块完成号段划分，并形成统一命名规则
3. 角色、装备、关卡、战斗、掉落、奖励、入包等核心主链的正式错误码已经覆盖
