

# 《山海巡厄录》认证与接口公共规则

## 1. 文档目的

本文档用于在《项目总览》《接口字段级设计》《接口示例文档》《错误码总表》《枚举总表》《Laravel 代码目录与命名规范》以及《Codex 主开发协作清单》基础上，进一步统一《山海巡厄录》第一阶段正式接口的认证方式、用户身份获取规则、公共请求约定、公共响应约定、分页约定、时间字段约定、幂等约定与环境差异说明。

本文档的目标不是替代单个接口字段设计，而是解决以下公共问题：

- 第一阶段接口到底采用什么认证方式
- 用户 ID 应从哪里取，不应该从哪里取
- 哪些接口必须认证，哪些接口可以匿名访问
- 分页、时间、布尔值、枚举值、空值、错误响应等公共口径如何统一
- Codex 落代码时，接口公共层应该怎么统一，不要每个接口各写一套

本文档定位为：

- 第一阶段正式接口公共规则文档
- 认证方式统一口径文档
- 中间件、基类 Controller、统一响应封装的实现依据文档
- Codex 落接口代码时的重要公共基线文档

---

## 2. 第一阶段认证总原则

### 2.1 第一阶段统一采用 Bearer Token 认证

第一阶段正式前台业务接口统一采用：

- `Authorization: Bearer <token>`

作为认证方式。

当前阶段不建议同时混用：

- session 认证
- cookie 登录态认证
- 自定义 header 多套认证方案

这样做的目的，是让：

- 移动端接口口径稳定
- 接口示例统一
- Codex 落代码时减少分歧

### 2.2 Laravel 实现建议优先使用 Sanctum 或现有统一 Token 中间件

如果项目尚未定死具体方案，当前阶段建议口径如下：

- Laravel 原生推荐方案优先考虑 `Sanctum`
- 如果项目已有稳定的统一 token 中间件，也可以沿用
- 但无论内部实现如何，对接口调用方统一表现为 `Bearer Token`

因此，本文档关注的是：

- 对外认证规则
- 用户身份获取规则
- 公共接口口径

而不是把认证内部实现绑死到某个特定包。

### 当前 backend 真实实现说明

当前 backend 已稳定收口为：

- 前台业务接口统一使用 `auth:api`
- `auth.guards.api.driver = token`
- `auth.guards.api.provider = users`
- `auth.guards.api.hash = true`
- Bearer Token 实际通过 `users.api_token` 的 hash token guard 校验

因此，第一阶段当前对外正式口径虽然仍然是 `Authorization: Bearer <token>`，但根目录文档回写时应以当前 hash token guard 实现为准，不再按“尚未确定具体认证方案”的旧假设描述。

### 2.3 用户身份必须从认证上下文中获取，不从请求体中传入

前台正式业务接口中，当前用户身份必须通过认证上下文获取。

也就是说：

- `user_id` 不应由前端在请求体中传入
- `user_id` 不应由前端在 query string 中传入
- `user_id` 应由服务端根据 token 解析得到

这是第一阶段的硬规则。

### 2.4 接口认证与后台认证分离

当前阶段应明确区分：

- 前台业务接口认证
- 后台管理接口认证

本文档主要约束：

- 前台业务接口

后台接口如已有独立后台认证体系，可单独维护，但不应污染前台业务接口口径。

---

## 3. 用户身份获取规则

## 3.1 前台业务接口统一从认证上下文获取当前用户

所有正式前台业务接口，统一使用类似以下方式获取当前用户：

```php
$user = $request->user();
$userId = $user->id;
```

或通过统一认证帮助方法获取。

### 当前阶段要求

- Controller 不接收 `user_id` 作为业务参数
- Workflow / Domain 接收的 `userId` 必须来自认证上下文
- 不允许前端伪造 `user_id` 操作他人数据

---

## 3.2 路径参数与用户身份必须分开

例如：

- `/api/characters/{character_id}` 中的 `character_id` 是资源 ID
- 当前用户是谁，必须从 token 获取

不能把：

- `character_id`
- `user_id`

混成一套资源归属判断。

正确顺序应为：

1. 从 token 取当前用户
2. 查询目标资源
3. 校验目标资源是否归属当前用户
4. 通过后再执行后续逻辑

---

## 3.3 禁止的做法

当前阶段禁止以下做法：

- 请求体直接传 `user_id`
- query 参数直接传 `user_id`
- Controller 直接信任前端传入的归属关系
- 只靠前端页面判断“这是我的角色”

---

## 4. 接口认证范围规则

## 4.1 默认规则：前台业务接口全部要求认证

当前阶段以下接口默认都必须认证：

- 角色信息接口
- 背包列表接口
- 穿戴列表接口
- 装备穿戴接口
- 装备卸下接口
- 关卡难度列表接口（如果包含玩家奖励状态）
- 首通奖励状态接口
- 战斗准备接口
- 战斗结算接口
- 装备实例详情接口
- 发奖记录查询接口

原因：

- 它们都涉及玩家状态或玩家归属数据

---

## 4.2 可匿名接口的范围

当前阶段只有“纯静态公共配置接口”才可以考虑匿名访问，例如：

- 纯章节列表
- 纯关卡基础信息
- 不含玩家进度与奖励状态的公共静态字典

但为了简化第一阶段实现，即使这些接口当前统一要求认证，也完全可接受。

### 当前阶段建议

为了减少分支口径，建议第一阶段：

- 前台 API 暂时统一要求认证

后续如果需要开放公共配置接口，再单独拆匿名接口组。

---

## 5. 中间件与路由分组规则

## 5.1 前台 API 路由建议统一挂认证中间件

推荐按业务域组织路由，并统一挂前台认证中间件。

示例：

```php
Route::middleware(['auth:sanctum'])->prefix('api')->group(function () {
    Route::prefix('characters')->group(function () {
        Route::get('{character}', ...);
        Route::get('{character}/equipment-slots', ...);
        Route::post('{character}/equip', ...);
        Route::post('{character}/unequip', ...);
    });

    Route::get('inventory', ...);

    Route::prefix('battles')->group(function () {
        Route::post('prepare', ...);
        Route::post('settle', ...);
    });
});
```

### 当前阶段要求

- 不要每个接口自己零散判断是否已登录
- 应统一通过中间件完成认证前置校验

---

## 5.2 后台接口应使用独立中间件组

后台管理接口不应直接复用前台业务接口认证组。

建议后台使用独立路由分组与独立认证 / 权限中间件，例如：

- `auth:admin`
- `can:*`
- `permission:*`

本文档不展开后台权限模型细节，但当前阶段应保持：

- 前台业务接口与后台管理接口分组分离

---

## 6. 公共请求规则

## 6.1 Content-Type 规则

所有需要请求体的正式业务接口，统一要求：

```http
Content-Type: application/json
```

GET 请求通常不需要请求体。

---

## 6.2 Accept 规则

建议所有正式接口统一：

```http
Accept: application/json
```

若服务端检测到非 JSON 请求，也应尽量返回 JSON 格式的标准错误结果，而不是 HTML 错误页。

---

## 6.3 请求字段命名规则

第一阶段接口请求字段统一使用：

- `snake_case`

例如：

- `character_id`
- `stage_difficulty_id`
- `equipment_instance_id`
- `target_slot_key`
- `battle_context_id`

禁止在同一项目中混用：

- `camelCase`
- `PascalCase`
- 缩写不统一字段

---

## 6.4 请求字段最小化原则

请求体中只传真正必要的业务输入。

例如：

### 合理

```json
{
  "equipment_instance_id": 5001,
  "target_slot_key": "main_weapon"
}
```

### 不合理

```json
{
  "user_id": 2001,
  "character_id": 1001,
  "equipment_instance_id": 5001,
  "target_slot_key": "main_weapon",
  "character_name": "青山"
}
```

因为：

- `user_id` 应从认证上下文获取
- `character_name` 不是本动作所必需输入

---

## 7. 公共响应规则

## 7.1 统一成功响应外壳

第一阶段正式成功响应统一为：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 当前阶段要求

- `code` 成功固定为 `0`
- `message` 成功固定建议为 `ok`
- `data` 承载正式业务结果

---

## 7.2 统一失败响应外壳

第一阶段正式失败响应统一为：

```json
{
  "code": 10101,
  "message": "角色不存在",
  "data": null
}
```

### 当前阶段要求

- 失败必须返回正式错误码
- `data` 失败统一为 `null`
- 不要返回不稳定的自定义结构

---

## 7.3 不要混用多套响应风格

当前阶段禁止同项目中同时出现：

- 一部分接口返回 `{ success: true }`
- 一部分接口返回 `{ code: 0 }`
- 一部分接口返回裸数组
- 一部分接口失败时返回 HTML 页面

统一必须以正式 JSON 外壳为准。

---

## 8. 分页公共规则

## 8.1 需要分页的接口

当前阶段建议以下接口支持分页：

- 背包列表接口（如数据量增长）
- 发奖记录查询接口
- 后台查询类接口

### 当前阶段说明

若第一阶段某些接口数据量较小，可先不强制分页，但分页结构一旦启用应统一。

当前真实实现补充：

- `GET /api/inventory` 当前会接收 `page`、`page_size` 以保持参数口径稳定
- 但当前真实响应仍是 `stack_items + equipment_items`
- 该接口尚未启用正式 `list + pagination` 返回结构，文档与示例不得伪造 `pagination`

---

## 8.2 分页请求参数规则

统一建议使用：

- `page`
- `page_size`

例如：

```http
GET /api/reward-grants?page=1&page_size=20
```

### 当前阶段默认值建议

- `page` 默认 `1`
- `page_size` 默认 `20`
- `page_size` 上限建议 `100`

---

## 8.3 分页响应结构规则

统一建议为：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "list": [],
    "pagination": {
      "page": 1,
      "page_size": 20,
      "total": 0
    }
  }
}
```

### 当前阶段要求

- 使用分页的接口统一返回 `list + pagination`
- 不要有的接口返回 `items`，有的接口返回 `rows`
- 未启用正式分页的接口，保持其当前真实结构，不要为了对齐旧示例硬塞 `pagination`

---

## 9. 时间字段公共规则

## 9.1 时间字段命名

当前阶段统一使用：

- `created_at`
- `updated_at`
- `granted_at`

不建议混用：

- `create_time`
- `grantTime`
- `updateAt`

---

## 9.2 时间字段格式

当前阶段正式接口返回时间字段时，统一建议使用：

```text
YYYY-MM-DD HH:mm:ss
```

例如：

```json
{
  "created_at": "2026-03-20 12:00:00"
}
```

### 当前阶段说明

- 先统一字符串格式
- 不强制同时返回时间戳
- 如后续确有需要，再统一扩展

---

## 10. 空值、布尔值、枚举值规则

## 10.1 空值规则

当前阶段接口中：

- 不存在的可空对象，统一返回 `null`
- 不要用空字符串 `""` 代替对象空值
- 不要用 `[]` 代替单对象空值

例如：

```json
{
  "equipment": null
}
```

---

## 10.2 布尔值规则

当前阶段与数据库口径兼容时，接口中允许使用：

- `0`
- `1`

例如：

- `is_active`
- `is_cleared`
- `has_reward`
- `has_granted`

### 当前阶段说明

为了与现有数据库与字段文档保持一致，第一阶段先接受 `0/1`。

后续若要统一为 `true/false`，必须整体评估并统一变更，不应局部改。

---

## 10.3 枚举值规则

接口中所有正式枚举字段统一返回：

- 正式英文 value

例如：

- `common`
- `equipment`
- `main_weapon`
- `weighted_repeat`
- `first_clear`
- `pending`

不应直接返回中文枚举值作为正式业务值。

中文展示由：

- 前端字典
- 后台映射
- 文档说明

来承担。

---

## 11. 幂等公共规则

## 11.1 当前阶段必须关注幂等的链路

以下链路当前阶段必须具备幂等意识：

- 奖励发放链
- 战斗结算链中的奖励发放承接
- 后台补发链

---

## 11.2 幂等键的使用原则

对于正式奖励发放链：

- `idempotency_key` 由服务端根据业务上下文生成
- 不要求前端直接传随机幂等键
- 相同业务请求重试，应落到同一正式幂等键

### 当前阶段要求

- 不允许每次重试都生成新的随机幂等键
- 不允许前端伪造业务幂等语义

---

## 11.3 接口层对幂等的态度

接口层本身不负责发明幂等规则，而是：

- 调正式 Workflow
- 由 Workflow / Domain 层根据文档生成与校验幂等键

---

## 12. 错误响应与异常映射公共规则

## 12.1 正式业务错误统一映射到错误码总表

当前阶段所有正式业务接口失败，都应优先映射到《错误码总表》中的正式错误码。

例如：

- 角色不存在 → `10101`
- 装备实例不存在 → `10201`
- 战斗上下文无效 → `10501`
- 奖励已发放 → `10706`
- 入包失败 → `10806`

---

## 12.2 不要直接把异常文本裸返回给前端

例如不应直接返回：

```json
{
  "code": 10005,
  "message": "SQLSTATE[23000]: Integrity constraint violation ...",
  "data": null
}
```

正式接口应返回：

- 稳定错误码
- 简洁业务 message

详细异常信息只应进入日志或 debug 环境。

---

## 12.3 非业务异常统一降级为系统错误

若发生未映射的系统级异常，当前阶段统一建议返回：

```json
{
  "code": 10005,
  "message": "系统异常",
  "data": null
}
```

同时记录完整日志。

---

## 13. 环境差异规则

## 13.1 开发环境

开发环境可额外支持：

- 更详细日志
- 调试堆栈
- SQL 错误输出到日志

但不建议直接暴露给正式前台接口响应。

---

## 13.2 测试环境

测试环境应尽量保持与正式环境一致的响应结构，只是在日志与调试辅助上更宽松。

---

## 13.3 生产环境

生产环境要求：

- 只返回正式响应结构
- 不暴露堆栈
- 不暴露底层异常详情
- 必须记录关键错误日志与 trace 信息

---

## 14. Codex 落接口代码时的公共要求

Codex 在实现接口时，必须统一遵守以下规则：

1. 用户身份从认证上下文获取，不从请求体中取
2. 前台业务接口默认挂认证中间件
3. 请求字段统一使用 `snake_case`
4. 成功响应统一使用 `code/message/data`
5. 失败响应统一使用正式错误码与 `data = null`
6. 分页接口统一使用 `page/page_size` 与 `list/pagination`
7. 正式枚举字段统一返回英文 value
8. 时间字段统一使用 `YYYY-MM-DD HH:mm:ss`
9. 不允许每个接口各自造响应风格
10. 不允许在 Controller 中散写认证与归属判断逻辑

---

## 15. 当前阶段建议补充的实现点

为了让 Codex 更稳定落地公共接口层，建议后续在代码中统一补这些基础设施：

1. 统一 API 响应封装类
2. 统一业务异常基类
3. 统一错误码映射层
4. 统一认证中间件配置
5. 统一分页响应帮助方法
6. 统一 Enum 显示映射工具（后台可选）

这些不一定要先全部做完，但至少要保持统一方向，不要每个模块各写一套。

---

## 16. 当前阶段结论

1. 第一阶段前台正式业务接口统一采用 `Bearer Token` 认证口径
2. 当前用户身份必须从认证上下文获取，不允许前端传 `user_id`
3. 前台业务接口建议统一挂认证中间件，后台接口与前台接口应分组分离
4. 成功 / 失败响应外壳、分页结构、时间字段、空值、布尔值、枚举值已经形成统一公共口径
5. 错误响应必须优先映射到《错误码总表》，正式枚举值必须优先映射到《枚举总表》
6. 后续 Codex 落接口代码时，应严格以本文档作为认证与公共接口层的统一基线

本文档可直接作为项目正式《认证与接口公共规则》文档使用，并作为中间件、统一响应层、异常映射层与前后端联调公共规则的统一基线。
