# 《山海巡厄录》最小联调种子数据

## 1. 文档目的

本文档用于在《项目总览》《数据库设计文档》《接口字段级设计》《接口示例文档》《错误码总表》《枚举总表》《认证与接口公共规则》以及各方法级设计文档基础上，进一步明确《山海巡厄录》第一阶段后端主链开发所需的**最小联调种子数据集合**。

本文档的目标不是一次性填满正式运营数据，而是为以下工作提供一套可以稳定复用的最小数据底座：

- Codex 开发后的本地联调
- 后端自测
- 接口联调
- 后台配置页验证
- 战斗准备 / 结算闭环验证
- 掉落链 / 奖励链 / 入包链联调

本文档定位为：

- 第一阶段最小种子数据基线文档
- Seeder / SQL / JSON 样例设计依据文档
- Codex 落地最小测试数据时的重要参考文档
- 第一阶段接口和主链联调的最小数据标准

---

## 2. 最小联调数据设计总原则

### 2.1 只追求“最小闭环可用”，不追求内容丰富

当前阶段的种子数据目标是：

- 能创建角色
- 能读角色状态
- 能读背包 / 穿戴
- 能准备战斗
- 能结算战斗
- 能验证掉落链
- 能验证奖励链
- 能验证入包链

而不是：

- 一次性填充大量职业、地图、怪物、掉落、奖励内容

### 2.2 数据必须覆盖第一阶段六条主链

最小联调数据必须至少覆盖：

1. 配置可读主链
2. 状态可写主链
3. 掉落可结算主链
4. 奖励可防重主链
5. 装备可实例化主链
6. 角色可重算主链

### 2.3 种子数据必须尽量稳定、可重复导入

当前阶段建议最小联调数据做到：

- 固定主键或稳定业务 ID
- 可重复执行
- 不依赖随机初始化顺序
- 可以由 Seeder 或 SQL 稳定生成

### 2.4 先保证结构链路，再逐步扩充内容

当前阶段优先满足：

- 有职业
- 有角色
- 有物品
- 有装备模板
- 有怪物
- 有章节 / 关卡 / 难度
- 有掉落组和奖励组
- 有来源绑定

而不是先堆大量数值和大量地图内容。

## 3. 当前阶段最小联调目标

最小联调数据导入完成后，至少应支持以下验证场景：

### 3.1 角色与换装
- 创建 1 个正式角色
- 自动初始化 12 个固定穿戴槽
- 背包中有至少 2 件装备实例可供穿戴
- 可以验证主武器穿戴
- 可以验证副武器穿戴 / 不兼容情况

### 3.2 战斗准备
- 至少存在 1 个章节
- 至少存在 1 个关卡
- 至少存在 2 个难度
- 每个难度至少绑定 2~3 个怪物
- 能正常生成战斗准备结果

### 3.3 战斗结算
- 至少有 1 套普通怪掉落组
- 至少有 1 套精英 / Boss 掉落组
- 至少有 1 套首通奖励组
- 结算后能产出材料
- 结算后能产出装备模板对象
- 产出后能通过入包链形成可堆叠物与装备实例

### 3.4 奖励防重
- 同一关卡难度首通奖励只能成功发放一次
- 重复结算不会重复发放首通奖励

## 6. 最小静态配置建议

- `class_id = class_jingang`
- `class_name = 金刚`
- `is_enabled = 1`
- 基础属性倾向：
  - 体魄较高
  - 物理防御较高
  - 法力较低

### `class_fashi`

建议字段：

- `class_id = class_fashi`
- `class_name = 法师`
- `is_enabled = 1`
- 基础属性倾向：
  - 法力较高
  - 法术强度较高
  - 物理防御较低

### 当前阶段建议

职业具体公式如果已另有文档，以职业系统设计和属性体系设计为准。

- 职业存在
- 可启用
- 可用于创建角色

即可。

## 6.2 物品最小配置建议

至少需要保证：

- 所有最小联调用到的 `item_id` 都存在

  - `item_name`

  - `icon`
  - `is_enabled`

- 材料：`common`
- 奖励物：`common` / `rare`
- 装备：至少覆盖 `common` / `rare`

## 6.3 装备模板最小配置建议

每个装备模板至少建议具备：

- `item_id`
- `item_name`
- `equipment_slot`
- `rarity`
- `level_required`
- `is_enabled`
- 基础属性字段
- 武器类别 / 副武器类别（若适用）
- 双手 / 单手规则（若适用）

### 最小联调重点

至少要满足两组验证：

1. **主武器 + 副武器可兼容穿戴**

## 14. 当前阶段结论
