# 《山海巡厄录》第一阶段测试与验收说明

## 1. 文档目的

本文档用于把第一阶段当前真实可执行的测试矩阵、统一验收入口、文档分工和最小自测顺序收口到一处。

本文档只记录当前仓库已经稳定落地的能力，不扩写未来系统。

---

## 2. 当前统一口径

第一阶段验收统一以当前真实 backend 为准：

- 路由以 `backend/routes/api.php`、`backend/routes/web.php`、`backend/routes/console.php` 为准
- 请求与响应以各 `ApiRequest`、`Resource`、`Admin` 控制器为准
- 错误码以 `backend/app/Support/ErrorCode.php` 与《错误码总表》为准
- 枚举值以 `backend/app/Enums` 与《枚举总表》为准
- 机器可读契约以 `backend/docs/api/phase-one-frontend.openapi.json` 为准
- 人读示例与联调说明以根目录 `doc/codex/*.md` 为准

如果代码、测试、OpenAPI 和根目录文档出现漂移，处理顺序必须是：

1. 先核对真实代码
2. 再修正测试 / OpenAPI 契约
3. 最后回写根目录正式文档

## 4. 统一验收入口

推荐从 `backend/` 目录执行以下单命令：

```bash
composer phase-one:acceptance
```

该入口会顺序执行：

1. `php artisan phase-one:diagnose --profile=acceptance`
2. 第一阶段定向验收测试集

当前聚合脚本位置：

- `backend/scripts/phase-one-acceptance.sh`

### 仓库自动验收

当前仓库最小 GitHub Actions workflow 位于：

- `.github/workflows/phase-one-acceptance.yml`

该 workflow 不重写验收逻辑，而是复用当前统一口径，顺序执行：

1. 准备 `.env` 与 `database/database.sqlite`
2. `composer install`
3. `php artisan key:generate --force`
4. `php artisan migrate:fresh --seed --force`
5. `php artisan phase-one:diagnose --profile=acceptance --json`
6. `php artisan phase-one:contract-drift-check --json`
7. `composer phase-one:acceptance`

同时会补两项轻量前置检查：

- `composer validate --strict`
- PHP lint（`artisan` + `app/config/database/routes/tests`）

建议排障顺序：

1. 如果卡在 `Migrate and seed database`，优先检查 migration / seed / sqlite 前置。
2. 如果卡在 `Diagnose phase-one acceptance readiness`，优先看 `APP_KEY`、`workflow_lock`、`seed_data`、`admin_bootstrap`、`contract`。
3. 如果卡在 `Run phase-one contract drift guard`，优先看真实 routes / `ApiRequest` / OpenAPI / 根目录正式接口文档是否同步。
4. 如果卡在 `Run phase-one acceptance suite`，直接按失败的定向测试文件回查对应链路。

## 5. 最小执行前置

执行统一验收前，建议先完成：

```bash
cd backend
cp .env.example .env
touch database/database.sqlite
composer install
php artisan key:generate
php artisan migrate:fresh --seed
```

当前最小环境建议值：

- `SESSION_DRIVER=file`
- `QUEUE_CONNECTION=sync`
- `MAIL_MAILER=log`
- `CACHE_STORE=file`
- `WORKFLOW_LOCK_STORE=file`

当前最小联调账号：

- `users.id = 2001`
- `Authorization: Bearer test-token-2001`
- 后台：`admin / admin123456`

如果部署为多实例，请把 `WORKFLOW_LOCK_STORE` 切到 `redis` 等共享 store；`file` 只适合单机。

## 6. 建议验收顺序

若需要人工分步验收，建议按以下顺序执行：

1. `GET /up`
2. `GET /readyz`
3. `GET /readyz?profile=acceptance`
4. `php artisan phase-one:diagnose --profile=interop --json`
5. `php artisan phase-one:diagnose --profile=acceptance --json`
6. `php artisan phase-one:contract-drift-check --json`
7. `php artisan workflow-lock:check --json`
8. `php artisan test tests/Feature/Api/PhaseOnePlayerJourneySmokeTest.php`
9. `php artisan test tests/Feature/Api/PhaseOneBattleSettlementApiTest.php`
10. `php artisan test tests/Feature/Admin/PhaseOneAdminPagesTest.php`
11. 最后执行 `composer phase-one:acceptance`

## 7. 最小发版 / 回归清单

发版前或回归收口时，最小只做以下 5 项：

1. `php artisan phase-one:diagnose --profile=acceptance --json`
2. `php artisan phase-one:contract-drift-check --json`
3. `GET /readyz?profile=acceptance`
4. `composer phase-one:acceptance`
5. 人工确认 `backend/docs/api/phase-one-frontend.openapi.json` 与根目录 `doc/codex/接口示例文档.md` 已按本轮真实代码回写

执行原则：

- `diagnose` 负责环境与依赖前置
- `contract drift check` 负责真实代码 / OpenAPI / 正式文档防漂移
- `readiness` 负责部署与联调入口可用性
- `acceptance` 负责 phase-one 主链闭环回归
- OpenAPI / 根目录文档确认只做“本轮真实改动是否已同步”，不再单独维护第三套规则

## 8. 文档分工

为了避免双份文档继续漂移，后续统一按以下分工维护：

- backend 真实代码（`routes/api.php`、各 `ApiRequest`、各 `Resource`、Feature Tests）
  第一真相，任何正式契约调整都先落这里
- `backend/docs/api/phase-one-frontend.openapi.json`
  机器可读正式契约产物，用于 route / request / response / 测试同步校验
- `backend/docs/api/README.md`
  backend 侧联调入口、最小环境要求、ready / diagnose / 契约使用说明
- `doc/codex/接口示例文档.md`
  根目录人读版正式示例文档，只保留当前真实存在的 phase-one 前台接口
- `doc/codex/认证与接口公共规则.md`
  认证、CORS、响应外壳、分页、时间等公共约束
- `doc/codex/第一阶段测试与验收说明.md`
  测试矩阵、验收入口、最小自测顺序

维护规则：

1. 先改 backend 真实实现
2. 再改测试与 OpenAPI
3. 最后回写根目录正式文档
