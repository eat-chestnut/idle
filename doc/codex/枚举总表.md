# 《山海巡厄录》枚举总表

## 1. 文档目的

本文档用于在《项目总览》《接口字段级设计》《接口示例文档》《错误码总表》《Laravel 代码目录与命名规范》以及各系统设计文档基础上，进一步统一《山海巡厄录》第一阶段正式使用的枚举项、枚举值、枚举中文含义、使用范围、命名规则与后续维护原则。

本文档的目标不是替代数据库字段设计，也不是替代业务规则文档，而是解决以下问题：

- 第一阶段正式枚举到底有哪些
- 同一枚举在数据库、后端、接口、后台中如何统一
- 哪些字段必须用枚举，不能随意写字符串
- Codex 落代码时该如何统一使用枚举，而不是到处手写 magic string
- 后续扩展新系统时，枚举应如何维护

本文档定位为：

- 第一阶段正式枚举基线文档
- 数据库字段 / Enum 类 / 接口字段统一口径文档
- Codex 开发时的字符串常量约束文档
- 后续测试、联调、后台开发与代码审查的重要基线文档

## 2. 枚举设计总原则

### 2.1 正式业务状态优先使用枚举，不直接写自由字符串
以下场景应优先使用枚举，而不是直接在代码、数据库、接口中散写字符串：

- 稀有度
- 物品类型
- 装备槽位
- 掉落抽取规则
- 奖励来源类型
- 发奖状态
- 绑定状态
- 难度 key
- 怪物角色
- 主副武器类别

### 2.2 枚举值一旦进入数据库或接口，应尽量保持稳定
当前阶段要求：

- 已定稿枚举值不要轻易变更
- 不要因为显示文案调整而去改枚举值本身
- 接口与数据库优先用稳定英文值
- 中文仅用于展示层与后台辅助展示

### 2.3 枚举值统一使用小写 snake_case
当前阶段正式枚举值统一建议：

- 小写
- 单词间使用下划线 `_`

例如：

- `main_weapon`
- `weighted_single`
- `first_clear`
- `normal_enemy`

不建议：

- `MainWeapon`
- `weightedSingle`
- `FIRST_CLEAR`

### 2.4 同一语义只能有一套正式枚举值
例如：

- 不允许同时出现 `main_weapon` 和 `mainWeapon`
- 不允许同时出现 `rare` 和 `blue`
- 不允许同时出现 `success` 和 `done`

当前阶段必须统一一套正式值。

### 2.5 后台显示中文，数据库与接口优先使用英文值
统一原则：

- 数据库存英文枚举值
- 接口返回英文枚举值
- 后台页面可映射显示中文说明
- 文档中同时记录英文值和中文含义

## 3. 枚举命名规则

当前项目后端 Enum 类名统一建议：

- PascalCase
- 单数名词
- 语义清晰

例如：

- `Rarity`
- `ItemType`
- `EquipmentSlot`
- `DropRollType`
- `RewardSourceType`
- `GrantStatus`
- `BindType`
- `MonsterRole`
- `DifficultyKey`

不建议：

- `RarityEnumType`
- `ItemTypeService`
- `AllStatusEnum`

## 4.1 稀有度枚举 `rarity`

### 使用范围
- `items.rarity`
- `equipments.rarity`（若冗余或衍生展示）
- 接口物品摘要 / 装备摘要

## 6. 数据库、接口、后台三层统一规则

### 6.1 数据库层
- 存正式英文枚举值
- 不存中文
- 不混用多套值

### 6.2 接口层
- 返回正式英文枚举值
- 如前端确实需要中文，可由前端字典映射
- 当前阶段不建议在正式接口里大规模混返回中文 label

### 6.3 后台层
- 表单提交使用正式英文值
- 页面显示可映射为中文
- 下拉框应显示“中文 + value”的明确对应关系

## 7. Codex 使用枚举时的要求

### 7.1 不得在代码中散写 magic string
例如不应长期直接写：

```php
if ($slotKey === 'main_weapon') {
    ...
}
```

而应优先通过 Enum 类或统一常量引用。

### 7.2 不得自己发明新值
若文档中没有：

- `super_hard`
- `mythic`
- `offhand_weapon`

之类值，Codex 不得自行新增并直接落代码。

### 7.3 同一枚举值必须在全项目保持一致
例如：

- `weighted_repeat` 不能有地方写成 `weight_repeat`
- `first_clear` 不能有地方写成 `firstclear`
- `unbound` 不能有地方写成 `unbind`

### 7.4 先补文档，再新增枚举
若后续业务确实需要新枚举值，建议顺序为：

1. 先补《枚举总表》
2. 再补 Enum 类
3. 再补接口字段文档
4. 再补后台配置与代码实现

## 8. 当前阶段推荐优先接入枚举的链路

建议 Codex 先在以下链路中优先正式接入枚举：

1. 装备模板配置
2. 角色穿戴槽
3. 换装链
4. 掉落解析链
5. 奖励发放链
6. 战斗准备接口
7. 战斗结算接口
8. 后台下拉筛选与表单项

这些地方最容易因为字符串不统一而产生返工。

## 9. 后续维护规则

### 9.1 新增枚举时必须补四项
后续新增枚举时，至少补充：

- 枚举名称
- 英文 value
- 中文含义
- 使用范围

### 9.2 不要为了展示文案修改正式 value
例如：

- 想把“困难”改成“艰难”，应改显示中文，不应改 `hard`
- 想把“稀有”改成“珍稀”，应改显示中文，不应改 `rare`

### 9.3 枚举变更应同步检查三层
若枚举有变更，应同步检查：

1. 数据库已有值
2. 后端 Enum 类
3. 接口字段与前端字典映射

## 10. 当前阶段结论

1. 第一阶段正式业务所需的核心枚举已经基本收敛完成
2. 稀有度、物品类型、装备槽位、掉落抽取规则、奖励来源、发奖状态、绑定状态、怪物角色、难度 key 等关键枚举已经具备可开发基础
3. 后续 Codex 落代码时，应严格以本文档为准使用 Enum 类与正式枚举值，不得到处散写 magic string
4. 后续数据库设计、接口设计、后台开发、联调与测试，都应以本文档为统一枚举基线

本文档可直接作为项目正式《枚举总表》文档使用，并作为 Codex 落代码、Enum 类编写、字段约束与前后端联调的统一枚举基线。
