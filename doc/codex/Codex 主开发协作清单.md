

# 《山海巡厄录》Codex 主开发协作清单

## 1. 文档目的

本文档用于在《项目总览》《核心开发总纲》《模块拆分与职责清单》《后端服务分层设计》《核心业务链路设计》《第一阶段六条主链清单》《高风险点与约束清单》《接口开发清单》《接口字段级设计》以及各方法级设计文档基础上，进一步明确《山海巡厄录》项目在以 Codex 为主开发代理时的协作规则、任务下发方式、执行边界、输出要求、验收要求与禁止事项。

本文档的目标不是替代系统设计文档，而是解决以下协作问题：

- 如何让 Codex 按项目既定文档主线稳定开发
- 如何避免 Codex 自行扩展未定稿功能
- 如何避免一次任务改动过大、跨模块、跨边界
- 如何让 Codex 的输出结构稳定，可复审、可验收、可追踪
- 如何让不同任务之间保持一致的开发风格与架构口径

本文档定位为：

- Codex 主开发长期协作文档
- Codex 任务执行边界文档
- Codex 输出规范与验收规范文档
- 后续 `Codex 任务下发 Prompt 模板.md` 的上层规则文档

---

## 2. Codex 在本项目中的角色定位

### 2.1 Codex 的正式角色
在本项目中，Codex 的定位不是“自由发挥型代码助手”，而是：

**严格依据既有文档与既定边界执行开发任务的主开发代理。**

### 2.2 Codex 应承担的职责
Codex 应承担：

1. 按正式文档实现代码
2. 按正式方法设计拆分服务与方法
3. 按正式接口清单与接口字段级设计输出接口
4. 按正式数据库结构、约束与链路规则实现功能
5. 在每次任务前给出改动计划、改动文件清单与自测建议

### 2.3 Codex 不应承担的职责
Codex 不应：

- 自行重做系统架构
- 自行新增未定稿系统
- 自行扩大任务边界
- 自行替用户做产品方向决策
- 自行把未来预留系统提前做成正式实现

### 2.4 本项目对 Codex 的使用原则
本项目不是让 Codex “想到哪做哪”，而是让 Codex：

- 严格以文档为准
- 严格以当前阶段目标为准
- 严格以单次任务边界为准
- 严格按先计划、再修改、再自测、再汇报的顺序执行

## 3. Codex 协作总原则

### 3.1 先读文档，再动代码
任何正式开发任务开始前，Codex 必须先阅读该任务要求的项目文档。

至少应优先读：

- 《项目总览》
- 本次任务相关的开发思路文档
- 本次任务相关的方法级设计文档
- 本次任务相关的接口字段级设计文档（如涉及接口）

### 3.2 先给计划，再做改动
Codex 在正式改代码前，必须先输出：

1. 对任务的理解
2. 拟实施方案
3. 将修改的文件列表
4. 关键风险点

未经这一轮结构化输出，不应直接开始大规模改动。

### 3.3 一次任务只解决一个明确边界的问题
每次 Codex 任务必须是“单任务边界”，例如：

- 只做角色创建链路
- 只做换装链路
- 只做战斗准备接口
- 只做奖励发放服务

不应一次 Prompt 同时要求：

- 后台 + 接口 + 多条业务链 + 多套重构

除非这是明确拆解过的大任务，并且已逐项确认边界。

### 3.4 不允许为了方便而绕过正式设计
Codex 不允许因为“实现更快”而：

- 把逻辑塞进 Controller
- 绕过统一 Service
- 直接改数据库状态代替正式链路
- 在接口层重写业务逻辑
- 在页面层拼正式状态

### 3.5 输出必须可复审、可验收
Codex 的每次任务输出，必须让人能快速核对：

- 改了什么
- 没改什么
- 为什么这样改
- 是否符合当前文档
- 是否还有未完成项

## 5. 当前项目正式开发边界

### 5.1 当前阶段允许开发的正式范围
当前阶段允许 Codex 正式开发的范围主要集中在：

1. 基础配置读取链
2. 角色创建链
3. 换装链
4. 战斗准备链
5. 战斗结算链
6. 掉落解析链
7. 奖励发放链
8. 入包承接链
9. 第一阶段后台配置与查询页
10. 第一阶段核心接口

### 5.2 当前阶段只允许预留、不允许正式实现的范围
以下内容当前阶段只允许保留结构兼容，不允许正式深做：

- 强化
- 洗练
- 宝石
- 套装
- 经卷
- world level
- 活动系统
- 商店系统
- 复杂 BI 后台
- 复杂高级表现系统

### 5.3 未经明确允许，不得新增的新范围
若 Prompt 未明确要求，Codex 不得自行新增：

- 新表
- 新模块
- 新接口组
- 新后台资源页
- 新中间层抽象
- 新规则系统

## 6. 五层服务分层协作规则

本项目后端服务以五层设计为唯一正式口径。

### 6.1 Config 层
职责：
- 读取静态配置

Codex 不得在 Config 层中：
- 写状态
- 写事务
- 做结算
- 做发奖

### 6.2 Query 层
职责：
- 读取玩家正式状态与业务查询结果

Codex 不得在 Query 层中：
- 偷偷写状态
- 做事务
- 完成完整业务流程

### 6.3 Domain 层
职责：
- 承担单点业务动作与领域规则

Codex 不得在 Domain 层中：
- 同时承担完整流程编排
- 写页面专用拼接逻辑
- 复制后台专用逻辑

### 6.4 Workflow 层
职责：
- 组织完整流程
- 管理事务边界
- 组织多服务调用

Codex 不得在 Workflow 层中：
- 重写所有底层 Domain 规则
- 写与任务无关的通用大而全抽象

### 6.5 Admin 层
职责：
- 承担后台校验、引用检查、补发、修复逻辑

Codex 不得在 Admin 层中：
- 复制正式玩家链路实现
- 绕过正式 Workflow 直接改正式状态

---

## 7. 模板层 / 实例层协作规则

### 7.1 必须始终分离的对象
Codex 在任何任务中都必须保持以下分层：

- `items` 是物品模板，不是玩家持有状态
- `equipments` 是装备模板，不是玩家装备实例
- `inventory_stack_items` 是可堆叠持有状态
- `inventory_equipment_instances` 是装备实例状态
- `reward_groups` 是奖励模板
- `user_reward_grants` / `user_reward_grant_items` 是正式发放记录

### 7.2 不允许的典型错误
Codex 不得：

- 把装备写入可堆叠背包表
- 把强化结果写回装备模板
- 把发奖结果写回奖励模板
- 把玩家持有数量写回物品模板

---

## 8. 掉落链 / 奖励链协作规则

### 8.1 掉落链职责
掉落链只负责：

- 根据怪物来源和掉落组生成标准掉落结果

掉落链不负责：

- 入包
- 发奖
- 奖励防重
- 装备实例创建

### 8.2 奖励链职责
奖励链只负责：

- 对正式首通奖励进行校验、防重、记录写入、入包承接

奖励链不负责：

- 怪物随机掉落
- 掉落权重抽取

### 8.3 Codex 不得混用
Codex 不得：

- 把怪物掉落直接并入首通奖励逻辑
- 把首通奖励写成怪物掉落逻辑的一部分
- 把两条链共用一套模糊配置与执行逻辑

## 9. Codex 任务下发结构要求

每次给 Codex 下任务，建议至少包含以下结构。

### 9.1 任务名称
明确本次任务名称。

例如：
- 角色创建链路开发
- 换装链路开发
- 战斗准备接口开发

### 9.2 本次任务目标
只说明这次要完成什么。

### 9.3 本次只做什么
把本次允许实现的内容列清楚。

### 9.4 本次不做什么
把本次禁止扩展的内容列清楚。

### 9.5 必读文档
明确本次必须先读哪些文档。

### 9.6 实现要求
明确：
- 服务分层要求
- 方法设计要求
- 接口要求
- 事务要求
- 风险约束

### 9.7 输出顺序要求
明确：
1. 先给实施计划
2. 再给文件改动清单
3. 再开始编码
4. 再给自测建议
5. 再给未完成项

### 9.8 验收标准
明确做到什么算完成。

## 10. Codex 输出格式要求

Codex 每次正式任务输出，建议固定包含以下部分。

### 10.1 任务理解
简要说明对本次任务的理解。

### 10.2 实施计划
列出准备如何实现。

### 10.3 文件改动清单
列出预计修改的文件。

### 10.4 风险点
列出本次任务的关键风险。

### 10.5 编码实施结果
说明实际完成了哪些改动。

### 10.6 自测步骤
给出最基本的验证步骤。

### 10.7 未完成项 / 待确认项
明确哪些内容未做、哪些内容仍待确认。

## 11. Codex 禁止事项清单

### 11.1 禁止越权扩功能
未经任务明确要求，Codex 不得：

- 主动新增未来系统
- 主动补做无关页面
- 主动“顺手优化”其他模块

### 11.2 禁止跨层乱写
Codex 不得：

- 在 Controller 里堆正式业务逻辑
- 在 Query 层写状态
- 在 Config 层做流程
- 在 Admin 层复制正式玩家链逻辑

### 11.3 禁止绕过正式服务链
Codex 不得：

- 直接手写 SQL 绕过正式 Service
- 在后台工具里直接改正式背包状态
- 在接口层直接结算奖励或掉落

### 11.4 禁止破坏现有文档口径
Codex 不得：

- 继续使用“四层服务分层”旧口径
- 混用模板层与实例层
- 混用掉落链与奖励链
- 擅自变更正式字段命名

### 11.5 禁止过度设计
Codex 不得为了“看起来更高级”而：

- 提前抽象未来十几种还没出现的场景
- 提前引入复杂基础设施
- 提前设计大量当前不用的通用引擎

## 12. Codex 验收要求

每次 Codex 任务完成后，至少要按以下维度自查。

### 12.1 文档一致性
- 是否符合当前项目文档
- 是否违反当前阶段边界
- 是否与方法级设计冲突

### 12.2 分层一致性
- 是否符合五层服务分层
- 是否把逻辑写到了错误层级

### 12.3 主链一致性
- 是否真正服务当前主链
- 是否引入与当前主链无关的额外实现

### 12.4 高风险一致性
- 是否违反高风险点与约束清单
- 是否破坏唯一约束或事务边界

### 12.5 改动范围一致性
- 是否改动了未授权文件
- 是否跨了未授权模块

## 13. Codex 任务粒度建议

### 13.1 推荐的单任务粒度
推荐 Codex 的单次任务粒度控制在：

- 一条业务链
- 一个接口组
- 一个后台资源页组
- 一个方法级设计文档对应的正式实现

### 13.2 不推荐的单任务粒度
不推荐一次让 Codex：

- 同时做多条业务链
- 同时做数据库、后台、接口、前端、测试
- 同时做系统扩展和主链落地

### 13.3 原则
任务越清晰、边界越明确，Codex 输出越稳定。

---

## 14. 当前阶段推荐 Codex 开发顺序

建议 Codex 按以下顺序推进正式开发：

1. 角色创建链
2. 换装链
3. 战斗准备链
4. 掉落解析链
5. 奖励发放链
6. 入包链
7. 战斗结算链
8. 角色 / 背包 / 穿戴接口
9. 战斗准备 / 战斗结算接口
10. 后台核心配置页与查询页

这样推进的好处是：

- 先把正式状态基础打稳
- 再把正式产出链打通
- 最后再做对外接口与后台联调

## 15. 当前阶段结论

1. Codex 在本项目中的定位是“严格依据文档执行开发任务的主开发代理”
2. Codex 协作必须以“先读文档、先给计划、再改代码、再自测、再汇报”为基本流程
3. Codex 必须严格服从当前阶段边界，不得自行扩展未定稿系统
4. Codex 必须严格遵守五层服务分层、模板层 / 实例层分离、掉落链 / 奖励链分离等核心规则
5. Codex 输出必须可复审、可验收、可追踪
6. 后续所有任务 Prompt，都应建立在本文档之上进一步具体化

本文档可直接作为项目正式《Codex 主开发协作清单》文档使用，并作为后续 `Codex 任务下发 Prompt 模板.md`、`Laravel 代码目录与命名规范.md` 与实际 Codex 开发任务的统一协作基线。
