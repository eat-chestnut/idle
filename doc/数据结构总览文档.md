

# 《山海巡厄录》数据结构总览文档

## 1. 文档目的

本文档用于统一《山海巡厄录》项目当前阶段的数据结构分层、核心表清单、表之间的关系、字段归属原则、系统职责边界与后续扩展方向。

本文档作为以下内容的统一基线：

- 职业系统设计文档
- 角色系统设计文档
- 属性体系设计文档
- 物品系统设计文档
- 装备系统设计文档
- 背包系统设计文档
- 掉落系统设计文档
- 奖励组系统设计文档
- 后台配置说明

后续所有表设计、字段补充、表关系扩展与迁移文件设计，均应以本文档定义的结构边界为准。

## 2. 数据结构总原则

当前阶段数据结构采用以下原则：

1. 先静态配置，后玩家状态
2. 先主表，后扩展表
3. 先统一对象入口，后系统专属扩展
4. 先状态承载，后计算输出
5. 不在表中重复存可推导结果

### 2.1 主表 + 扩展表原则
对于同时具备“统一共性”与“系统专属字段”的对象，采用：

**主表 + 扩展表**

例如：
- 物品主表 + 装备扩展表
- 掉落组主表 + 掉落组明细表
- 奖励组主表 + 奖励组明细表

### 2.2 静态配置与状态分离原则
静态定义数据与玩家状态数据必须分离。

例如：
- 职业表负责职业定义
- 角色表负责玩家成长状态
- 物品表负责物品定义
- 背包表负责玩家当前持有状态

### 2.3 计算结果不固化原则
最终基础属性、最终战斗属性、最终面板结果等可通过规则计算得到的结果，不应直接固化在核心主表中。

## 3. 数据层分层结构

当前阶段数据结构按职责分为以下四层：

1. 静态配置层
2. 玩家状态层
3. 产出配置层
4. 关系与承接层

### 3.1 静态配置层
负责定义系统中的正式对象本体。

当前包括：
- 职业表
- 物品主表
- 装备扩展表

### 3.2 玩家状态层
负责承载玩家当前成长与持有状态。

当前包括：
- 角色主表
- 角色穿戴槽表
- 背包记录表

### 3.3 产出配置层
负责定义随机掉落与固定奖励。

当前包括：
- 掉落组主表
- 掉落组明细表
- 奖励组主表
- 奖励组明细表

### 3.4 关系与承接层
当前阶段由外键与引用字段直接表达，不单独拆复杂中间层。

例如：
- `class_id`
- `item_id`
- `character_id`
- `drop_group_id`
- `reward_group_id`

- 定义职业身份

- 定义职业基础四维初值
- 定义职业修正系数

- 玩家当前状态
- 最终战斗属性结果
- 技能列表
- 装备状态

核心字段包括：
- `class_id`
- `class_name`
- `class_desc`
- `sort_order`
- `icon`
- `avatar`
- `is_enabled`
- `base_strength`
- `base_mana`

## 6. 核心表关系总览

### 6.1 职业与角色关系
- `characters.class_id` → `classes.class_id`

说明：
- 一个角色绑定一个职业
- 一个职业可被多个角色引用

### 6.2 物品与装备关系
- `equipments.item_id` → `items.item_id`

说明：
- 装备必须先存在于物品主表
- 并通过 `item_id` 扩展装备专属字段

### 6.3 角色与穿戴槽关系
- `character_equipment_slots.character_id` → `characters.character_id`

说明：
- 一个角色拥有一组固定槽位
- 一个槽位当前最多穿一件装备

### 6.4 穿戴槽与装备关系
- `character_equipment_slots.equipped_item_id` → `items.item_id`

说明：
- 当前阶段穿戴槽记录当前穿着的装备物品 ID
- 若为装备类，可进一步联动 `equipments`

### 6.5 背包与物品关系
- `inventory_items.item_id` → `items.item_id`

说明：
- 背包统一按 `item_id` 记录持有对象

### 6.6 掉落组与物品关系
- `drop_group_items.item_id` → `items.item_id`

说明：
- 掉落对象统一引用物品主表

### 6.7 奖励组与物品关系
- `reward_group_items.item_id` → `items.item_id`

说明：
- 奖励对象统一引用物品主表

## 8. 关键字段归属原则

### 8.1 共性字段归主表
名称、图标、描述、稀有度、启用状态、排序等共性字段，应归对象主表。

例如：
- `items`
- `classes`

### 8.2 专属字段归扩展表
装备专属行为字段、掉落权重字段、奖励数量字段等系统专属字段，应归扩展表或明细表。

### 8.3 玩家状态字段归状态表
当前持有数量、是否穿戴、是否锁定、等级经验、加点结果等状态字段，必须归玩家状态层。

### 8.4 计算结果不归核心主表
最终基础属性、最终战斗属性、最终面板值，不应直接固化在：
- `classes`
- `characters`
- `items`
- `equipments`

这类核心主表中。

## 10. 后续扩展原则

### 10.1 扩展不破坏主结构原则
后续增加新系统时，应优先沿用：
- 主表 + 扩展表
- 静态定义 + 玩家状态分离
- 共性字段上收
- 结果字段不固化

### 10.2 新系统优先引用现有主键
例如：
- 新成长系统优先引用 `character_id`
- 新掉落扩展优先引用 `item_id`
- 新奖励扩展优先引用 `reward_group_id`

### 10.3 避免平行重复表原则
不得在后续开发中新增与现有主表职责重叠的平行表。

例如：
- 不再新增第二套物品主表
- 不再新增第二套职业定义表
- 不再新增第二套背包状态表

## 11. 当前阶段结论

当前阶段《数据结构总览文档》已确认以下内容：

1. 当前数据结构按静态配置层、玩家状态层、产出配置层分层
2. 当前核心表清单已固定为 10 张核心表
3. 职业、物品、装备、角色、穿戴槽、背包、掉落组、奖励组的职责已明确分离
4. 当前阶段统一采用“主表 + 扩展表”结构思路
5. 当前阶段统一采用“静态定义与玩家状态分离”的结构思路
6. 当前阶段不固化最终属性结果与最终面板结果
7. 后续新增系统必须在不破坏当前主结构前提下扩展
