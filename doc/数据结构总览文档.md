

# 《山海巡厄录》数据结构总览文档

## 1. 文档目的

本文档用于统一《山海巡厄录》项目当前阶段的数据结构分层、核心表清单、表之间的关系、字段归属原则、系统职责边界与后续扩展方向。

本文档作为以下内容的统一基线：

- 职业系统设计文档
- 角色系统设计文档
- 属性体系设计文档
- 物品系统设计文档
- 装备系统设计文档
- 背包系统设计文档
- 掉落系统设计文档
- 奖励组系统设计文档
- 后台配置说明
- 接口设计文档
- 方法级开发设计文档

后续所有表设计、字段补充、表关系扩展与迁移文件设计，均应以本文档定义的结构边界为准。

---

## 2. 数据结构总原则

当前阶段数据结构采用以下原则：

1. 先静态配置，后玩家状态
2. 先主表，后扩展表
3. 先统一对象入口，后系统专属扩展
4. 先状态承载，后计算输出
5. 不在表中重复存可推导结果

### 2.1 主表 + 扩展表原则
对于同时具备“统一共性”与“系统专属字段”的对象，采用：

**主表 + 扩展表**

例如：
- 物品主表 + 装备扩展表
- 掉落组主表 + 掉落组明细表
- 奖励组主表 + 奖励组明细表

### 2.2 静态配置与状态分离原则
静态定义数据与玩家状态数据必须分离。

例如：
- 职业表负责职业定义
- 角色表负责玩家成长状态
- 物品表负责物品定义
- 背包表负责玩家当前持有状态

### 2.3 计算结果不固化原则
最终基础属性、最终战斗属性、最终面板结果等可通过规则计算得到的结果，不应直接固化在核心主表中。

## 3. 数据层分层结构

当前阶段数据结构按职责分为以下四层：

1. 静态配置层
2. 玩家状态层
3. 产出配置层
4. 关系与承接层

### 3.1 静态配置层
负责定义系统中的正式对象本体。

当前包括：
- 职业表
- 物品主表
- 装备扩展表

### 3.2 玩家状态层
负责承载玩家当前成长与持有状态。

当前包括：
- 角色主表
- 角色穿戴槽表
- 可堆叠物持有表
- 装备实例表

### 3.3 产出配置层
负责定义随机掉落与固定奖励。

当前包括：
- 掉落组主表
- 掉落组明细表
- 掉落来源绑定表
- 奖励组主表
- 奖励组明细表
- 奖励来源绑定表

### 3.4 关系与承接层
负责表达发奖幂等、一次性领取与正式发放记录。

当前包括：
- 奖励发放记录表

## 6. 核心表关系总览

### 6.1 职业与角色关系
- `characters.class_id` → `classes.class_id`

说明：
- 一个角色绑定一个职业
- 一个职业可被多个角色引用

### 6.2 物品与装备关系
- `equipments.item_id` → `items.item_id`

说明：
- 装备必须先存在于物品主表
- 并通过 `item_id` 扩展装备专属字段

### 6.3 角色与穿戴槽关系
- `character_equipment_slots.character_id` → `characters.character_id`

说明：
- 一个角色拥有一组固定槽位
- 一个槽位当前最多穿一件装备

### 6.4 穿戴槽与装备实例关系
- `character_equipment_slots.equipped_instance_id` → `inventory_equipment_instances.equipment_instance_id`

说明：
- 穿戴槽只记录当前穿着的装备实例 ID
- 通过装备实例再反查装备模板与装备扩展信息

### 6.5 背包可堆叠物与物品关系
- `inventory_stack_items.item_id` → `items.item_id`

说明：
- 可堆叠物统一按 `item_id` 记录持有对象

### 6.6 装备实例与物品关系
- `inventory_equipment_instances.item_id` → `items.item_id`

说明：
- 每个装备实例都通过 `item_id` 回指装备模板

### 6.7 掉落组与物品关系
- `drop_group_items.item_id` → `items.item_id`

说明：
- 掉落对象统一引用物品主表

### 6.8 掉落来源绑定与掉落组关系
- `drop_group_bindings.drop_group_id` → `drop_groups.drop_group_id`

说明：
- 具体来源对象通过绑定表引用掉落组

### 6.9 奖励组与物品关系
- `reward_group_items.item_id` → `items.item_id`

说明：
- 奖励对象统一引用物品主表

### 6.10 奖励来源绑定与奖励组关系
- `reward_group_bindings.reward_group_id` → `reward_groups.reward_group_id`

说明：
- 具体奖励来源对象通过绑定表引用奖励组

### 6.11 奖励发放记录与奖励组关系
- `user_reward_grants.reward_group_id` → `reward_groups.reward_group_id`

说明：
- 每次正式发奖都落奖励发放记录
- 首通场景通过来源维度唯一约束防重

## 8. 关键字段归属原则

### 8.1 共性字段归主表
名称、图标、描述、稀有度、启用状态、排序等共性字段，应归对象主表。

例如：
- `items`
- `classes`

### 8.2 专属字段归扩展表
装备专属行为字段、掉落权重字段、奖励数量字段、来源绑定字段等系统专属字段，应归扩展表、明细表或绑定表。

### 8.3 玩家状态字段归状态表
当前持有数量、装备实例、是否锁定、等级经验、加点结果等状态字段，必须归玩家状态层。

### 8.4 发放幂等字段归承接表
`idempotency_key`、发奖状态、一次性领取记录等承接字段，必须归奖励发放记录表。

### 8.5 发奖事务规则
奖励发放记录写入与背包承接写入，必须归同一事务边界。

规则如下：

- 只有 `grant_status = success` 时，才允许同时落成功发奖记录与背包结果
- 任一成功链路写入失败必须整体回滚
- 发放失败时应落 `grant_status = failed` 记录
- `grant_status = failed` 记录允许后台重试补发
- 不允许通过异步补偿形成第一版正式主链

### 8.6 实例扩展字段预留规则
装备实例表应预留 `extra_attributes` 这类扩展口子，用于承接未稳定的实例级动态字段。

规则如下：

- 先用扩展 JSON 承接低频、易变化字段
- 再按查询频率决定是否拆显式列
- 避免一开始就过度拆分实例扩展表

### 8.7 计算结果不归核心主表
最终基础属性、最终战斗属性、最终面板值，不应直接固化在：
- `classes`
- `characters`
- `items`
- `equipments`

这类核心主表中。

## 10. 后续扩展原则

### 10.1 扩展不破坏主结构原则
后续增加新系统时，应优先沿用：
- 主表 + 扩展表
- 静态定义 + 玩家状态分离
- 共性字段上收
- 结果字段不固化

### 10.2 新系统优先引用现有主键
例如：
- 新成长系统优先引用 `character_id`
- 新掉落实例扩展优先引用 `equipment_instance_id`
- 新奖励扩展优先引用 `reward_group_id`

### 10.3 避免平行重复表原则
不得在后续开发中新增与现有主表职责重叠的平行表。

例如：
- 不再新增第二套物品主表
- 不再新增第二套职业定义表
- 不再新增第二套背包状态表

## 11. 当前阶段结论

当前阶段《数据结构总览文档》已确认以下内容：

1. 当前数据结构按静态配置层、玩家状态层、产出配置层、关系与承接层分层
2. 当前核心表清单已固定为 14 张核心表
3. 装备模板与装备实例已拆分
4. 掉落与奖励来源必须通过绑定表关联具体来源对象
5. 奖励发放必须通过 `user_reward_grants` 做幂等与防重
6. 当前阶段统一采用“主表 + 扩展表”结构思路
7. 当前阶段统一采用“静态定义与玩家状态分离”的结构思路
8. 当前阶段不固化最终属性结果与最终面板结果
9. 后续新增系统必须在不破坏当前主结构前提下扩展

本文档可直接作为项目正式结构文档使用，并作为后续迁移文件设计、Seeder 设计、后台配置设计、接口设计与方法级开发设计的数据结构总基线。
