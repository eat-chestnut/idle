

# 《山海巡厄录》项目总览

## 1. 文档说明

本文档用于将当前阶段已经整理完成的项目开发结论、系统设计结论、数据库设计结论、开发思路、方法级设计与接口设计，合并收敛为一份统一总览文档。

本文档定位为：

- 项目当前阶段的统一总入口文档
- 后续开发、联调、验收、补文档时的总索引文档
- Codex / 开发人员 / 后台开发 / 接口开发的统一参考基线

本文档不替代各专项文档的细节内容，但会把当前已经定稿的核心思路统一整理到一个地方，避免反复在多个文档之间来回切换。

## 2. 项目定位

《山海巡厄录》是一款以中国神话幻想氛围为核心、面向移动端的竖屏 2D 刷图成长型游戏。

当前项目的核心体验围绕三条主线展开：

1. **战斗推进主线**  
   玩家通过进入主线关卡、击败怪物、推进不同难度，不断解锁新的战斗目标。

2. **装备成长主线**  
   玩家通过获得装备、穿戴装备、持续提升战斗能力，形成长期成长体验。

3. **产出 / 回流主线**
   玩家通过掉落、奖励、背包、穿戴与角色成长承接，逐步形成再开一局的闭环。

当前项目在客户端运行模式上已经正式调整为：

- **单机为主，弱联网为辅**
- 启动时只做一次后台检查
- 进入游戏后，主循环逐步以本地 runtime state 为真相
- 当前后台主要负责版本检查兼容与存档上传 / 下载
- 排行榜、交易所等未来联网能力后续再扩

当前阶段并不是继续做方向探索，而是已经进入：

**开发执行收敛阶段**

也就是：

- 先把第一阶段主链做通
- 先把结构做正确
- 先把正式状态与正式链路打通
- 再逐步扩展后续系统

## 3. 当前阶段总目标

当前阶段的真正目标不是“先把页面堆出来”，而是：

**做出一个结构正确、链路闭环、后续可以持续扩展的第一阶段最小可运行版本。**

当前阶段必须优先完成：

1. 职业、物品、装备、怪物、主线、掉落、奖励可后台配置
2. 角色可以正式创建
3. 角色可读取正式状态
4. 装备可实例化
5. 装备可穿戴 / 卸下
6. 战斗准备可形成正式准备结果
7. 战斗结算可生成正式掉落与奖励结果
8. 正式奖励可防重复发放
9. 正式产出可通过统一入包链承接
10. 角色战斗属性可随加点与换装重算

同时，客户端当前阶段必须同步完成：

11. 启动检查边界收缩为单次联网
12. 游戏运行期的角色、主线、战斗、结算、背包、穿戴、成长结果逐步切换为本地真相
13. 掉落到成长回流的页面主链按单机刷图成长体验收口

当前 backend 已稳定收口的强相关基线包括：

- 第一阶段 backend 主链
- battle context 持久化
- failed reward_grant 沉淀与补发承接
- 最小后台配置页与查询页
- 后台运维工具
- workflow lock 统一封装
- phase-one OpenAPI 契约产物
- 玩家主链闭环 smoke test
- 环境诊断能力

当前 client 已开始收口的强相关基线包括：

- 角色页
- 主线页
- 出战页
- Battle 页
- Settle 页
- 背包页
- 穿戴页
- 本地 runtime / local game state 迁移骨架
- 启动检查兼容层

## 4. 当前阶段不做的内容

以下系统当前阶段只保留结构兼容，不进入正式实现：

- 强化
- 洗练
- 宝石
- 套装
- 经卷
- world level
- 商店系统
- 活动运营系统
- 复杂 BI 后台
- 复杂战斗表现与高级特效演出

原则是：

- 先打通正式主链
- 不提前深做未来系统
- 不为了第一阶段方便而破坏长期结构

## 5. 当前阶段核心设计原则

### 5.1 先打通主链，再做表现层
第一阶段优先级高于页面表现与运营功能。

### 5.1.1 客户端运行期以本地真相为主
客户端在运行期不再默认以实时 API 为主循环真相。
当前应优先收口：

- 当前角色
- 当前目标关卡
- 当前 battle context
- 当前结算结果
- 当前背包
- 当前穿戴
- 当前成长承接

实时 API 当前不再是这些页面的默认反复请求来源。

### 5.2 模板层与实例层必须分离
必须始终区分：

- 物品模板 vs 玩家持有状态
- 装备模板 vs 装备实例
- 奖励模板 vs 发奖记录
- 掉落模板 vs 正式掉落结果

### 5.3 掉落链与奖励链必须分离
当前项目明确：

- 掉落链：怪物随机掉落
- 奖励链：副本不同难度首通固定奖励

二者只能在结算展示层汇总，不能在配置与执行层混用。

### 5.4 正式写入必须走统一服务入口
以下链路不允许散落在 Controller、页面逻辑或后台脚本中：

- 角色创建
- 换装
- 战斗准备
- 战斗结算
- 掉落解析
- 奖励发放
- 入包

### 5.5 数据库约束与服务层约束必须双重存在
高风险链路不能只靠代码判断，必须尽量同时具备：

- 数据库唯一约束 / 外键 / 索引
- 服务层统一入口 / 事务边界 / 正式校验

### 5.6 联调工具不等于正式运行模式
当前 merge gate、online smoke、diagnose 与 readiness 只用于开发保障。
它们可以证明：

- 现有 backend 可联调
- 现有契约未明显漂移
- 最小链路仍可跑通

但它们不代表：

- 游戏运行期必须持续联网
- 页面应继续围绕实时 API 驱动
- 当前项目仍以“在线联调客户端”作为正式产品方向

## 6. 第一阶段六条主链

第一阶段正式以以下六条主链作为开发与验收基线：

### 6.1 配置可读主链
要求：

- 职业、物品、装备模板、主线、怪物、掉落、奖励等正式配置可通过后台维护
- 服务层可稳定读取正式配置

### 6.2 状态可写主链
要求：

- 角色、穿戴槽、可堆叠背包、装备实例都可被正式创建与更新

### 6.3 掉落可结算主链
要求：

- 怪物击败后能通过正式掉落配置生成正式掉落结果

### 6.4 奖励可防重主链
要求：

- 副本不同难度首通固定奖励可正式发放，且不会重复发放

### 6.5 装备可实例化主链
要求：

- 装备对象进入玩家持有状态时，必须先实例化

### 6.6 角色可重算主链
要求：

- 角色战斗属性能基于职业、加点、已穿戴装备进行正式计算

这六条主链共同构成第一阶段最小闭环。

## 8. 后端服务分层定稿

后端服务以 **五层设计** 为准，不再使用“四层”口径。

五层分别为：

### 8.1 Config 层
负责读取静态配置。

例如：

- `ClassConfigService`
- `ItemConfigService`
- `EquipmentTemplateConfigService`
- `DropConfigService`
- `RewardConfigService`

### 8.2 Query 层
负责读取玩家状态与业务查询结果。

例如：

- `CharacterQueryService`
- `InventoryStackQueryService`
- `EquipmentInstanceQueryService`
- `RewardGrantQueryService`

### 8.3 Domain 层
负责单点业务动作与领域规则执行。

例如：

- `CharacterCreateService`
- `EquipmentWearService`
- `CharacterStatService`
- `DropResolverService`
- `RewardGrantRecordService`

### 8.4 Workflow 层
负责组织完整流程与事务边界。

例如：

- `CharacterCreateWorkflow`
- `BattlePrepareWorkflow`
- `BattleSettlementWorkflow`
- `RewardGrantWorkflow`
- `EquipmentChangeWorkflow`

### 8.5 Admin 层
负责后台校验、引用检查、补发与修复类逻辑。

例如：

- `AdminConfigValidationService`
- `AdminReferenceCheckService`
- `AdminRewardRetryService`
- `AdminDataRepairService`

## 9. 核心业务链总览

当前阶段最关键的正式业务链共七条：

1. 角色创建链路
2. 换装链路
3. 战斗准备链路
4. 战斗结算链路
5. 掉落解析链路
6. 奖励发放链路
7. 入包链路

### 9.1 角色创建链路
目标：

- 创建正式角色
- 初始化 12 个固定穿戴槽

### 9.2 换装链路
目标：

- 正式穿戴 / 卸下装备实例
- 正确处理主副武器联动

### 9.3 战斗准备链路
目标：

- 基于正式配置与正式状态生成战斗准备结果

### 9.4 战斗结算链路
目标：

- 基于正式结算上下文，组织掉落链、奖励链、入包链，输出正式结算结果

### 9.5 掉落解析链路
目标：

- 根据正式怪物来源与正式掉落组生成标准掉落结果

### 9.6 奖励发放链路
目标：

- 对首通奖励进行正式发放、幂等控制、业务防重与正式记录写入

### 9.7 入包链路
目标：

- 将正式产出结果安全写入玩家正式持有状态
- 可堆叠物进入 `inventory_stack_items`
- 装备对象进入 `inventory_equipment_instances`

## 11. 当前阶段关键数据库约束

### 11.1 穿戴槽唯一约束
- `UNIQUE(character_id, slot_key)`

保证一个角色一个槽位只有一条正式记录。

### 11.2 实例唯一占用约束
- `UNIQUE(equipped_instance_id)`

保证同一装备实例不会被多个槽位同时占用。

### 11.3 可堆叠物唯一约束
- `UNIQUE(user_id, item_id)`

保证同一玩家同一可堆叠物只有一条正式记录。

### 11.4 掉落来源唯一绑定
- `UNIQUE(source_type, source_id)` on `drop_group_bindings`

保证一来源一掉落组。

### 11.5 奖励来源唯一绑定
- `UNIQUE(source_type, source_id)` on `reward_group_bindings`

保证一来源一奖励组。

### 11.6 奖励幂等约束
- `UNIQUE(user_id, idempotency_key)` on `user_reward_grants`

保证请求幂等。

### 11.7 奖励业务防重约束
- `UNIQUE(user_id, source_type, source_id, reward_group_id)` on `user_reward_grants`

保证同一业务来源不会重复成功发奖。

## 13. 方法级设计总览

当前已经形成的第一阶段核心方法级设计文档包括：

1. `角色创建方法设计.md`
2. `换装方法设计.md`
3. `战斗准备方法设计.md`
4. `战斗结算方法设计.md`
5. `掉落解析方法设计.md`
6. `奖励发放方法设计.md`
7. `入包方法设计.md`

### 13.1 角色创建方法设计
重点：

- `createCharacter()` 作为唯一正式 Workflow 入口
- 角色主表与 12 槽位必须同事务初始化

### 13.2 换装方法设计
重点：

- `equip()` / `unequip()` 作为正式入口
- 主副武器联动必须统一处理

### 13.3 战斗准备方法设计
重点：

- `prepareBattle()` 作为正式入口
- 正式战斗属性必须来自 `CharacterStatService`

### 13.4 战斗结算方法设计
重点：

- `settleBattle()` 作为正式入口
- 掉落链、奖励链、入包链分链执行后统一汇总

### 13.5 掉落解析方法设计
重点：

- `resolve()` 作为唯一正式入口
- `weighted_single` / `weighted_repeat` 只有一套正式实现

### 13.6 奖励发放方法设计
重点：

- `grant()` 作为唯一正式入口
- 必须同时具备幂等控制与业务防重

### 13.7 入包方法设计
重点：

- `write()` 作为唯一正式入口
- 可堆叠物与装备对象必须分流
- 装备对象必须实例化后才算正式入包

## 15. 高风险点总览

当前阶段最重要的高风险点包括：

1. 角色槽位重复初始化
2. 同一装备实例重复穿戴
3. 可堆叠背包重复行
4. 掉落来源绑定错乱
5. 奖励来源绑定错乱
6. 首通奖励重复发放
7. 发奖成功但未正确入包
8. 模板层与实例层混用
9. 角色属性来源不完整或口径漂移

### 15.1 高风险控制原则
必须优先做到：

- 数据库约束兜底
- 服务层统一入口兜底
- 关键链路事务边界明确
- 结果可追溯

### 15.2 当前阶段重点约束
- 角色创建必须同事务初始化槽位
- 换装必须同事务完成联动清理
- 发奖必须同事务完成主记录、明细、入包、状态更新
- 入包必须同事务完成可堆叠物写入与装备实例创建
