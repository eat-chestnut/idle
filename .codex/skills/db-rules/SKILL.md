# 目标
本技能用于约束数据库设计，防止“功能勉强可用，但结构不可维护”。

数据库设计必须优先服务以下原则：
- 配置层与实例层分离
- 正式链路可追踪
- 幂等可验证
- 约束优先于代码侥幸
- 第一阶段只做必要表，不扩未来系统

# 核心规则
## 1. 模板层 / 实例层必须分离
- 装备模板 ≠ 装备实例
- 掉落规则 ≠ 玩家获得记录
- 奖励模板 ≠ 奖励发放记录

## 2. 正式业务链必须有记录表
优先保证有正式记录：
- 奖励发放
- 奖励明细
- 战斗上下文
- 装备实例生成
- 关键状态迁移

## 3. 唯一性尽量交给数据库约束
例如：
- 每角色每槽位唯一
- 一件装备实例只能被一个槽位占用
- 奖励发放业务键唯一
- battle context 外部唯一键唯一

## 4. 幂等必须可落库校验
对于发奖、结算、补发等操作：
- 必须设计业务唯一键
- 必须考虑重复请求
- 必须能从表结构上校验是否已执行

## 5. 第一阶段禁止为未来系统提前铺太多表
不要为了“以后可能要用”而引入：
- 强化
- 洗练
- 宝石
- 套装
- 技能树
- 商店
- 活动
- 邮件系统

# Migration 编写要求
每个 migration 都应满足：
- 表名清晰且贴合业务
- 主键、外键、唯一键明确
- 时间字段完整
- 必要的状态字段可读
- 不做与当前需求无关的扩张

# 自检清单
- 是否把模板与实例混在一起了？
- 是否缺少关键唯一约束？
- 是否缺少可追踪记录表？
- 是否为了未来系统提前扩表？
- 是否让应用层逻辑替代了数据库约束？
- 是否会导致奖励 / 结算重复执行无法判断？
