description: 当用户提出新功能、扩系统、加玩法、提前设计未来能力时使用。用于严格约束 idle 项目当前阶段的开发边界，避免 Codex 过度发挥。
---

# 目标
你的职责不是无条件实现所有功能，而是保护项目节奏，优先完成第一阶段主链。

# 当前阶段允许的内容
当前阶段优先支持：
- 角色创建
- 装备实例化
- 装备穿戴 / 卸下
- 属性汇总 / 重算
- 战斗准备
- 战斗结算
- 掉落解析
- 奖励发放
- 正式入包
- 必要后台配置页
- 必要运维 / 修复工具
- 与上述主链直接相关的表结构、接口、资源输出

# 当前阶段默认不做的内容
除非文档明确纳入，否则以下内容默认不实现：
- 强化系统
- 洗练系统
- 宝石系统
- 套装系统
- 技能树 / 技能成长
- 商店系统
- 活动系统
- 邮件系统
- 公会 / 社交
- 排行榜
- 世界等级
- 复杂任务系统
- 多货币复杂经济系统

# 判断规则
1. 是否直接服务第一阶段主链？
2. 是否只是“未来可能会用”？
3. 是否会引入新的独立系统？

# 正确的收敛方式
当需求偏大时，优先收敛为：
- 只补当前主链必需字段
- 只做最小接口
- 只保留扩展点，不做完整玩法
- 只做后台配置，不做前台系统
- 只做第一步，不做完整生态

# 自检清单
- 这个需求是否直接服务当前主链？
- 是否引入了新的独立系统？
- 是否只是为了未来可能性而过度设计？
- 是否存在更小的替代实现？
- 是否会显著增加代码与测试面？
