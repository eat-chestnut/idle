# 目标
对该项目，默认采用“小步闭环交付”：
- 先理解文档
- 再核对真实代码
- 再确定边界
- 再设计改动
- 再实现代码
- 再做最小验证
- 最后汇报影响面和 git 结果

如果任务是 client / Godot / 页面运行时方向，额外要求：
- 先核对是否存在把客户端误导成“在线联调客户端”的文档或 skills
- 先做文档纠偏，再做代码
- 启动联网只收口为一次检查
- 运行期页面优先读本地 runtime state

# 标准开发步骤
## 第 1 步：先读文档与现有代码
开始实现前，先确认：
- `doc` 中相关设计文档
- 现有服务目录结构
- 已有 Workflow / Domain / Query 是否可复用
- 当前 migration / model / route / admin / tests 是否已有近似能力

如果任务是 client：
- 同时确认 `client/README.md`
- 同时确认 `doc/codex/单机运行时与弱联网边界.md`
- 同时确认当前主协调器与页面是否仍强绑定实时 API

## 第 2 步：明确需求属于当前阶段吗
判断本需求是：
- 第一阶段主链范围内
- 第一阶段辅助支撑范围内
- 主链收口 / 修复 / 测试补强 / 后台增强
- 还是未来系统范围

## 第 3 步：先出改动方案，再动代码
改动方案至少包含：
- 当前真实代码与任务差异点
- 修改文件清单
- 新增文件清单
- 每个文件职责
- 数据表是否变化
- API / 后台 / Seeder / 测试 是否变化
- 风险点

## 第 4 步：按增量顺序实现
推荐顺序：
1. 先确认是否真的需要 migration
2. Query / Domain
3. Workflow
4. Controller / Route
5. Resource / 后台页面 / 后台工具
6. Seeder（仅当任务确实需要）
7. 测试 / 最小验证
8. 文档最小回写
9. git commit / push

如果任务是 client，推荐顺序调整为：
1. 文档 / skills 纠偏
2. runtime state 骨架
3. 启动检查层
4. 主协调器切换读取方向
5. 关键页面承接本地状态
6. 最小验证
7. git commit / push

## 第 5 步：做最小闭环验证
至少说明：
- 入口是什么
- 调用链是否闭环
- 数据会写到哪里
- 输出结构是什么
- 哪些边界情况已处理
- 本轮运行了哪些命令

# 输出代码时的固定格式
1. 当前真实代码与任务差异
2. 实现思路
3. 改动文件
4. 代码
5. 验证方式
6. git 提交信息
7. 风险与后续

# 自检清单
- 是否先对齐 doc 和真实代码了？
- 是否说明了改动文件与职责？
- 是否避免在无必要时乱改 migration？
- 是否按 Query / Domain / Workflow / Controller 顺序实现？
- 是否给了最小验证路径？
- 是否一次性扩了太多无关功能？
- 是否复用了已有结构而不是重写？
- 是否输出了分支、commit、push 结果？
- 如果是 client 任务，是否先纠偏文档与 skills 了？
- 如果是 client 任务，是否避免继续强化实时 API 真相？
