# idle 仓库给 Codex 的总规则

你正在开发 `idle` 项目。请先严格遵守这里的规则，再开始写代码。

## 1. 当前项目状态
当前仓库已经不是“空工程初始化阶段”，而是**第一阶段 backend 主链已基本打通、客户端主链正在从在线联调壳迁移到单机运行时**的持续迭代阶段。

当前已存在并应优先复用的能力包括：
- 角色创建
- 装备实例化
- 装备穿戴 / 卸下
- 属性汇总 / 重算
- 战斗准备
- 战斗结算
- 掉落解析
- 奖励发放
- 正式入包
- battle context 状态收口
- 最小后台配置页与查询页
- 后台运维工具（引用检查 / 补发 / 修复）
- 客户端角色 / 主线 / 出战 / Battle / Settle / 背包 / 穿戴页面骨架
- 客户端运行期本地状态迁移入口

默认目标不是“从 0 到 1 搭工程”，而是：
- 在现有真实代码基础上增量迭代
- 收口差异
- 加固事务、幂等、校验、测试、后台支撑
- 不扩未来系统
- 不把客户端继续做成“实时 API 真相客户端”

## 2. 文档优先级
开发时优先参考：
1. `doc/项目总览.md`
2. `/AGENTS.md`
3. `/backend/AGENTS.md`
4. `doc/codex/单机运行时与弱联网边界.md`（当任务涉及 client / 运行时 / 启动检查时）
5. 与当前功能直接相关的专题文档
6. 现有代码中的相邻实现
7. `/code_review.md`
8. `/backend/code_review.md`

如果文档与现有代码冲突：
- 优先判断是否是文档未回写
- 不要擅自扩系统
- 先选择最保守、最接近现有主链的实现
- 明确列出“文档-代码差异清单”后再继续实现

## 3. 架构规则
必须遵守五层职责：
- Controller：只做入口
- Workflow：主链编排、事务、幂等入口
- Domain：规则与计算
- Query：只读
- Model：数据映射

明确禁止：
- 在 Controller 写核心业务逻辑
- 在 Query 中写状态
- 在 Model / Observer 中偷偷执行主链
- 直接跳过 Workflow 在入口拼完整业务
- 因为已有功能能跑就复制一套旁路逻辑

## 4. 当前阶段开发原则
默认开发原则是：
1. 先检查 git 当前状态
2. 先核对当前真实代码是否已经有近似实现
3. 先给出差异点与改动方案
4. 再增量修改现有代码
5. 做最小验证
6. git 提交并 push

如果任务是 client 方向，额外遵守：
- 先纠正文档 / README / skills 中会误导成在线联调客户端的部分
- 启动联网只允许收口为一次检查
- 运行期页面优先读本地 runtime state
- 不为了迁移方便继续强化实时 API 驱动

除非任务明确要求，不要默认：
- 先建 migration
- 先重写 model
- 先重搭框架

优先判断这是哪一类任务：
- 主链增强
- 主链收口
- 后台增强
- 修复 / 校验加固
- 测试补强
- 文档对齐
- 客户端单机化迁移

## 5. 战斗主链规则
战斗相关需求必须遵守：
1. prepare 与 settle 分离
2. battle context 有清晰状态流转
3. 掉落链与奖励链分离
4. 普通掉落统一入包
5. 首通奖励统一走奖励发放链
6. 不允许直接写背包最终表替代正式链路
7. 必须尊重当前仓库已实现的 battle context 持久化、failed reward grant 沉淀与补发承接语义

如果需求涉及结算收口：
- 先判断现有事务语义
- 不要把当前已收口的“成功链原子、失败链可沉淀、补发可承接”语义回退为单纯的大事务整体抹除模型

## 6. 数据结构规则
数据库必须遵守：
- 模板层 / 实例层分离
- 关键唯一性用数据库约束表达
- 发奖、结算、补发必须具备幂等落库能力
- 关键业务要可追踪
- 不为未来系统提前扩大量表结构

默认不要为“可能将来会用”提前扩表。
如果任务不涉及 schema，优先不要改 migration。

## 7. 当前阶段禁止扩张的系统
除非文档明确纳入，否则不要主动实现：
- 强化
- 洗练
- 宝石
- 套装
- 技能树
- 商店
- 活动
- 邮件
- 社交 / 公会
- 排行榜
- 交易所
- 世界等级
- 大型任务系统

如果用户提到这些，优先收敛为最小替代方案，而不是完整实现。

如果任务是客户端联网边界调整，也禁止：
- 提前做复杂在线状态系统
- 提前做重服务端驱动 UI
- 把 merge gate / online smoke 当成正式运行模式

## 8. 默认开发流程
实现任何需求时，按以下顺序输出：
1. 当前真实代码现状与任务差异
2. 实现思路
3. 改动文件清单
4. 每个文件职责
5. 代码
6. 最小验证方式
7. git 提交信息
8. 风险与后续建议

推荐开发顺序（增量迭代优先）：
1. 审查现有代码与文档
2. 判断是否需要 schema 变化（无必要则不要动 migration）
3. Query / Domain
4. Workflow
5. Controller / Route
6. Resource / Admin / Tool
7. Test / Validation
8. 文档最小回写
9. git commit / push

## 9. 默认启用的 Skills
当仓库存在 `.codex/skills` 时，优先应用：
- `architecture-guard`
- `dev-flow`
- `scope-guard`

以下场景额外启用：
- 战斗相关：`battle-flow`
- migration / 表结构 / 数据修复：`db-rules`

如果任务明确落在 client 且不涉及 backend 业务实现：
- `architecture-guard` 只作为“避免误改后端”的边界约束使用
- 不要把五层 backend 规则硬套到 Godot 页面脚本上

## 10. 提交前自检
提交前必须检查：
- 是否对齐 `doc`
- 是否落在正确业务域
- 是否遵守五层结构
- 是否避免了越界扩系统
- 是否说明了事务边界与幂等点
- 是否给出最小验证路径
- 是否输出了 git 分支、commit message、commit hash、push 结果

如果以上任一项不明确，先收敛再提交。
