

# 《山海巡厄录》Code Review 清单

## 1. 文档目的

本文档用于统一《山海巡厄录》项目在第一阶段开发过程中的 Code Review 标准，作为项目根目录 `AGENTS.md` 的配套审查清单使用。

本文档的目标不是重复系统设计，而是让每次评审都能快速检查：

- 是否符合项目文档
- 是否符合五层服务分层
- 是否存在越权改动
- 是否存在事务、幂等、唯一约束风险
- 是否存在模板层 / 实例层混用
- 是否存在掉落链 / 奖励链混用
- 是否存在接口口径漂移
- 是否存在过度设计或无关重构

本文档定位为：

- 第一阶段统一 Code Review 清单
- Codex 自查与人工评审共用清单
- 合并前必过的基础审查规则文档

---

## 2. Review 总原则

### 2.1 先看边界，再看实现

评审时不要先陷入代码细节，先确认：

- 这次改动是否在任务边界内
- 是否动了不该动的模块
- 是否引入了未定稿功能
- 是否违背现有文档

如果边界已经错了，后面的实现越漂亮越危险。

### 2.2 先看正式主链，再看局部优雅

当前项目优先级是：

- 主链正确
- 状态正确
- 边界正确
- 事务正确
- 接口稳定

而不是：

- 抽象看起来更高级
- 类拆得更花哨
- 代码风格更“优雅”

### 2.3 先看风险点，再看功能点

第一阶段评审时，应优先检查：

- 是否可能留下半成状态
- 是否可能重复发奖
- 是否可能重复入包
- 是否可能重复初始化槽位
- 是否可能同实例多槽位占用

这些问题优先级高于一般功能细节。

---

## 3. 文档一致性检查

每次评审至少先检查以下问题：

### 3.1 是否符合《项目总览》
- 是否仍然围绕第一阶段主链开发
- 是否没有擅自扩展未来系统

### 3.2 是否符合《Codex 主开发协作清单》
- 是否存在任务越权
- 是否存在未授权模块改动

### 3.3 是否符合《Laravel 代码目录与命名规范》
- 目录是否正确
- 类名、方法名、文件名是否统一

### 3.4 是否符合对应方法级设计文档
- Query / Domain / Workflow 是否按文档拆分
- 方法职责是否清晰
- 事务边界是否符合文档

### 3.5 是否符合接口字段级设计 / 接口示例文档
- 请求字段是否一致
- 响应字段是否一致
- 错误结构是否一致

### 3.6 是否符合错误码总表 / 枚举总表
- 是否用了正式错误码
- 是否用了正式枚举值
- 是否出现 magic string 漂移

---

## 4. 任务边界检查

### 4.1 是否只实现了本次任务要求
- 是否只改了本次任务涉及的链路
- 是否没有顺手补做无关模块

### 4.2 是否存在无关重构
- 是否修改了与本任务无关的目录
- 是否把局部任务扩成了大规模重构

### 4.3 是否擅自补了未来系统
重点检查是否偷偷加入了：

- 强化
- 洗练
- 宝石
- 套装
- 经卷
- world level
- 商店
- 活动系统

### 4.4 是否新增了未定稿字段 / 表 / 接口
- 是否擅自新增表结构
- 是否擅自新增正式接口
- 是否擅自扩展正式响应字段

---

## 5. 五层服务分层检查

项目正式以五层设计为准：

1. Config
2. Query
3. Domain
4. Workflow
5. Admin

### 5.1 Config 层检查
- 是否只负责读取静态配置
- 是否没有写状态
- 是否没有做事务
- 是否没有承担流程编排

### 5.2 Query 层检查
- 是否只负责读取正式状态
- 是否没有偷偷写状态
- 是否没有承担完整业务流程

### 5.3 Domain 层检查
- 是否承担单点业务动作或规则判断
- 是否没有把整条链全塞进去
- 是否没有变成页面拼装层

### 5.4 Workflow 层检查
- 是否负责完整流程编排
- 是否负责事务边界控制
- 是否没有重写底层所有规则

### 5.5 Admin 层检查
- 是否只负责后台校验 / 引用检查 / 补发 / 修复
- 是否没有复制正式玩家链路逻辑
- 是否没有绕过正式 Workflow 直接改正式状态

---

## 6. Controller / Request / Resource 检查

### 6.1 Controller 检查
- 是否保持轻量
- 是否只做收参、调服务、回响应
- 是否没有写正式业务逻辑
- 是否没有直接操作多个 Model 完成完整流程

### 6.2 Request 检查
- 是否只做基础参数校验
- 是否没有把业务规则塞进 Request

### 6.3 Resource / 响应检查
- 是否符合统一响应外壳
- 是否字段命名统一为 `snake_case`
- 是否没有临时拼出一套新风格响应结构

---

## 7. 模板层 / 实例层检查

这是当前项目最容易出错的地方之一。

### 7.1 物品与持有状态是否分离
- `items` 是否仍然只表示模板
- 是否没有把持有数量写回模板

### 7.2 装备模板与装备实例是否分离
- 是否明确区分 `equipments` 与 `inventory_equipment_instances`
- 换装是否操作实例而不是模板
- 入包时装备是否先实例化

### 7.3 奖励模板与发奖记录是否分离
- 是否区分 `reward_groups` 与 `user_reward_grants`
- 是否没有把正式发奖结果写回奖励模板

---

## 8. 掉落链 / 奖励链 / 入包链检查

### 8.1 掉落链检查
- 掉落链是否只负责解析正式掉落结果
- 是否没有直接写背包
- 是否没有承担发奖职责

### 8.2 奖励链检查
- 奖励链是否只负责正式奖励发放、防重、记录、入包承接
- 是否没有承担怪物随机掉落逻辑

### 8.3 入包链检查
- 入包链是否作为统一正式入口
- 是否正确区分可堆叠物与装备对象
- 装备是否实例化后才算正式入包

### 8.4 是否存在链路混用
重点检查：

- 是否把怪物掉落混进首通奖励逻辑
- 是否把首通奖励伪装成掉落
- 是否在战斗结算里直接跳过正式子链手工写状态

---

## 9. 事务、幂等与唯一约束检查

### 9.1 角色创建链
- 是否角色主表与 12 槽位在同一事务中初始化
- 是否失败时不会留下半成角色状态

### 9.2 换装链
- 是否目标槽位更新与联动清理在同一事务中
- 是否不会出现主武器已更新但副武器未清理
- 是否不会出现同一实例被多个槽位占用

### 9.3 奖励发放链
- 是否正确使用业务防重
- 是否正确使用幂等键
- 是否主记录、明细、入包、状态更新在同一事务中
- 是否不会出现奖励已入包但状态仍 pending

### 9.4 入包链
- 是否可堆叠物写入与实例创建在同一事务中
- 是否不会出现堆叠物成功而装备实例失败却仍返回成功

### 9.5 数据库唯一约束对应实现是否匹配
重点检查：

- `character_equipment_slots` 唯一约束
- `inventory_stack_items` 唯一约束
- 掉落绑定唯一约束
- 奖励绑定唯一约束
- 奖励幂等约束
- 奖励业务防重约束

---

## 10. 接口契约检查

### 10.1 请求字段检查
- 是否符合接口字段级设计
- 是否没有多传 `user_id`
- 是否没有新增未定稿字段

### 10.2 响应字段检查
- 是否符合接口字段级设计与接口示例文档
- 是否没有擅自删字段或改字段名
- 是否没有把中文展示值替代正式枚举值

### 10.3 错误响应检查
- 是否统一使用正式错误码
- 是否失败时 `data = null`
- 是否没有直接把底层异常文本暴露给前端

### 10.4 认证规则检查
- 是否从认证上下文获取当前用户
- 是否没有信任前端传入的 `user_id`
- 是否业务接口都在认证中间件保护下

---

## 11. 枚举与错误码检查

### 11.1 枚举检查
- 是否严格使用《枚举总表》中的正式值
- 是否存在 `mainWeapon` / `main_weapon` 混用
- 是否存在 `success` / `done` 混用

### 11.2 错误码检查
- 是否使用《错误码总表》中的正式错误码
- 是否同一语义用了多个错误码
- 是否临时自造错误码但没补文档

### 11.3 magic string 检查
- 是否在关键链路中散写枚举字符串
- 是否应提取为 Enum 或统一常量却没有提取

---

## 12. Seeder 与最小联调数据检查

如果本次改动涉及 Seeder 或联调数据，重点检查：

### 12.1 是否符合《最小联调种子数据》
- 是否使用固定业务 ID
- 是否覆盖最小联调闭环

### 12.2 是否可重复执行
- 是否使用稳定 upsert / updateOrCreate / firstOrCreate 方案
- 是否不会产生脏重复数据

### 12.3 是否没有引入过量数据
- 是否只做最小联调所需数据
- 是否没有塞入大量正式运营内容

---

## 13. 测试与验证检查

### 13.1 是否有最小验证动作
- 是否运行了相关测试
- 是否至少做了语法检查 / 路由检查 / Seeder 检查 / 功能链验证中的一部分

### 13.2 是否给出自测步骤
- 是否说明如何验证改动
- 是否说明前置条件

### 13.3 验证失败时是否诚实说明
- 是否明确指出无法验证的原因
- 是否没有把“未验证”写成“已完成”

---

## 14. 文档回写检查

如果本次改动改变了正式契约，需检查是否同步更新文档：

### 14.1 需要同步更新文档的典型场景
- 新增或调整正式接口字段
- 新增或调整错误码
- 新增或调整枚举值
- 新增或调整最小联调种子数据 ID
- 变更正式业务边界或正式实现口径

### 14.2 不需要为无意义风格改动频繁改文档
- 不要纯为了措辞微调而大面积 churn 文档
- 只在正式行为、契约、边界变化时回写文档

---

## 15. 常见高风险反例清单

评审时如果看到以下情况，应重点拦截：

1. Controller 里直接写事务和多表逻辑
2. 奖励发放跳过正式 Workflow 直接入包
3. 掉落解析直接写背包
4. 装备模板和装备实例混用
5. 首通奖励重复发放没有业务防重
6. 只做服务层判断，没有数据库唯一约束意识
7. Seeder 重复执行会插出重复脏数据
8. 同一正式枚举写出多套字符串值
9. 响应结构与接口示例文档不一致
10. 任务明明只做一条链，却顺手改了半个项目

---

## 16. Review 结论输出建议

每次 Code Review 建议至少输出以下结论结构：

### 16.1 结论等级
- `通过`
- `有条件通过`
- `不通过`

### 16.2 必改问题
列出会影响：

- 正式状态正确性
- 主链正确性
- 接口契约稳定性
- 数据约束安全性

的问题。

### 16.3 建议优化项
列出：

- 结构可优化
- 命名可优化
- 文档可补充
- 测试可补充

但不影响当前主链正确性的内容。

### 16.4 风险提示
说明：

- 当前尚未验证的风险
- 环境限制导致未验证的内容
- 后续建议继续补的测试或文档

---

## 17. 当前阶段结论

1. 第一阶段 Code Review 必须优先围绕主链正确性、正式状态正确性、事务边界、唯一约束、接口契约稳定性来审查
2. 评审顺序建议为：先看任务边界，再看分层，再看主链风险，再看接口契约，最后看代码细节
3. 模板层 / 实例层分离、掉落链 / 奖励链分离、幂等 / 防重 / 入包事务边界，是当前阶段最关键的 Review 检查点
4. 后续无论是 Codex 自查还是人工评审，都建议以本文档作为统一 Review 清单

本文档可直接作为项目正式 `code_review.md` 使用，并作为 `AGENTS.md` 的配套审查基线文档。
