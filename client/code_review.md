# 《山海巡厄录》Client Code Review 清单

## 1. 文档目的

本文档用于统一《山海巡厄录》项目游戏端 / 客户端阶段的 Code Review 标准，作为 `client/AGENTS.md` 的配套审查清单使用。

本文档重点聚焦：
- 页面流是否正确
- 接口接入是否真实
- UI 状态是否完整
- 客户端是否复制了后端业务逻辑
- 是否存在接口契约漂移
- 是否存在“能展示但不可联调”的假实现

---

## 2. Review 总原则

### 2.1 先看玩家主链是否打通
优先检查：
- 能否进入角色创建
- 能否查看角色状态
- 能否查看背包与穿戴
- 能否进入章节/难度
- 能否进行 battle prepare
- 能否展示 battle settle 结果

### 2.2 先看真实接入，再看表现层细节
如果一个页面只是本地 mock 成功，而没有真实接 API，就不算完成。

### 2.3 不允许客户端复制后端业务真相
客户端可以做显示态转换，但不能重新发明：
- 掉落规则
- 奖励发放规则
- 所有权规则
- battle_context 合法性规则

---

## 3. 页面与入口检查

- 页面是否符合《游戏端页面与入口清单》
- 入口是否清晰
- 跳转是否合理
- 是否存在死路由 / 无入口页面
- 是否存在“从设计上应串联，但实际上是孤页”的情况

---

## 4. 接口接入检查

- 是否符合《游戏端接口接入清单》
- 是否使用真实 backend 接口
- 是否正确处理 bearer token
- 是否存在字段名漂移
- 是否存在把后端错误当成功吞掉的情况

---

## 5. UI 状态检查

每个页面至少检查：
- loading
- success
- empty
- error
- unauthorized
- locked/unlocked
- reward available/claimed

重点拦截：
- 只有成功态，没有失败态
- 空列表直接白屏
- token 过期没有处理
- battle prepare 失败没有反馈
- reward 状态展示与真实后端不一致

---

## 6. 后端契约漂移检查

- OpenAPI、接口示例文档、真实接口是否一致
- 客户端是否还在调用已移除接口
- 是否还依赖旧字段
- 是否显示了后端已经不返回的数据

---

## 7. 客户端状态管理检查

- 是否只保存必要 UI 状态
- 是否没有把后端状态重复建模成第二套业务真相
- 是否没有把“服务端待确认状态”伪装成客户端确定状态

---

## 8. 错误与提示检查

- 是否对正式错误码做了明确映射
- 是否没有把一切错误都显示成同一句
- 是否没有静默失败
- 是否能区分登录失效、资源不存在、业务禁止、网络错误

---

## 9. 联调检查

- 是否能在当前 backend readiness/diagnose/acceptance 基线上联调
- 是否写清楚了启动方式
- 是否说明了所需 token、环境地址、依赖前提

---

## 10. 结论输出建议

每次 Client Review 建议输出：
- 结论：通过 / 有条件通过 / 不通过
- 必改问题
- 建议优化项
- 联调风险
- 未验证项
