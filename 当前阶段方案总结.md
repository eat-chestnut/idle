# 《山海巡厄录》当前阶段方案总结（讨论结论整理）

## 一、当前讨论目标

当前目标不是直接进入全量开发，而是先把游戏的核心玩法框架、后台数据结构、武器与职业模块边界讨论清楚，逐步形成一份足够细的开发方案。  
后续希望能继续细化到：

- 数据结构
- 模块职责
- 核心方法
- 方法逻辑
- 配置边界
- 后台实现范围
- 一期与后续期的分层

本文件只整理目前已经讨论并基本确认的结论，不扩展到尚未确认的新方向。

---

## 二、第一版整体开发策略结论

### 1. 开发顺序
当前更推荐的开发顺序是：

1. 先讨论并确定方案
2. 先完善后台与正式数据模型
3. 再做接口
4. 最后做游戏端与对接

原因：
- 当前玩法和系统边界仍在收敛
- 如果先大规模做游戏端，会产生大量返工
- 后台和数据模型稳定后，前端只需要读取与表现

### 2. 第一版不追求功能多
第一版目标不是做全，而是先把最小闭环跑通。  
目前第一版被理解为：

主线推进 + 战斗 + 掉落 + 背包 + 装备成长 + 回刷再推进

---

## 三、第一版核心玩法结论

### 1. 第一版核心玩法
当前第一版核心玩法已经收敛为：

主线自动/半自动刷图 + 装备成长 + 卡关回刷

更完整的表述是：

- 玩家通过主线关卡推进
- 进入战斗，击败普通怪、精英怪、Boss
- 获得装备与基础材料
- 通过更换装备提升角色强度
- 打不过下一关时，回刷旧关继续成长
- 形成“刷图 → 掉落 → 变强 → 推进”的循环

### 2. 第一版暂不重点做的内容
以下内容目前明确不作为第一版优先项：

- 经卷回刷等级制
- world level 体系
- 副本矩阵
- 宝石
- 蓝词条
- 紫洗练
- 套装
- 商店
- 交易所
- 活动系统
- 任务系统
- 多页面大导航体系
- 高复杂 UI 美术

---

## 四、页面与前端方向结论

### 1. 第一版页面方向
第一版页面建议采用：

低保真线框页 / 线条型页面 / 不使用高复杂美术 / 不填充正式图片

原因：
- 优先验证玩法和流程
- 便于反复修改结构
- 避免大量资源浪费在美术细节上

### 2. 第一版页面范围
当前已确认第一版先只围绕这 6 个页面展开：

1. 大厅页
2. 主线页
3. 战斗页
4. 结算页
5. 背包页
6. 角色页

### 3. 技能页结论
技能页当前不作为第一版独立页面开发。  
现阶段结论是：

- 技能页先不单独展开
- 角色页可预留“技能”入口
- 后续需要时再单独做技能专页

## 五、战斗方向结论

### 1. 战斗模式结论
战斗不建议做成纯全自动观看，也不建议一上来做成重操作动作游戏。

当前更推荐的方向是：

手动移动走位 + 自动攻击 + 少量技能参与

也就是：
- 玩家可以移动
- 普攻自动攻击
- 技能先不做太复杂
- 战斗偏轻操作、重选择、重反馈

### 2. 战斗体验重点
战斗体验不主要依赖复杂连招，而主要依赖：

- 清怪效率感
- 怪物层级（普通 / 精英 / Boss）
- 掉落期待感
- 变强后的体感变化
- Boss 压力与突破感

### 3. Boss 与战斗反馈
目前倾向于保留这些基本反馈方向：

- 普通怪 / 精英怪 / Boss 明显区分
- 飘字反馈
- 受击反馈
- Boss 预警
- Boss 登场感

## 六、属性体系结论

### 1. 玩家可加点属性
当前已确认将可加点属性统一为四大基础属性：

- 力量
- 法力
- 体魄
- 灵巧

### 2. 战斗属性
攻击、防御、生命、攻速等不再作为直接加点属性，而作为战斗属性，由基础属性换算得出。

目前讨论中涉及的战斗属性主要包括：

- 攻击
- 防御
- 生命
- 攻速
- 暴击率（可选）

### 3. 属性层级结论
当前属性体系层级已经基本明确：

#### 第一层：基础属性
- 力量
- 法力
- 体魄
- 灵巧

#### 第二层：通用换算公式
将基础属性换算为基础战斗属性

#### 第三层：职业修正
不同职业对最终战斗属性进行修正

### 4. 属性来源结论
最终基础属性会来自：

- 职业基础属性
- 升级加点
- 装备加成
- 技能加成

---

## 七、职业模块结论

### 1. 当前职业方向
当前确认保留三职业方向：

- 金刚
- 灵羽
- 符箓

### 2. 职业模块边界
职业模块不再直接存“基础生命 / 基础攻击 / 基础防御”这类最终战斗数值，  
而应以：

- 基础力量
- 基础法力
- 基础体魄
- 基础灵巧

作为职业初始属性来源。

### 3. 职业差异方式
职业差异不建议通过完全不同的属性公式实现，而应采用：

统一基础公式 + 职业修正系数

即：
- 所有职业共用同一套基础属性 → 战斗属性换算公式
- 不同职业通过修正倍率来体现差异

这样更容易维护和扩展。

## 八、武器系统结论

### 1. 武器系统总体方向
当前已经确认，武器系统长期方向应具备一定深度，但第一版不必全部开放。

核心规则如下：

- 主武器分单手 / 双手
- 双手主武器不能装备副武器
- 单手主武器可以装备副武器
- 副武器不提供直接伤害
- 武器本身不设置技能

### 2. 主武器职责
主武器负责：

- 主要输出
- 攻击方式
- 攻击距离
- 攻击节奏
- 攻击形态

### 3. 副武器职责
副武器负责：

- 防御增强
- 技能/伤害强化
- 属性加成
- 战斗风格修正

副武器不形成独立攻击循环，不应喧宾夺主。

### 4. 攻击范围结论
当前已确认：

- 攻击范围应主要由主武器决定
- 远程武器或特定武器可以通过副武器、物品、技能等方式获得攻击范围加成
- 职业本身不直接定义攻击距离

- 长枪
- 双手锤
- 长弓
- 弩
- 法杖
- 魔杖

### 2. 当前确认的副武器
当前确认保留的副武器为：

- 盾牌
- 箭囊
- 机括
- 灵珠
- 法器
- 符印

### 3. 单手 / 双手结论
#### 单手主武器
- 单手剑
- 弩
- 魔杖

#### 双手主武器
- 长枪
- 双手锤
- 长弓
- 法杖

### 4. 远程与法系武器结论
已明确：

- 长弓 = 双手
- 弩 = 单手
- 法杖 = 双手
- 魔杖 = 单手

这个拆分解决了远程和法系副武器适配的问题，使体系更清晰。

### 5. 双手锤结论
当前已决定：

- 保留双手锤
- 去掉双手大剑
- 双手锤具备击退敌人的特性

对击退效果的理解为：
- 对普通怪击退明显
- 对精英怪击退弱化
- 对 Boss 基本不做位移击退

这样双手锤能形成自己的独特定位：
- 重击
- 压制
- 打开敌阵
- 近战空间控制

## 十、装备系统结论

### 1. 当前装备位
当前确认的装备位包括：

- 主武器
- 副武器
- 盔甲
- 护腿
- 手套
- 鞋子
- 披风
- 项链
- 戒指 ×2
- 手镯 ×2

### 2. 装备系统结构结论
目前更推荐：

物品主表 + 装备扩展表

而不是：
- 武器独立一张表、其他装备再各起很多表

### 3. 为什么采用“装备表”而非“武器表”
结论是：

- 武器应视为装备的一种
- 主武器、副武器、防具、饰品都属于装备
- 更合理的是直接做“装备表”，武器只是其中一个子类

### 4. 戒指和手镯结论
当前倾向于：

- 装备定义层只定义 ring、bracelet
- 角色穿戴层再定义：
  - 戒指槽1 / 戒指槽2
  - 手镯槽1 / 手镯槽2

这样职责更清晰。

## 十一、物品与装备关系结论

### 1. 武器是否写入物品表
当前结论是：

武器要写入物品表。

### 2. 原因
因为武器首先是“玩家可获得的对象”，需要统一参与：

- 掉落
- 背包
- 奖励展示
- 稀有度显示
- 排序
- 后续出售 / 分解 / 锁定等功能

### 3. 推荐结构
#### 物品表
作为统一主表，承载所有可获得对象的共性字段

#### 装备扩展表
通过同一个 item_id 承载装备专属字段

### 4. 稀有度结论
稀有度建议统一放在物品主表，而不是武器/装备表里重复存一份。

## 十六、阶段性总总结

当前阶段已经达成的关键结论可以概括为：

### 玩法方向
第一版先做最小可玩闭环，不追求功能堆积。

### 页面方向
第一版先做低保真线框页，不优先做复杂游戏端表现。

### 开发顺序
先方案、再后台、再接口、最后游戏端。

### 属性体系
使用四大基础属性，战斗属性通过公式计算。

### 职业体系
三职业共用一套基础换算公式，通过职业修正体现差异。

### 武器体系
主武器分单手 / 双手，副武器不输出，武器不带技能。

### 装备体系
武器属于装备的一部分，装备统一走“物品表 + 装备扩展表”。

### 数据结构方向
当前已经基本确定应走“统一主表 + 专属扩展表”的思路，减少系统碎片化。
