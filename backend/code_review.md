

# 《山海巡厄录》Backend Code Review 清单

## 1. 文档目的

本文档用于统一《山海巡厄录》项目在 `backend/` 目录下的 Code Review 标准，作为 `backend/AGENTS.md` 的配套审查清单使用。

本文档重点聚焦 Laravel 后端、后台管理、API、Seeder、Migration、Model、Service、Workflow 等后端内容，不覆盖根目录游戏运行时代码的具体审查细节。

本文档的目标是让每次后端评审都能快速检查：

- 是否符合后端文档与项目总文档
- 是否符合五层服务分层
- 是否存在 Controller 过重、跨层泄漏、旁路写状态
- 是否存在事务、幂等、唯一约束、重复写入风险
- 是否存在接口口径、错误码、枚举值漂移
- 是否存在 Seeder 脏重复数据问题
- 是否存在后台逻辑替代正式业务链的问题

本文档定位为：

- `backend/` 目录专用 Review 清单
- Codex 自查与人工评审共用清单
- Laravel 后端合并前必过的基础审查规则文档

---

## 2. Backend Review 总原则

### 2.1 先看任务边界，再看代码实现

评审时优先确认：

- 本次改动是否只发生在 `backend/` 范围内
- 是否动了与任务无关的后端模块
- 是否偷偷扩展了未来系统
- 是否为了方便而绕过正式链路

如果任务边界已经跑偏，后续实现细节就没有继续细看的价值。

### 2.2 先看正式主链正确性，再看代码优雅程度

第一阶段后端的优先级是：

- 正式状态正确
- 事务边界正确
- 链路边界正确
- 接口契约稳定
- 数据可追溯

而不是：

- 抽象层数更多
- 类拆得更细
- 写法更“高级”

### 2.3 先看高风险点，再看一般逻辑

优先检查：

- 是否可能留下半成状态
- 是否可能重复发奖
- 是否可能重复入包
- 是否可能重复初始化槽位
- 是否可能重复创建可堆叠记录
- 是否可能让同一装备实例多槽位占用

这些风险点优先级高于一般代码风格问题。

## 3. 文档一致性检查

后端评审时，至少先检查以下文档是否被遵守。

### 3.1 是否符合《项目总览》
- 是否仍围绕第一阶段主链开发
- 是否没有擅自扩展未来系统

### 3.2 是否符合 `backend/AGENTS.md`
- 是否只在允许范围内改动
- 是否遵守了后端专用规则

### 3.3 是否符合《Codex 主开发协作清单》
- 是否存在任务越权
- 是否存在未授权目录或模块改动

### 3.4 是否符合《Laravel 代码目录与命名规范》
- 目录落位是否正确
- 类名、方法名、文件名是否统一
- 是否存在模糊命名 Service

### 3.5 是否符合对应方法级设计文档
- Query / Domain / Workflow 是否按文档拆分
- 方法职责是否清晰
- 事务边界是否符合文档

### 3.6 是否符合接口字段级设计 / 接口示例文档
- 请求字段是否一致
- 响应结构是否一致
- 错误响应结构是否一致

### 3.7 是否符合错误码总表 / 枚举总表 / 认证与接口公共规则
- 是否使用正式错误码
- 是否使用正式枚举值
- 是否遵守认证与公共接口规则

---

## 4. 任务边界检查

### 4.1 是否只实现了本次任务要求
- 是否只改了本次后端任务涉及的链路或模块
- 是否没有顺手补做无关后台页、无关接口、无关服务

### 4.2 是否存在无关重构
- 是否修改了与当前任务无关的后端目录
- 是否把局部任务扩成了大规模后端重构

### 4.3 是否擅自补了未来系统
重点检查是否偷偷加入了：

- 强化
- 洗练
- 宝石
- 套装
- 经卷
- world level
- 商店系统
- 活动运营系统

### 4.4 是否擅自新增未定稿表 / 字段 / 接口
- 是否擅自新增 schema 字段
- 是否擅自新增正式 API
- 是否擅自扩展正式响应字段

---

## 5. 五层服务分层检查

后端正式以五层设计为准：

1. Config
2. Query
3. Domain
4. Workflow
5. Admin

### 5.1 Config 层检查
- 是否只负责读取静态配置
- 是否没有写状态
- 是否没有做事务
- 是否没有承担完整流程

### 5.2 Query 层检查
- 是否只负责读取正式状态
- 是否没有隐藏写入
- 是否没有变成流程服务

### 5.3 Domain 层检查
- 是否承担单点业务动作或规则判断
- 是否没有把整条链所有步骤全塞进去
- 是否没有承担接口拼装职责

### 5.4 Workflow 层检查
- 是否负责完整流程编排
- 是否负责事务边界控制
- 是否协调多个服务而不是重写所有底层逻辑

### 5.5 Admin 层检查
- 是否只负责后台校验、引用检查、补发、修复
- 是否没有复制正式玩家链逻辑
- 是否没有绕过正式 Workflow 直接改正式状态

---

## 6. Controller / Request / Resource 检查

### 6.1 Controller 检查
- 是否保持轻量
- 是否只做收参、调服务、回响应
- 是否没有直接开启事务并操作多表完成全流程
- 是否没有直接实现掉落 / 发奖 / 入包逻辑

### 6.2 Request 检查
- 是否只做基础参数校验
- 是否没有把正式业务规则塞进 Request
- 是否没有在 Request 中做复杂跨表访问

### 6.3 Resource / 响应构造检查
- 是否符合统一响应外壳
- 是否字段命名统一为 `snake_case`
- 是否没有每个接口各造一套返回风格

---

## 7. Model / Migration / Schema 检查

### 7.1 Model 检查
- Model 命名是否准确
- 是否没有出现过度抽象或模糊命名模型
- 关系定义是否符合当前业务边界
- 是否没有把模板层与实例层关系写混

### 7.2 Migration 检查
- 是否只新增本任务明确需要的表或字段
- 字段命名是否清晰稳定
- 是否符合已定稿数据库口径
- 是否没有为了方便开发擅自弱化唯一约束

### 7.3 唯一约束与索引检查
重点检查：

- `character_equipment_slots(character_id, slot_key)`
- `character_equipment_slots(equipped_instance_id)`
- `inventory_stack_items(user_id, item_id)`
- `drop_group_bindings(source_type, source_id)`
- `reward_group_bindings(source_type, source_id)`
- `user_reward_grants(user_id, idempotency_key)`
- `user_reward_grants(user_id, source_type, source_id, reward_group_id)`

### 7.4 Schema 与文档差异检查
- 若 migration / model / 文档不一致，是否明确指出
- 是否没有在未说明的情况下自行重写业务口径

---

## 8. 模板层 / 实例层检查

这是后端最容易出结构性错误的部分之一。

### 8.1 物品模板与玩家持有状态是否分离
- `items` 是否仍只表示模板
- 是否没有把持有数量写回模板

### 8.2 装备模板与装备实例是否分离
- 是否明确区分 `equipments` 与 `inventory_equipment_instances`
- 换装是否操作实例而不是模板
- 入包时装备是否实例化后再入正式状态

### 8.3 奖励模板与发奖记录是否分离
- 是否区分 `reward_groups` 与 `user_reward_grants`
- 是否没有把正式发放结果写回奖励模板

---

## 9. 掉落链 / 奖励链 / 入包链检查

### 9.1 掉落链检查
- 掉落链是否只负责解析正式掉落结果
- 是否没有直接写背包
- 是否没有承担发奖职责

### 9.2 奖励链检查
- 奖励链是否只负责正式奖励发放、防重、记录、入包承接
- 是否没有承担怪物随机掉落逻辑

### 9.3 入包链检查
- 入包链是否为统一正式入口
- 是否正确分流可堆叠物与装备对象
- 装备是否实例化后才算正式入包

### 9.4 是否存在链路混用
重点检查：

- 是否把怪物掉落混进首通奖励逻辑
- 是否把首通奖励伪装成掉落结果
- 是否在战斗结算里直接手写入包而跳过正式子链

---

## 10. 事务、幂等与一致性检查

### 10.1 角色创建链
- 是否角色主表与 12 个槽位在同一事务中初始化
- 是否失败时不会留下半成角色状态

### 10.2 换装链
- 是否目标槽位更新与联动清理在同一事务中
- 是否不会出现主武器已更新但副武器未清理
- 是否不会出现同一实例被多个槽位占用

### 10.3 奖励发放链
- 是否正确使用业务防重
- 是否正确使用幂等键
- 是否主记录、明细、入包、状态更新在同一事务中
- 是否不会出现奖励已入包但状态仍 `pending`

### 10.4 入包链
- 是否可堆叠物写入与实例创建在同一事务中
- 是否不会出现堆叠物成功而实例失败却仍返回成功

### 10.5 战斗结算链
- 是否不会把理论结果当正式成功结果返回
- 是否掉落、奖励、入包任一失败后仍伪装成结算成功

---

## 11. API 契约检查

### 11.1 认证检查
- 是否从认证上下文获取当前用户
- 是否没有信任前端传入的 `user_id`
- 是否业务接口都在认证中间件保护下

### 11.2 请求字段检查
- 是否符合接口字段级设计
- 是否字段使用 `snake_case`
- 是否没有新增未定稿字段

### 11.3 响应字段检查
- 是否符合接口字段级设计与接口示例文档
- 是否没有擅自删字段或改字段名
- 是否没有把中文展示值替代正式枚举值

### 11.4 公共结构检查
- 是否成功统一返回 `code/message/data`
- 是否失败统一返回正式错误码且 `data = null`
- 分页接口是否统一使用 `list + pagination`

### 11.5 时间与空值检查
- 时间字段是否统一格式
- 单对象空值是否返回 `null`
- 是否没有用空字符串代替对象空值

---

## 12. 枚举与错误码检查

### 12.1 枚举检查
- 是否严格使用《枚举总表》中的正式值
- 是否存在 `mainWeapon` / `main_weapon` 混用
- 是否存在 `success` / `done` / `ok_status` 混用

### 12.2 错误码检查
- 是否使用《错误码总表》中的正式错误码
- 是否同一语义用了多个错误码
- 是否临时自造错误码却未补文档

### 12.3 magic string 检查
- 是否在关键链路中散写枚举字符串
- 是否该提取为 Enum / 常量却没有提取

---

## 13. Seeder 与最小联调数据检查

如果本次改动涉及 Seeder、Factory、测试数据或联调数据，重点检查：

### 13.1 是否符合《最小联调种子数据》
- 是否使用固定业务 ID
- 是否覆盖最小联调闭环

### 13.2 是否可重复执行
- 是否使用稳定 `upsert` / `updateOrCreate` / `firstOrCreate`
- 是否不会产生脏重复数据

### 13.3 是否没有引入过量数据
- 是否只做最小联调所需数据
- 是否没有塞入大量正式运营内容

### 13.4 是否兼容当前真实 schema
- 如果真实表结构与文档略有差异，是否有明确适配说明

---

## 14. 后台资源页与管理动作检查

### 14.1 后台资源页检查
- 是否服务正式主链，而不是另起一套后台逻辑真相
- 表单字段、列表字段、筛选字段是否与文档一致
- 是否存在保存前关键校验

### 14.2 后台业务动作检查
- 若后台动作触发正式业务行为，是否复用正式 Workflow / Domain
- 是否没有在页面或后台 action 中复制一套玩家主链逻辑

### 14.3 后台补发 / 修复工具检查
- 是否有必要的权限与风险说明
- 是否没有直接裸改正式状态而绕过约束
- 是否结果可追溯

---

## 15. 测试与验证检查

### 15.1 是否有最小验证动作
- 是否运行了相关测试
- 是否至少做了语法检查 / route:list / Seeder 校验 / 功能链验证中的一部分

### 15.2 是否给出自测步骤
- 是否说明如何验证本次后端改动
- 是否说明前置条件

### 15.3 验证失败时是否诚实说明
- 是否明确指出无法验证的原因
- 是否没有把“未验证”写成“已完成”

---

## 16. 文档回写检查

如果本次改动改变了正式契约，需检查是否同步更新文档：

### 16.1 需要同步更新文档的典型场景
- 新增或调整正式接口字段
- 新增或调整错误码
- 新增或调整枚举值
- 新增或调整最小联调种子数据 ID
- 变更正式业务边界、事务边界、正式实现口径

### 16.2 不需要为无意义风格改动频繁改文档
- 不要只为措辞微调而 churn 文档
- 只在正式行为、契约、边界变化时回写文档

---

## 17. 常见高风险反例清单

评审时如果看到以下情况，应重点拦截：

1. Controller 里直接写事务和多表逻辑
2. Request 里塞正式业务规则
3. 奖励发放跳过正式 Workflow 直接入包
4. 掉落解析直接写背包
5. 装备模板和装备实例混用
6. 首通奖励重复发放没有业务防重
7. 只做服务层判断，没有数据库唯一约束意识
8. Seeder 重复执行会插出重复脏数据
9. 同一正式枚举写出多套字符串值
10. 响应结构与接口示例文档不一致
11. 后台页复制正式玩家主链逻辑
12. 任务明明只做一条后端链，却顺手改了大量无关模块

## 18. Review 结论输出建议

每次 Backend Code Review 建议至少输出以下结论结构：

### 18.1 结论等级
- `通过`
- `有条件通过`
- `不通过`

### 18.2 必改问题
列出会影响：

- 正式状态正确性
- 后端主链正确性
- 接口契约稳定性
- 数据约束安全性

的问题。

### 18.3 建议优化项
列出：

- 结构可优化
- 命名可优化
- 文档可补充
- 测试可补充

但不影响当前主链正确性的内容。

### 18.4 风险提示
说明：

- 当前尚未验证的风险
- 环境限制导致未验证的内容
- 后续建议继续补的测试或文档

---

## 19. 当前阶段结论

1. `backend/` 目录的 Code Review 必须优先围绕正式状态正确性、事务边界、唯一约束、幂等防重、接口契约稳定性来审查
2. 评审顺序建议为：先看任务边界，再看分层，再看高风险主链，再看接口契约，最后看代码细节
3. 模板层 / 实例层分离、掉落链 / 奖励链分离、奖励防重、入包一致性、Seeder 可重复执行，是当前阶段后端最关键的 Review 检查点
4. 后续无论是 Codex 自查还是人工评审，都建议以本文档作为 `backend/` 目录的统一 Review 清单

本文档可直接作为项目正式 `backend/code_review.md` 使用，并作为 `backend/AGENTS.md` 的配套审查基线文档。
